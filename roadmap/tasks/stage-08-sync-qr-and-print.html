Stage 8: Sync, QR, And Print

Goal​

Guests and operators can export, share, and print media through reliable service-backed workflows.

Design Source — Convert The Prototype​

Build the booth UI from the generated prototype. The agreed visual and interaction reference is design-prototypes/booth-touchscreen/ (open index.html), specified in booth-touchscreen-design-brief.md.

This stage makes the Result screen's share / QR / print actions functional (the toolbar shell is built in Stage 4) and adds the admin remote-config / queue-status surfaces. Rebuild in apps/web with Tailwind + semantic tokens (use the palette→token table in the prototype's README). Do not import prototype code; preserve the 1280x800 no-scroll guarantee.

Context​

Add media export, guest sharing, and printing without blocking capture or letting UI code cross privileged hardware/service boundaries. Pixelbooth is local-first Raspberry Pi booth software; preserve fake/off-hardware paths and service boundaries.

Hardware/decision prerequisites: rclone packaged on the Pi with configured remotes; a Canon Selphy (or CUPS/IPP-reachable printer) for the print path; an Immich endpoint confirmed against current official docs. The local-folder and fake remotes are fully testable off-hardware first.

Implementation Guidance​

  • Start with CLAUDE.md and the references below.
  • Keep sync, QR, and print work queued or service-backed so capture remains responsive.
  • Put USB, printer, and other privileged host operations behind helpers or explicit services.
  • Treat payment gating as future architecture only, not an initial payment implementation.

Acceptance Checks​

  • Docs are updated before or alongside behavior changes, and the owning docs are listed in the PR/summary.
  • Guest-facing capture feedback is never blocked by expensive rendering, sync, or print work.
  • Hardware or privileged operations stay behind backend services, adapters, or host helpers.
  • Fake/off-hardware tests or verification steps cover the riskiest behavior; real-hardware checks are documented when required.

Features​

Media sync remotes​

Media can leave the booth through durable queued remotes. Each remote shares one common operator model and carries type-specific adapter settings. Transport engines are decided in ADR-0005: rclone is the default for cloud + SFTP, rsync/SSH serves LAN/self-hosted copies, and Immich uses its native upload path. Because booth media is write-once, the priorities are parallelism, bandwidth control, integrity verification, resumability, and a durable retry queue — not delta transfer.

Tasks​

  • Task: Add the sync outbox and remote runner.
    • Implement durable jobs with retry/status behavior, output selection, trigger policy, and per-remote queue health, reusing the Stage 4 shared job/queue primitive rather than a second queue implementation.
  • Task: Add local folder/export remote.
    • Provide the simplest fully testable remote first so queue behavior and admin UX can mature before cloud/provider work.
  • Task: Add USB remote.
    • Add USB export behind a privileged/helper boundary with clear mount, copy, and error handling.
  • Task: Add rclone remote (default cloud/SFTP engine).
    • Use rclone for Google Drive and other cloud providers and for SFTP targets, exposing its bandwidth (--bwlimit), parallelism (--transfers/--checkers), checksum-verification, and retry controls; keep rclone config/secrets behind the ConfigService secret-redaction boundary. Document rclone packaging/auth on the Pi.
  • Task: Add Immich remote (native upload path).
    • Use Immich's native upload (CLI/API) so server-side dedup and albums work; confirm the exact endpoint against current official Immich docs at implementation time and document auth/config.
  • Task: Add rsync/SSH remote.
    • Support simple LAN/self-hosted server copies over SSH where its speed and robustness are wanted.

QR sharing​

Guests can access event/media links through customizable QR experiences.

Stories​

  • Story: Guest can use customized QR codes for media and event links.
    • As a guest, I can use customized QR codes for per-media and event links.

Printing workflow​

Operators can configure and preview print jobs before sending them to the target printer.

Stories And Tasks​

  • Task: Add CUPS/IPP print adapter and Canon Selphy target.
    • Implement print service boundaries and verify the Canon Selphy path on real hardware when available.
  • Story: Guest or operator can preview print crop and safe area.
    • As a guest or operator, I can preview print crop and safe-area before printing.
  • Task: Define future payment-provider gate for print jobs.
    • Document how a payment gate could fit later without implementing payment processing now.

Sharing, sync, and print frontend​

The frontend should cover the reference app share/QR/print workflows with clearer queue state, guest-safe progress feedback, and admin configuration surfaces.

Stories And Tasks​

  • Story: Admin can configure sync remotes and see queue health.
    • As an admin, I can configure one or more local, USB, rclone, Immich, and rsync/SFTP remotes with shared policy fields and type-specific settings, then see pending, failed, retrying, and completed jobs.
  • Story: Admin can test each remote.
    • As an admin, I can run a connection/config test for a remote and see a clear success/failure result before enabling automatic sync.
  • Story: Guest can see QR, share, and print progress after capture.
    • As a guest, I can scan a QR code or request print/share and see pending, success, or failure feedback without blocking the booth.
  • Task: Build print safe-area preview interaction and validation.
    • Add frontend crop/safe-area controls for the Canon Selphy target, including warnings for unsafe crop, unavailable printer, or queued print failure.
