Stage 4: Media Processing

Goal​

Captured media is stored, previewed, rendered, and extended through deterministic service boundaries that work off-hardware.

Design Source — Convert The Prototype​

Build the booth UI from the generated prototype. The agreed visual and interaction reference is design-prototypes/booth-touchscreen/ (open index.html), specified in booth-touchscreen-design-brief.md.

This stage converts the Result screen (finished media as hero, delete action, and share/QR/print toolbar buttons) and backs the Mode Select options (still, collage, animation/GIF-boomerang, video) with real render outputs. Rebuild in apps/web with Tailwind + semantic tokens (use the palette→token table in the prototype's README). Do not import prototype code; preserve the 1280x800 no-scroll guarantee. The Result toolbar's share/QR/print actions become functional in Stage 8.

Context​

Build durable media storage and background processing so guest capture feedback stays immediate while richer outputs render asynchronously. Pixelbooth is local-first Raspberry Pi booth software; preserve fake/off-hardware paths and service boundaries.

Implementation Guidance​

  • Start with CLAUDE.md and the references below.
  • Keep scope limited to this epic and its child features.
  • Reuse current patterns in apps/web, services/api, adapters, events, and tests.
  • Do not pull future-stage work forward unless the docs make it necessary.

Acceptance Checks​

  • Docs are updated before or alongside behavior changes, and the owning docs are listed in the PR/summary.
  • Guest-facing capture feedback is never blocked by expensive rendering, sync, or print work.
  • Documented performance budgets exist and are checked: instant preview appears within a target after capture, full-resolution render runs in the background within a target ceiling, and render progress is emitted over the WebSocket contract. Record the chosen numeric targets in the docs.
  • Fake/off-hardware tests or verification steps cover the riskiest behavior; real-hardware checks are documented when required.

Features​

Media storage and API contract foundation​

Capture metadata and generated media have a durable model and testable fake implementation.

Tasks​

  • Task: Add the media catalog on the existing SQLite foundation.
    • Add the media metadata catalog and file store behind a service boundary on the Stage 2.5 SQLite foundation (engine/migrations/connection already exist); files remain authoritative for bytes and SQLite for metadata. Include a fake/in-memory adapter for off-hardware tests.
  • Task: Add the shared job/queue primitive.
    • Introduce one durable, retryable, status-emitting job/queue primitive that RenderWorkerPool uses here and the Stage 8 sync outbox reuses, so render and sync do not grow divergent queue implementations. It must emit progress over the WebSocket contract and never block capture acknowledgement.
  • Task: Add storage-capacity and retention handling.
    • Report disk usage, define capacity thresholds, and implement a retention/eviction policy so a busy event cannot silently fill the disk; surface capacity state to the dashboard and degrade safely when low.
  • Task: Add deterministic fake media/render adapter.
    • Create fake rendering outputs suitable for tests, UI development, and CI without camera/media hardware.
  • Task: Adopt generated frontend TypeScript API types from FastAPI OpenAPI.
    • Stand up the repeatable type-generation workflow now that the session/media API shape is stable, and migrate the Stage 2.5 hand-written admin types onto the generated types.

Fast preview and background render pipeline​

Guests see immediate results while expensive processing happens asynchronously.

Stories​

  • Story: Guest gets thumbnails and previews from captures.
    • As a guest, I get thumbnails and previews from captures so gallery/admin/guest surfaces update quickly.
  • Story: Guest sees instant preview before full-resolution render completes.
    • As a guest, I see an instant preview followed by a background full-resolution render so the booth feels responsive.

Overlay-aware rendering​

The media pipeline consumes overlay definitions from OverlayService instead of hardcoded file paths.

Stories And Tasks​

  • Story: Operator can apply V1 PNG overlays to final output.
    • As an operator, I can assign a transparent PNG overlay globally, by event/template, or by capture preset, and final renders include it.
  • Story: Operator can trust overlay fallbacks.
    • As an operator, I get validation and safe fallback behavior for missing overlay assets, wrong dimensions, missing alpha channels, or incompatible media types.
  • Task: Add overlay composition stage.
    • Composite validated transparent PNG overlays into still/collage outputs while keeping instant preview fast and full-resolution work in the background render path.

Composed media formats​

The pipeline can produce booth-friendly collages, animations, boomerangs, and short videos.

Stories​

  • Story: Guest can receive generated collages.
    • As a guest, I can receive generated collages through the media pipeline.
  • Story: Operator can show collage wall or canvas mode.
    • As an operator, I can show a collage wall/canvas mode suitable for kiosk/gallery presentation.
  • Story: Guest can receive GIF or animation outputs.
    • As a guest, I can receive GIF/animation outputs generated through the worker-style pipeline.
  • Story: Guest can receive boomerang outputs.
    • As a guest, I can receive boomerang outputs using a worker/ffmpeg-style pipeline.
  • Story: Guest can capture and render short video with optional sound.
    • As a guest, I can capture and render short video with optional sound when hardware and preset support it.

Advanced media extension points​

Future effects can plug into the pipeline without reshaping storage or capture contracts.

Stories And Tasks​

  • Story: Operator can enable chromakey or background removal.
    • As an operator, I can enable chromakey/background-removal processing while provider/model choices remain replaceable.
  • Task: Add filter and face-landmark pipeline extension point.
    • Define extension interfaces and docs without committing to a specific CV implementation too early.

Capture result and gallery frontend​

The frontend should carry the guest from instant result preview into gallery/detail/share actions, using the reference app gallery list/detail as a baseline while improving state handling, empty states, and typed React source ownership.

Stories And Tasks​

  • Story: Guest sees an instant result screen while render work continues.
    • As a guest, I can see the captured preview, render status, and available QR/share/print/gallery actions without waiting for full-resolution outputs.
  • Story: Guest sees safe overlay render feedback.
    • As a guest, I see the final result without technical errors; admin surfaces can show whether an overlay was applied or skipped with a diagnostic id.
  • Story: Guest or admin can browse gallery list and detail views.
    • As a guest or admin, I can browse newest media with thumbnails, media type, timestamp, camera role, preset, and detail actions for hide, delete, download, share, and print.
  • Task: Build frontend media API client and query cache integration.
    • Add typed API/query hooks for capture, render status, gallery list/detail, and media actions, with loading, empty, success, and error states.
