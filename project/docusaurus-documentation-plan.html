Docusaurus Documentation Plan

This document is the implementation handoff for adding a Docusaurus documentation site to Pixelbooth. It records the agreed plan before any Docusaurus scaffolding, package installation, workspace configuration, or deployment automation is added.

Purpose And Audience​

The Docusaurus site should make Pixelbooth easier to build, operate, and extend without replacing the current docs source of truth.

Primary audience:

  • Builders working on the FastAPI backend, React frontend, hardware adapters, media pipeline, and staged roadmap.
  • Operators setting up or running the Raspberry Pi booth for events.
  • Future AI or human contributors who need the architecture, roadmap, and decision history in one navigable site.

The site is not a marketing page for the first version, even if it is published publicly. It should favor clear handoff, practical runbooks, architecture context, and future-stage documentation discipline.

Agreed Decisions​

  • Add the Docusaurus app later at apps/docs.
  • Keep .docs/ as the canonical documentation content root.
  • Configure Docusaurus to read or mirror .docs/ rather than moving the docs into apps/docs.
  • Serve builders and operators first.
  • Target local preview and GitHub Pages as the first publishing path.
  • Publish generated static output to the public Katrines-Kaelder/pixelbooth-docs repository while keeping the private Katrines-Kaelder/pixelbooth repository as the source of truth.
  • Disable blog/versioning for the first pass unless implementation-time needs prove otherwise.
  • Verify current Docusaurus package versions from the registry at implementation time, following the current-package-versions rule.
  • Do not scaffold Docusaurus as part of this documentation-only change.

Proposed Site Shape​

The future apps/docs package should be a TypeScript Docusaurus site in the existing pnpm workspace. It should provide local and production docs commands without changing the runtime app APIs.

Root scripts to add later:

  • docs:start: run local Docusaurus development server.
  • docs:build: build the static documentation site.
  • docs:serve: serve the production build locally.
  • docs:clear: clear Docusaurus caches when sidebar/routes get stale.

Recommended sidebar sections:

  • Start Here: project overview, current baseline, quick start, docs rules.
  • Product: workflows, design direction, moodboard, touchscreen brief.
  • Architecture: system architecture, runtime decisions, admin control plane, camera/lighting, expanded capabilities, frontend standards, overlay schema.
  • API And Events: REST contract, WebSocket events, and later generated OpenAPI/type links once the API contract stabilizes.
  • Roadmap: staged roadmap and task pages for Stage 2 through Stage 10.
  • Operations: setup, Pi bring-up, degraded modes, diagnostics, backups, opening-night checks, and future rental/operator runbooks.
  • ADRs: architecture decision record index and accepted decisions.
  • AI Collaboration: root CLAUDE.md, .claude/README.md, rules, skills, agents, and review workflow.

Asset And Link Migration Notes​

Docusaurus will be less forgiving than plain Markdown in a file browser. The implementation should include a cleanup pass before treating the site as done.

  • Keep existing relative Markdown links where they resolve cleanly inside .docs/.
  • Fix links that point outside .docs/ so the published site has stable routes or explicit repository links.
  • Images that were under .docs/design ideas/ now live in .docs/assets/design-ideas/.
  • Image files use lowercase kebab-case names to avoid fragile URL encoding and path escaping.
  • Markdown image links have been updated to the new asset paths.
  • Enable Mermaid rendering for architecture diagrams.
  • Add Docusaurus category metadata only where it improves navigation; avoid front matter churn across every page.

Future-Stage Documentation Checklist​

Every future implementation stage must update the relevant docs in the same change set as the code. The docs should explain what changed, how to use it, how to test it, and what remains deferred.

  • Stage 3, cameras: hardware setup, adapter behavior, degraded camera modes, preview switching, and Pi verification checklist.
  • Stage 3.5, sessions: state machine, guest flow, abort/retry/conflict behavior, REST endpoints, and WebSocket events.
  • Stage 4, media: storage layout, render queue, retention policy, gallery behavior, and generated OpenAPI/TypeScript type adoption.
  • Stage 5, lighting: fake/WLED/custom ESP32 setup, exposure profiles, ring-light calibration, and preset testing.
  • Stage 6, marquee: design schema, canvas constraints, agent protocol, admin editor behavior, and hardware verification.
  • Stage 7, physical booth: Pi deployment, GPIO trigger mapping, Wi-Fi fallback, rental onboarding, smoke tests, and recovery steps.
  • Stage 8, sync/QR/print: remote setup, queue behavior, Immich/rclone/rsync notes, QR links, printer setup, and failure recovery.
  • Stage 9, challenges: challenge presets, scoring/result metadata, guest prompts, gallery tags, and admin editing.
  • Stage 10, observability: log viewer, diagnostic ids, safe diagnostic bundles, field troubleshooting, and support handoff.

When a stage changes system behavior, service boundaries, public APIs/events, or hardware contracts, update the owning architecture doc before or alongside the implementation. Significant hard-to-reverse decisions still need an ADR.

CI And Deployment Plan​

The later Docusaurus implementation should add a docs workflow after the local build is stable.

  • On pull requests, build the docs but do not publish.
  • On pushes to the docs branch, build the docs and publish automatically.
  • The publish run pushes the static Docusaurus build to the public Katrines-Kaelder/pixelbooth-docs repository. The workflow may also be run manually from GitHub Actions when a one-off publish is useful.
  • Configure the public docs repository to serve GitHub Pages from main and /.
  • Store the write-capable deploy key or fine-grained token only in the private source repository, not in the public docs repository.
  • Prefer storing the private deploy key as the base64-encoded PIXELBOOTH_DOCS_DEPLOY_KEY_B64 secret to avoid multiline secret parsing issues in GitHub Actions.
  • Configure the site for the /pixelbooth-docs/ base URL.
  • Keep docs deployment independent from apps/web and apps/api builds.
  • Use workflow concurrency so overlapping publish runs do not race the published output.

Later Implementation Acceptance Checks​

The Docusaurus implementation is done when:

  • pnpm docs:start serves the docs locally.
  • pnpm docs:build passes without broken links, missing images, or MDX errors.
  • pnpm docs:serve serves the production build locally.
  • All existing .docs Markdown pages are reachable from the sidebar or index.
  • The Mermaid system architecture diagram renders.
  • Moodboard and touchscreen reference images render.
  • GitHub Actions validates the docs build on pull requests.
  • GitHub Pages serves the published site at https://katrines-kaelder.github.io/pixelbooth-docs/.

Scope Boundaries​

This plan does not implement Docusaurus. It deliberately avoids:

  • Creating apps/docs.
  • Installing Docusaurus packages.
  • Editing pnpm-workspace.yaml or package scripts.
  • Adding GitHub Actions workflows.
  • Moving images or rewriting existing links.

Those changes belong to the later Docusaurus setup implementation.
