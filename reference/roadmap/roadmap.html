Roadmap Tasks

Known issues, partial hardware behavior, and off-hardware validation gaps are tracked in Known Issues And Hardware Validation. Check that register before marking a roadmap item complete if the feature depends on booth hardware.

Stage 0: Project Brain​

  • Analyze inspiration app.
  • Document workflows.
  • Document design direction.
  • Document architecture and API contract.
  • Document camera and lighting plan.
  • Document expanded capabilities.
  • Document frontend/development standards.
  • Document backend runtime decision.

Stage 1: Architecture Handoff And AI Setup​

  • Create documentation index.
  • Prepare CLAUDE.md, rules, skills, and agents guidance.
  • Create root CLAUDE.md.
  • Add project rules for docs-first architecture, package version checks, semantic tokens, and hardware boundaries.
  • Add project skills for Pixelbooth architecture, frontend, backend, hardware, and review workflows.
  • Add scoped agents for architecture, backend, frontend kiosk, hardware integration, and test/release review.
  • Promote relevant design inspiration images into a formal moodboard/design brief.

Stage 2: Deliberate Project Skeleton​

See ADR-0002 for the dev-shape decision.

  • Scaffold monorepo folders only after architecture mode ends. (apps/web, apps/api)
  • Add Docker Compose-first development/runtime shape. (docker-compose.yml)
  • Add backend service interfaces and fake adapters. (CameraAdapter + FakeCameraAdapter; CameraService, DiagnosticsService)
  • Add backend API shell from the documented contract. (/api/v1 diagnostics, cameras, WebSocket events)
  • Add React kiosk/admin/gallery shell. (booth /, gallery /gallery, admin /admin)
  • Configure pnpm workspace and lockfile. (pnpm-workspace.yaml, pnpm-lock.yaml)
  • Configure Tailwind CSS, semantic theme tokens, and base themes. (kiosk + admin token themes)
  • Add frontend lint/type/test standards. (ESLint flat config, tsc strict, Vitest)
  • Add backend test standards with fake camera adapters. (pytest across all camera-availability states)
  • Add minimal local dev commands that wrap the container/dev environment. (root package.json scripts)

Stage 2.5: Unified Frontend And Admin Control Plane Foundation​

Expanded epic: stage-02-admin-control-plane-foundation.md

Sequence the foundation items first (SQLite foundation, auth gate, logging/diagnostic-id substrate), then the admin feature panels that build on them.

Foundation (do first):

  • Establish the SQLite foundation: engine, migrations, and connection lifecycle that all later catalogs (config, assets, overlays, remotes, command history, media, queue) build on. (pixelbooth/db/; ADR-0007)
  • Add a minimal admin access gate (operator secret) protecting all /admin/* and mutating admin APIs; keep guest booth/gallery reads open. See ADR-0004.
  • Add the logging + diagnostic-id foundation: structured JSON logs, stable diagnostic-id generation, and persisted/rotated local log storage (admin viewer/bundle come in Stage 10). See ADR-0006.

Admin control plane:

  • Refactor the React app around one frontend with booth, gallery, admin, setup, and future standalone display route groups.
  • Build the admin dashboard as the first admin view with readiness, system health, queue health placeholders, warnings, and safe quick actions.
  • Add admin navigation for dashboard, settings, assets, overlays, media, remotes, Commander, logs, and setup.
  • Add typed settings persistence with SQLite-backed active settings, settings templates, validation, and JSON import/export. (Backend: full templates + import/export endpoints. Frontend V1: per-section editor; templates/import-export UI deferred.)
  • Add an event entity (active event/profile that scopes overlays, QR links, and gallery) modeled in config/templates.
  • Add safe asset/file-browser API and admin UI for upload, browse, preview, zip/download, recycle/delete, and dependency-aware destructive actions. (Backend: full asset + file-browser + zip API. Frontend V1: upload/list/recycle; folder-browse + zip UI deferred.)
  • Add overlay library and PNG overlay preview against live/fake preview frames.
  • Define the future overlay designer data model and keep V1 PNG overlays compatible with it.
  • Add Commander service/API/UI foundation for allowlisted tasks, confirmations, status, logs, and audit history.
  • Add sync remote model foundation with common remote fields, type-specific settings, queue status placeholders, and remote test action.
  • Decide frontend API typing for admin surfaces: hand-written types now, with generated OpenAPI types adopted in Stage 4 once the session/media contract stabilizes.
  • Add frontend tests for admin navigation, dashboard readiness states, settings persistence flows, asset upload/browse states, overlay preview states, and Commander confirmations.

Stage 3: Raspberry Pi Camera Spike​

Expanded epic: stage-03-raspberry-pi-camera-spike.md

Booth UI design source: build from the current handoff at design-prototypes/booth-handoff/ (this stage converts the Start/Attract preview-only surface). The original concept brief remains in booth-touchscreen-design-brief.md.

  • Implement USB/V4L2 (UVC) camera adapter as the first real adapter, with a single-device role mapping and the MJPEG preview stream. (ADR-0008)
  • Add the adapter preview-frame seam (stream_frames) and serve GET /cameras/{id}/stream.mjpg as a real multipart/x-mixed-replace MJPEG response (fake + USB).
  • Verify USB camera discovery, live preview, and still capture on the Pi (blocked only on a USB webcam, not 2× Camera Module 3).
  • Implement real Picamera2 adapter for two Camera Module 3 devices, on bare-metal Pi OS. (ADR-0009) (Adapter + dual-camera role mapping landed and unit-tested off-hardware; on-Pi discovery/preview/capture verification tracked below.)
  • Verify normal and wide camera discovery on target hardware.
  • Verify degraded mode with only normal camera connected.
  • Verify degraded mode with only wide camera connected.
  • Define preset-level camera fallback policies.
  • Add guest-facing pre-capture camera switching for the live preview.
  • Verify the physical capture button uses the selected preview camera role.
  • Verify unavailable or preset-incompatible preview roles are disabled. (Component-tested off-hardware; the resolvableRoles resolver mirrors the backend. Full preset-catalog wiring lands in Stage 3.5.)
  • Verify 720p MJPEG preview at 15fps.
  • Verify still capture from both camera roles.
  • Enable Camera Module 3 autofocus (continuous default, AfSpeed=Fast) with an admin-editable focus mode/range and a manual lens_position lock (ADR-0025, KI-023). (Hardware-verified 2026-06-22.)
  • Spike exposure controls and metadata capture. (Camera-only slice — sensor AEC/AGC/AWB controls + focus/exposure metadata on the dashboard, ADR-0025/KI-006. The coordinated ExposureService with the ring light + sensors stays a Stage 5 spike.)
  • Spike dual-white ring-light control and warm/cold mix.
  • Define pre-capture metering algorithm for normal lights and club mode.
  • Add a preview-only kiosk surface: live preview, camera-role switching UI, and unavailable/incompatible-role states (no active session required). Full session-state kiosk flow lands in Stage 3.5.
  • Add kiosk E2E coverage for preview switching and unavailable-role states. (Playwright harness + e2e/booth-preview.spec.ts: real preview-switch round-trip vs fake cameras + route-mocked degraded/none states.)

Hardware/decision prerequisites: 2× Raspberry Pi Camera Module 3 (normal + wide) on target Pi (now on hand); the Pi OS/container strategy for Picamera2/libcamera access is resolved — bare-metal Pi OS (ADR-0009). Capture quality is not tuned until Stage 5 lighting; this stage is a spike plus the preview surface.

Stage 3.5: Capture Session Orchestration​

Expanded epic: stage-03b-capture-session-orchestration.md. See ADR-0003.

Booth UI design source: build from the current handoff at design-prototypes/booth-handoff/ (this stage converts Mode Select, Countdown, Processing, and the result hand-off). The original concept brief remains in booth-touchscreen-design-brief.md.

Phase A (backend orchestration), Phase B (session-state kiosk flow), and Phase C (kiosk E2E) landed — async SessionService, /sessions REST surface, and session.* events on fake cameras + a fake render port, a session.*-driven kiosk flow in apps/web, and Playwright coverage of the full session loop (plus reject/abort, 409 conflict, and error states) against fake cameras and the fake render adapter.

  • Implement SessionService and the documented session state machine and transitions. (Async owner; transition table in domain/sessions.py; see system-architecture.md §Legal transitions.)
  • Enforce the single-active-session lock and 409 conflict behavior.
  • Wire /sessions endpoints (POST, active, confirm, reject, abort).
  • Emit session.created/state/completed/error over the WebSocket contract.
  • Coordinate camera capture, exposure, lighting, and the render handoff from session transitions without blocking capture acknowledgement on rendering. (Capture off the event loop via to_thread; render via RenderPort + FakeRenderAdapter; exposure/lighting/marquee via a no-op SessionCoordinator seam until Stage 5/6.)
  • Build the session-state kiosk flow (attract → mode-select → countdown → capture → review → processing → present) driven by session.* events via a pure boothFlowReducer, with a GET /sessions/active hydrate for reload recovery. (Phase B. Mode-select ships the still "Photo" preset only; result media is not displayed until Stage 4 since fake render URLs are not yet servable.)
  • Add kiosk E2E for the full session loop, including reject/abort, 409 conflict, and error states against fake cameras and a fake render adapter. (Phase C. apps/web/e2e/booth-session.spec.ts drives the real WebSocket loop; booth-preview.spec.ts was updated for the Phase B attract→mode-select split. The 409 and error frames are forced deterministically — the fake render/camera adapters have no fault-injection seam until Stage 4.)

Stage 4: Media Processing​

Expanded epic: stage-04-media-processing.md

Booth UI design source: continue from the current handoff at design-prototypes/booth-handoff/ (this stage replaces placeholder result chrome with real media and backs the Mode Select options with real renders).

  • Add the media catalog (SQLite metadata + media file store) on the Stage 2.5 SQLite foundation, with a fake/in-memory adapter for off-hardware tests. (MediaService + migration 0007_media.sql; bytes under <data_dir>/media, SQLite authoritative for metadata; :memory: DB + temp data-dir tests.)

  • Add the shared job/queue primitive (durable, retryable, status-emitting) used by RenderWorkerPool here and reused by the Stage 8 sync outbox. (JobQueue + migration 0008_jobs.sql; emits job.progress/completed/failed; ADR-0011. Stage 8 migrates sync_jobs onto it.)

  • Add storage-capacity handling: disk-usage reporting, thresholds, and a retention/eviction policy so a busy event cannot silently fill the disk. (MediaService.capacity()/enforce_retention(); surfaced in diagnostics; SessionService blocks captures (507) when critical.)

  • Add a fake media/render adapter producing deterministic outputs so the pipeline runs off-hardware. (LocalRenderAdapter behind the RenderPort seam; deterministic vs the fake camera, no image lib required on the dev/CI path.)

  • Adopt generated TypeScript API types from the FastAPI OpenAPI schema for the frontend now that the session/media contract is stable; migrate the Stage 2.5 hand-written admin types onto it. (pnpm gen:api exports the FastAPI schema → openapi-typescript → apps/web/src/api/generated/schema.ts; guest + admin types now re-export thin aliases onto it. Only WebSocket event payloads and the error envelope stay hand-written.)

  • Generate thumbnails and previews from captures. (Instant preview on the capture path; thumb generated in the render job — real downscale via Pillow, a core dependency (ADR-0014).)

  • Add instant preview followed by background full-resolution render. (Instant preview on the fast path; render.final queued on JobQueue with progress events; processing awaits it before present.)

  • Apply V1 PNG overlays from overlay assignments during final render. (LocalRenderAdapter resolves overlays by scope precedence preset > event > global — the event scope is the active event entity resolved at render time via ConfigService — and composites the PNG into the final output during the render.final job; instant preview stays overlay-free.)

  • Add overlay render validation and safe fallback behavior. (Overlay compositing uses Pillow, a core backend (ADR-0014): the overlay scales to the frame and an overlay without an alpha channel is composited as opaque, so a valid assigned overlay is applied. Missing asset, undecodable/invalid asset, and incompatible media record overlay_status='skipped' with a logged diagnostic id; the render never fails.)

  • Add collage generation. (Multi-frame capture + compose_collage (Pillow grid, core backend); ADR-0012, ADR-0014.)

  • Add collage wall/canvas mode.

  • Add GIF/animation generation. (Multi-frame capture + compose_animation (Pillow GIF, core backend — ADR-0014).)

  • Add boomerang generation using worker/ffmpeg-style pipeline. (compose_boomerang (forward+reverse GIF via Pillow); ffmpeg-style worker pipeline reuse remains a hardware follow-up.)

  • Add short video capture and render path, with optional sound. (v1: still-sequence → browser-playable H.264 MP4 via ffmpeg (imageio-ffmpeg, core backend — ADR-0014); real motion video capture + audio deferred — see ADR-0012.)

  • Add the composed-format guest capture UX — booth Mode Select tiles for Collage/GIF/Video, posed multi-shot countdown for collage ("Shot k of N"), and the timed record-state flow for GIF/boomerang/video. (Two capture choreographies — posed (collage) vs record (gif/boomerang/video) — per ADR-0013; Session carries shot_index/shot_count, SessionResult carries final_content_type. Boomerang is folded under the GIF tile. Gallery/Present play video and animate GIF; real frame-0 thumbnail downscaling and real motion/audio capture remain deferred — ADR-0012.)

  • Add chromakey/background-removal pipeline stage.

  • Add future filter/face-landmark pipeline extension point.

  • Add capture result frontend with instant preview, render status, and share/print/gallery actions. (Partial: the Present screen shows the real rendered result as hero — <video> for video, animated <img> for GIF, still otherwise (branches on final_content_type) — and the attract counter is real. The gallery is reachable via a dedicated attract GALLERY button and its fullscreen detail view carries DELETE/QR. The Stage 8 share/QR slice has replaced the old placeholder with a live ShareQrPanel that moves from "SAVING..." to scannable once the share upload is ready; end-to-end remote verification remains tracked in KI-018. Print remains deferred to the Stage 8 print slice.)

  • Add gallery list/detail frontend with thumbnails, media type, timestamp, camera role, preset, hide/delete/download, and empty/error states inspired by the reference app gallery. (Guest gallery (features/gallery/, the only scrollable guest screen), rebuilt to the redesigned Claude direction (KI-007): an arcade scoreboard grid of landscape tiles (per-mode neon frames, stable rank, relative-time + motion badges) whose grid scrolls inside the fixed stage; tapping a tile opens a fullscreen detail view (hero render, DELETE + QR toolbar mirroring the Present screen) with prev/next navigation (arrows, swipe, keyboard; hard stops). QR is the only share path and PRINT is deferred to Stage 8, so the previous download action and the metadata info-box were dropped from the guest flow — the info-box is preserved unused as MediaInfoPanel for later. Loading/error/empty states covered; admin media panel still a placeholder.)

  • Add frontend media API client/query cache integration for capture, render, and gallery state. (api/media.ts: useMediaCount/useMediaList/useMedia/useRemoveMedia (shared ["media"] cache key with invalidation) + mediaFileUrl. Capture-result media still rides the session.state result payload.)

ADR-0026. The remaining Stage 5 items — WLED/ESP32 adapters, exposure profiles, preset-mapping persistence, calibration, and the admin lighting UI — are still open.)
  • Implement WLED adapter. (WledLightingAdapter over the WLED JSON API (POST /json/state; version-safe fx:0 solid looks; {"ps":id} operator-scene override; cct for white temperature), injectable httpx transport, fully unit-tested off-hardware. (Esp32LightingAdapter over the documented newline-JSON contract ({"command":"preset",…} → {"ok":true,…}), behind an injectable Esp32Transport; TcpEsp32Transport (stdlib socket, loopback-tested) + SerialEsp32Transport (lazy pyserial, optional esp32 extra).   • Add exposure profile UI.   • Add ring-light brightness and white-temperature calibration. (The active exposure profile's ring_brightness ceiling + warm_cold mix flow into the lighting adapter — scaling each look's brightness and driving the WLED cct (0 warm…255 cold) — and recalibrate live via the config-driven LightingService. ADR-0029. Real dual-white hardware verification tracked in known-issues.)
  • Add admin lighting frontend for fake/WLED status, discovery/manual setup, test actions, preset mapping, exposure profiles, and ring-light calibration. (A "Lighting" admin nav category with four sub-views — Status & Test (live health + per-preset test), Device (backend + WLED/ESP32 connection), Preset Mapping, and Exposure & Calibration (profile editor + ring brightness/warm-cold) — built on TanStack Query hooks (api/admin/lighting.ts), semantic-token panels, and vitest coverage. The generated OpenAPI types now include the lighting/exposure schemas.)

Stage 8: Sync, QR, And Print​

Expanded epic: stage-08-sync-qr-and-print.md

Booth UI design source: continue from the current handoff at design-prototypes/booth-handoff/ (this stage makes the Result screen's share/QR/print actions functional).

Sync transport engines are decided in ADR-0005: rclone is the default engine for cloud + SFTP, rsync/SSH serves LAN/self-hosted copies, and Immich uses its native upload path. All remotes share one common model with type-specific settings.

QR sharing is decided in ADR-0018: guest phones cannot reach the Pi at the venue, so the booth uploads each finished render to a designated share remote and the QR encodes a public URL the guest opens over their own mobile data. Build the share/sync services first (the upload engine + durable outbox + admin config is independently valuable for off-site backup/export and is QR's hard dependency); the QR slice is a follow-on on top.

Phase 1 — Share/sync services (foundation, first):

  • Add the SQLite-backed sync outbox and remote runner, reusing the Stage 4 shared job/queue primitive (durable jobs, retry/status, per-remote queue health). (Sync runs on JobQueue kind sync; jobs derived from the shared jobs table; sync_jobs retired — migration 0012.)
  • Add rclone remote as the default cloud/SFTP engine, with bandwidth, parallelism, checksum, and retry controls. Land S3-compatible end-to-end first (concretely MinIO, self-hosted; MinIO-in-Docker for local/CI), then self-hosted WebDAV/FTP/SFTP, then Google Drive — all behind the one generalized remote model. (rclone backend matrix complete: S3/MinIO, WebDAV/FTP/SFTP, and Google Drive. Passwords obscured via rclone obscure; secrets (incl. the Drive OAuth token) travel via RCLONE_CONFIG_* env, never argv. Drive auth is OAuth2 — operator supplies a token from rclone authorize "drive" plus optional client_id/client_secret; kept as a backup/sync remote, not the recommended QR backend.)
  • Add local folder/export remote. (Testable local_folder adapter.)
  • Add USB remote (behind the privileged helper boundary). (UsbAdapter copies the media tree (skipping hidden/transient files) to a mounted drive's mount_path/subdir, degrading with a user-safe "drive not mounted" message; the privileged mount/unmount stays in a host helper. Off-hardware tested; real-drive verification tracked by KI-019.)
  • Add Immich remote using its native upload path; confirm the exact endpoint against current official Immich docs at implementation time. (ImmichAdapter posts each file to POST /api/assets (verified against the current Immich OpenAPI spec) with x-api-key; validateToken for the connectivity test; idempotent via a stable deviceAssetId. Backup/sync remote, not a QR backend. Off-hardware tested via an injected transport; real-server verification tracked by KI-019.)
  • Add rsync/SSH remote for LAN/self-hosted server copies. (RsyncAdapter runs rsync over SSH in BatchMode (SSH key auth only — no password ever on argv); --mkpath for keyed single-file uploads. Off-hardware tested via an injected runner; real-host verification tracked by KI-019.)
  • Add the admin Remotes panel: remote config, designate the share remote, queue status, and remote test. (Remotes create/list/test/delete + SharingPanel: designate share remote, link mode/template/sharepage/expiry, and a live GET /sync/share-preview.)
  • Auto-share on render.final: enqueue an upload on every enabled on_capture_complete remote, off the capture path. (Render-pipeline hook → RemoteService.enqueue_capture_complete_syncs.)

Phase 2 — QR slice (on top, after Phase 1):

  • Add the media_shares table for per-media share status keyed by object key. (Migration 0013; MediaShareStore with idempotent ensure() (reuses the key on retry) and build_object_key() → {event}/{stem}-{token}.{ext} with an unguessable token. The share remote uploads each final to its key via rclone copyto/upload_file (vs the bulk dir sync).)
  • Add per-media share status, the media.shared WS event, and GET /media/{id}/qr.svg (Python qrcode 8.2). QR goes scannable only once the upload is verified live. (RemoteService routes the share remote's per-media job through a keyed upload, marks the row ready/failed, resolves the public URL, and emits media.shared {id,status,url}. Endpoints: GET /media/{id}/share, GET /media/{id}/qr.svg (200 SVG when ready, 409 pending, 404 disabled), POST /media/{id}/share (re-enqueue). Self-contained black-on-white SVG with quiet zone for scan reliability.)
  • Add the static arcade sharepage (apps/sharepage/; client-side only, no server-side script — distinct from the Codex-owned apps/docs) that reads ?id={key}, renders the media, and offers a big arcade SAVE button. (Single self-contained index.html — no build step; resolves the media relative to its own URL (same host serves page + uploads), detects image/GIF/video, reveals SAVE with a download filename, and degrades to friendly loading/error states. Smoke-verified via static server + Playwright.)
  • Replace the placeholder QR in the kiosk Present screen and gallery detail with the live QR inside an arcade frame (pending "SAVING…" → scannable; offline/degraded states; QR modules stay black-on-white for scan reliability). (Shared features/share/ShareQrPanel: useMediaShare polls GET /media/{id}/share (settles on ready/failed/disabled), renders GET /media/{id}/qr.svg on a white card, and offers retry via POST /media/{id}/share. Wired into PresentScreen (via SessionResult.media_id) and gallery MediaDetail.)
  • Add the share-host setup helper (local operator shell/Python script) that pushes the static sharepage + configures the rclone upload target and outputs the exact public_link_template / sharepage_url / rclone config to paste into admin. The host stays a dumb static/file server. (scripts/share-host-setup/sharehost.py: a local Python CLI that deploys apps/sharepage via rclone copyto (S3/MinIO, WebDAV, FTP, SFTP — same backend keys as the API) and prints the admin Remotes + Sharing values; secrets via PB_* env, passwords obscured, --dry-run previews. Operator guide + tests alongside.)

Print (after sharing):

  • Add CUPS/IPP print adapter and Canon Selphy target.
  • Add print crop/safe-area preview.
  • Define future payment-provider gate for print jobs.
  • Add sharing/sync/print frontend for remote setup, queue status, remote tests, guest QR/share actions, and print progress/error states inspired by the reference app share and QR flows.

Stage 10: Observability​

Expanded epic: stage-10-observability.md

The logging + diagnostic-id foundation (structured JSON logs, stable diagnostic ids, persisted/rotated local storage) ships in Stage 2.5. This stage builds the operator-facing surfaces on top of it. Observability is local-first and there is no Sentry — see ADR-0006.

  • Add admin log viewer with filtering and diagnostic-id lookup.
  • Connect user-visible errors to their diagnostic ids end-to-end (kiosk/admin/gallery).
  • Add safe diagnostic bundle support (version, config summary, recent logs, queue/hardware status, task history, secrets redacted).
  • Add Commander task history and output summaries to diagnostics.
  • Add admin diagnostics frontend with appliance status cards, log filtering, diagnostic-id lookup, and safe error boundaries inspired by the reference app status/admin pages.
