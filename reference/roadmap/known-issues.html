Known Issues And Hardware Validation

Pixelbooth is often built away from the physical booth hardware. This register tracks known issues, partial behavior, design follow-ups, and features that exist in code or design but still need target-device validation.

Use this page when a feature is "done" off-hardware but still carries real-world risk: camera timing, Pi OS behavior, browser kiosk quirks, USB/CSI differences, GPIO, lighting, printing, sync storage, or anything that depends on the physical booth.

How To Use This Register​

Add an entry when:

  • A feature works with fake adapters, mocks, unit tests, or local development but has not been verified on the target Raspberry Pi setup.
  • A feature works on a laptop but behaves differently on booth hardware.
  • A hardware-backed feature fails, degrades, or produces surprising results.
  • A roadmap checkbox says "verified" but the evidence is missing or stale.

Move an entry out of the active register when:

  • The behavior is verified on target hardware and linked to the owning doc, test, issue, or PR.
  • The gap is converted into a roadmap task with enough detail to execute.
  • The feature is deliberately removed or deferred.

Status Values​

StatusMeaning
UnverifiedBuilt or designed, but not checked on target hardware yet.
BrokenTested on hardware and does not work as intended.
PartialWorks in one hardware state but not all required states.
BlockedCannot be validated until hardware, access, setup, or a dependency exists.
VerifiedChecked on target hardware with notes and evidence.

KI-012Stage 3 / Stage 7 / Stage 10Shutdown and app restart handling for camera streamsPartialTarget Pi, real camera streams, app restart/shutdown pathRoot-caused: the live preview is an endless MJPEG response, and the Pi launched the API with a bare python -m uvicorn pixelbooth.main:app. uvicorn's graceful shutdown waits for the open stream.mjpg to finish, so the first Ctrl+C hung; the forced second Ctrl+C exited without running the FastAPI lifespan, so CameraService.close() never ran and the CSI/V4L2 handle leaked (Pipeline handler in use by another process, Camera __init__ sequence did not complete) — restart needed a reboot (uvicorn #1579). Fix (ADR-0023): the Pi runs the new python -m pixelbooth entrypoint. The booth keeps two endless responses open (the MJPEG preview and the /events WebSocket); a first attempt to end them cooperatively under uvicorn's graceful drain proved fragile on the Pi and made shutdown worse — the /events socket still stalled the full timeout_graceful_shutdown and force-cancelled tasks with CancelledError tracebacks, and the process then hung after Application shutdown complete and never quit. Revised 2026-06-21 to the reference photobooth's approach (uvicorn #1579): the entrypoint sets server.force_exit = True, so uvicorn abandons both endless responses on the first signal (no graceful-drain wait, no force-cancel). Because force_exit skips the FastAPI lifespan, handle_exit calls CameraService.close() synchronously in the signal handler to release the cameras; and because a non-daemon libcamera/V4L2 thread can outlive Picamera2.close() and block interpreter shutdown forever, main() then os._exit(0)s so the process actually dies and frees the camera fds. Verified by entrypoint tests; not yet validated on the Pi.Re-run the restart/lifecycle checks on hardware: open a live preview and a screen that holds the /events socket, then a single Ctrl+C — confirm the app exits promptly without a second interrupt, no graceful-timeout stall or CancelledError tracebacks, the process fully exits (no hang after the last log line), the CSI handles are released (no Pipeline handler in use), and the app restarts cleanly without rebooting. Verification Session Checklist​

Before testing:

  • Pull the branch or build that contains the feature.
  • Record Pi model, OS image, camera devices, browser, and app version/commit.
  • Start the API and web app in the same mode an event would use.
  • Keep logs open or confirm where diagnostic logs will be written.

During testing:

  • Test the happy path first, then degraded or failure states.
  • Record whether behavior differs between fake/local development and hardware.
  • Capture measurements for timing-sensitive paths: FPS, latency, CPU, memory, bandwidth, queue time, and capture-to-result time.
  • Save at least one concrete artifact for broken or partial behavior: log line, screenshot, photo, video, media id, diagnostic id, or exact command output.

After testing:

  • Update the active register row.
  • Update the owning roadmap or architecture doc if the expected behavior changed.
  • Convert unresolved hardware failures into issues or roadmap tasks before the context gets cold.
