Validation Test Plan

Pixelbooth is built mostly away from the physical booth and away from real share infrastructure. A large body of work is "done" against fake adapters, unit tests, and local development, but has never been exercised on the target Raspberry Pi, a real camera, a real share remote, or a phone on mobile data.

This plan turns that backlog into ordered test sessions grouped by the setup each one needs, so they can be run in efficient batches. It is the execution companion to known-issues.md: every session names the known-issue (KI-###) rows it validates, and every result should flow back into that register.

How To Use This Plan​

  1. Pick the lowest-numbered session whose prerequisites you can satisfy.
  2. Record the environment baseline (Session 0) once per test day.
  3. Work the session steps top to bottom: happy path first, then degraded and failure states.
  4. Capture the listed measurements and at least one concrete artifact per finding (log line, screenshot, photo, media id, diagnostic id, command output).
  5. Type findings into test-notes.inbox.md as you go — rough is fine. The known-issues-intake Codex agent (.agents/agents/known-issues-intake.md) normalizes those notes into proper known-issues.md rows afterwards, so you do not have to format anything mid-test.

Out of scope for this plan: KI-016 (Docusaurus documentation site) — owned by Codex on a separate branch.

Session Block Format​

Each session uses the same shape:

  • Goal — what "verified" means for this session.
  • Setup / prerequisites — exact hardware, accounts, and config needed.
  • Steps — ordered, repeatable actions.
  • Measurements — numbers to capture for timing-sensitive paths.
  • Pass / fail criteria — the bar for moving a KI toward Verified.
  • Artifacts — what to save as evidence.
  • KI rows touched — which register entries this session updates.

Status vocabulary (Unverified, Broken, Partial, Blocked, Verified) matches known-issues.md.

Session 0 — Bring-Up And Environment Record​

Goal: a running dev stack on the target Pi and a recorded baseline so later sessions are reproducible.

Setup / prerequisites:

  • Target Raspberry Pi with a fresh Raspberry Pi OS image.
  • Repository cloned at the branch/commit under test.
  • Network access from a second device (laptop/phone) on the same LAN.

Steps:

  1. Review the editable variables, then run first-boot prep: sudo bash ./pi-first-boot.sh (see raspberry-pi-setup.md).
  2. Run app/camera setup as the normal user: ./pi-pixelbooth-setup.sh setup.
  3. Run ./pi-pixelbooth-setup.sh doctor and read the report.
  4. Start the stack: ./pi-pixelbooth-setup.sh run.
  5. From a second device, open Booth (http://<pi>:5173/), Admin (http://<pi>:5173/admin), and the cameras API (http://<pi>:8000/api/v1/cameras).

Measurements: none — this is a baseline, not a benchmark.

Pass / fail criteria: doctor reports the expected interpreters, venv, and camera tooling; Booth and Admin load over the LAN; the cameras API responds.

Artifacts: save ~/firstboot-report.txt, the full doctor output, and a note of: Pi model, OS image, camera devices present, kiosk browser + version, and the app commit hash. This is the "before testing" record every later session refers back to.

KI rows touched: none directly; this is a precondition for Sessions 1–4.

Session 1 — Camera Spike​

Goal: confirm camera discovery, live preview, still capture, and the hardware trigger behave on the real Pi across every camera-availability state.

Setup / prerequisites:

  • Session 0 complete.
  • One or more real cameras. Where possible, run the matrix below across: normal-only, wide-only, both, and no-real-camera.
  • For KI-004: the physical capture button wired to the trigger path.

Steps:

  1. USB / V4L2 (KI-001): with a USB/UVC webcam, run in USB mode (PIXELBOOTH_CAMERA_BACKEND=usb PIXELBOOTH_USB_CAMERA_DEVICE=0 PIXELBOOTH_USB_CAMERA_ROLE=normal ./pi-pixelbooth-setup.sh run). Confirm discovery, MJPEG preview stability, and still capture output.
  2. Dual CSI (KI-002): with Camera Module 3 normal + wide, run the default Picamera2 backend. Verify normal/wide discovery, the role→index mapping, single-camera degraded states (unplug one), dual-role behavior, and the capture handoff.
  3. Preview performance (KI-003): with a real feed at the documented 720p / ~15fps budget, record the measurements below under normal booth lighting and load.
  4. Trigger follows role (KI-004): switch the selected preview role in the UI, fire the physical button, and confirm the capture used the selected role (compare captured metadata/output, not just the on-screen preview).

Measurements: preview FPS, end-to-end preview latency, preview bandwidth, CPU load during preview, and capture-to-result time. Note any difference between fake/local development and the Pi.

Pass / fail criteria: each available camera state previews and captures without crashing; a missing camera degrades rather than crashes; the physical trigger demonstrably uses the selected role.

Artifacts: a captured still per camera state, the measurement table, and any API error / diagnostic id seen.

KI rows touched: KI-001, KI-002, KI-003, KI-004.

Blocked sub-items (do not invent test steps):

  • KI-005 (preset fallback policy) — Blocked until the required / preferred / fallback / unavailable role policy is defined. Decision to resolve first: what each preset does when its preferred camera role is absent. Capture that decision in camera-lighting.md before treating preset capture as testable.
  • KI-006 (exposure metadata) — Blocked; exposure is a spike item. Decision to resolve first: which exposure settings and diagnostic metadata are worth capturing. Prototype on hardware, then document before writing a repeatable test.

Session 2 — Media Render Re-Verify​

Goal: confirm the render fixes that landed off-device (ADR-0014..0017) actually produce browser-playable, correctly-sized output on a real Pi capture, and that the camera survives an app restart.

Setup / prerequisites:

  • Session 0 complete, with at least one real camera (Session 1 ideally passed).
  • A browser to view kiosk + gallery playback.
  • A known full-frame overlay asset configured.

Steps:

  1. Video playback (KI-008): capture a video item, let it render, and play it back in both the kiosk and the gallery. Confirm the ffmpeg H.264 / yuv420p MP4 plays (the old mp4v output did not).
  2. GIF / boomerang feel (KI-009): capture a gif and a boomerang. Confirm the length and motion feel right at real Pi capture rates (frame-count / fps budget from ADR-0014: animation ≈ 2.0s, boomerang ≈ 2.8s, video ≈ 3.0s). Note whether the counts/fps need tuning to the actual capture rate.
  3. Overlay scaling (KI-014): apply a known full-frame overlay to a real capture and confirm it scales to the frame and composites correctly in the final (including an overlay with no alpha channel).
  4. Restart / lifecycle (KI-012): capture once, restart the app, and confirm the camera reacquires without unplugging. Then force a held device and confirm the user-safe lock diagnostic (with a diagnostic id) appears.

Measurements: capture-to-final render time per format; gif/boomerang/video output durations as actually produced.

Pass / fail criteria: video plays in both browsers; gif/boomerang durations match the budget within tuning tolerance; the overlay is correctly placed and scaled; the camera reacquires after restart; the lock diagnostic appears only when a device is genuinely held.

Artifacts: the rendered media ids, a screen recording or screenshot of playback, the overlaid final, and the lock diagnostic log line + id.

KI rows touched: KI-008, KI-009, KI-012, KI-014.

Session 3 — Sharing And QR End-To-End​

Goal: prove the full guest share path against a real remote and a phone that is not on the booth LAN.

Setup / prerequisites:

  • A configured, network-reachable share remote — self-hosted S3/MinIO first, or WebDAV/FTP/SFTP.
  • A deployed static sharepage (apps/sharepage).
  • A phone on mobile data (not the booth Wi-Fi).
  • See ADR-0018.

Steps:

  1. Run scripts/share-host-setup (sharehost.py) against the real MinIO/WebDAV host.
  2. In Admin, add the remote, designate it the share remote, and confirm the link preview.
  3. Capture an item and watch the sync outbox / upload job succeed.
  4. Watch the kiosk QR flip from "SAVING…" to a scannable code.
  5. Scan it from the phone on mobile data → open the grab page → press SAVE and confirm the media downloads.
  6. Re-share an already-shared item and confirm the deterministic keyed upload (rclone copyto) does not duplicate.

Measurements: capture → "shared" latency (the time the QR shows "SAVING…"); upload job duration.

Pass / fail criteria: the upload job succeeds; the QR becomes scannable; an off-LAN phone reaches the public URL and downloads the media; reshare dedupes.

Artifacts: the share remote config used (redact secrets), the media.shared event / media-share status, a photo of the scannable kiosk QR, and a screenshot of the grab page on the phone.

KI rows touched: KI-018.

Session 4 — Remote Transports​

Goal: validate the three non-rclone transports behind the common remote model against real endpoints.

Setup / prerequisites:

  • An Immich server reachable over the network plus an API key.
  • An SSH host on the LAN with the Pi's key installed.
  • A real USB drive that can be mounted on the Pi.
  • See ADR-0005.

Steps:

  1. Immich: configure the immich remote in Admin, run "test remote", then a capture/sync. Confirm the asset appears in the Immich library and that a re-upload dedupes.
  2. rsync over SSH: configure the rsync_sftp remote, run "test remote", then a capture/sync. Confirm files land on the SSH host and the key-auth path works under BatchMode (no interactive prompt).
  3. USB: with the drive mounted, configure the usb remote and run a capture/sync. Confirm the export tree appears on the drive. Then unmount and confirm the "not mounted" message shows.

Measurements: per-transport sync duration for one item.

Pass / fail criteria: each transport's "test remote" passes; a capture lands at the destination; Immich dedupes; rsync runs non-interactively; USB shows the correct "not mounted" message when absent.

Artifacts: a screenshot/listing of the asset at each destination, the "test remote" result, and the USB "not mounted" message.

KI rows touched: KI-019.

Coverage Gaps To Confirm​

  • Printing. Stage 8 is titled "Sync, QR, and Print", but there is no print known-issue and no print test session here. Confirm whether any print path exists yet; if it does, open a new KI-### and add a session. If it does not, it is future roadmap work, not a validation gap.

After Each Session​

  • Drop findings into test-notes.inbox.md.
  • Run the known-issues-intake agent to normalize them into known-issues.md (assign IDs, set status, write the next action). It edits the register in place — review and commit yourself.
  • Update the owning architecture / roadmap doc if the expected behavior changed.
