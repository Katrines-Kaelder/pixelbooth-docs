Stage 2.5: Unified Frontend And Admin Control Plane Foundation

Goal​

Turn the current booth/gallery/admin shell into one deliberate frontend and admin control plane before feature panels scatter across later stages.

Context​

The reference frontend source revealed important product structure that was missing from the first roadmap: a real admin dashboard, persistent admin navigation, file upload/browser, schema-assisted configuration, live preview overlays, setup pages, logs, multicam tools, and confirmed system commands.

Pixelbooth should keep the concepts, not copy the implementation. The target is a faster, clearer React/TypeScript app with typed backend services, safe admin boundaries, and better operator UX.

Implementation Guidance​

  • Start with admin-control-plane.md.
  • Sequence the foundation first (SQLite foundation, admin auth gate, logging/diagnostic-id substrate), then the admin feature panels that depend on it. The foundation is small but unblocks everything else and removes the security/observability ordering gaps.
  • Keep booth, gallery, admin, setup, and future standalone views in one React app.
  • Prefer custom UI for high-value controls and schema-assisted forms for advanced settings.
  • Use SQLite as the authoritative store for active settings, templates, assets, remotes, command history, and queue state. This stage establishes the SQLite engine/migrations/connection lifecycle that later catalogs (including the Stage 4 media catalog) build on.
  • Protect all /admin/* and mutating admin APIs behind a minimal operator-secret gate from the first admin surface; keep guest booth/gallery reads open. See ADR-0004.
  • Stand up the logging + diagnostic-id substrate here so diagnostic ids referenced from Stage 4 onward have a real backing store. The admin viewer/bundle come in Stage 10. See ADR-0006.
  • Use JSON import/export for settings templates, event/rental profiles, overlay definitions, marquee definitions, and backups.
  • Keep admin file browsing under safe roots only.
  • Use allowlisted Commander tasks instead of arbitrary shell commands configured in the UI.
  • Type admin frontend surfaces by hand for now; adopt generated OpenAPI types in Stage 4 once the session/media contract stabilizes, to avoid churning generated types against an unstable contract.
  • Preserve fake/off-hardware paths so admin work can be developed before Pi hardware is available.

Acceptance Checks​

  • The SQLite foundation (engine, migrations, connection lifecycle) is in place and reused by every catalog introduced in this stage.
  • All /admin/* routes and mutating admin APIs require the operator secret; guest booth/gallery reads work without it; off-hardware dev has a documented default credential or bypass.
  • Structured JSON logs are written with diagnostic ids, persisted and rotated locally, and survive an API restart.
  • Admin dashboard is the first /admin view and gives a clear readiness answer.
  • Admin navigation has stable destinations for dashboard, settings, assets, overlays, media, remotes, Commander, logs, and setup.
  • Settings changed in admin persist across API restarts.
  • Settings templates can be saved, listed, exported, imported, and applied after validation.
  • Uploaded overlay PNGs are recorded as assets and can be previewed over a fake or live camera frame.
  • File browser cannot escape allowed roots.
  • Commander runs only allowlisted tasks and records status/output history.
  • Sync remotes have a common model and at least a fake/testable remote path before provider-specific work starts.
  • Frontend tests cover loading, empty, success, error, and confirmation states for the new admin surfaces.

Features​

Foundation (do first)​

The control plane sits on a small foundation that also resolves the security and observability ordering gaps.

Tasks​

  • Task: Establish the SQLite foundation.
    • Add the database engine, migration mechanism, and connection lifecycle that every later catalog (config, assets, overlays, remotes, command history, media, queue) builds on. Provide a fake/in-memory path for off-hardware tests.
  • Task: Add a minimal admin access gate.
    • Protect all /admin/* and mutating admin APIs with an operator secret behind a single API enforcement boundary; keep guest reads open; back the secret with environment-backed bootstrap settings, not the SQLite settings it protects. See ADR-0004.
  • Task: Add the logging + diagnostic-id foundation.
    • Emit structured JSON logs with event/context fields and stable diagnostic ids; persist and rotate logs locally so field diagnosis works offline. Admin viewer/bundle are Stage 10. See ADR-0006.

Unified frontend route model​

The React app exposes one coherent route tree.

Tasks​

  • Task: Add admin layout and navigation.
    • Add persistent admin navigation for dashboard, settings, assets, overlays, media, remotes, Commander, logs, and setup.
  • Task: Prepare route groups for future standalone displays.
    • Reserve route structure for slideshow/gallery companion screens without creating a separate app.
  • Task: Keep kiosk constraints intact.
    • Ensure guest capture screens remain non-scrolling at 1280x800; admin routes may scroll.

Dashboard​

The dashboard is the operator's first view.

Tasks​

  • Story: Admin can see whether the booth is ready.
    • Show cameras, storage, temperature, network, lighting, overlays, remotes, printer, queue placeholders, version, and recent errors.
  • Story: Admin can run safe readiness checks.
    • Expose camera test capture, lighting test, overlay preview, remote test, printer test, and diagnostic bundle actions where available.
  • Task: Add dashboard event projection.
    • Use REST for initial state and WebSocket events for status updates.

Settings and templates​

Settings are typed, persistent, and portable.

Tasks​

  • Task: Add ConfigService.
    • Store active settings in SQLite with schema version, validation, revision metadata, and secret redaction boundaries.
  • Task: Add settings template model.
    • Support named event, rental, venue, and recovery templates.
  • Task: Add the event entity.
    • Model the active event/profile that scopes overlays, QR links, and the gallery, with start/reset semantics so later overlay assignment (this stage), media (Stage 4), QR/sharing (Stage 8), and rental reset (Stage 7) attach to one consistent concept rather than redefining "event" each time.
  • Task: Add JSON import/export.
    • Import/export active settings and templates as validated JSON.
  • Story: Admin can edit settings by section.
    • Build initial settings sections for booth UI, capture presets, overlays, remotes, and advanced system settings.

Assets and file browser​

Operators can manage booth assets without terminal access.

Tasks​

  • Task: Add AssetService.
    • Track uploaded assets in SQLite and store files under safe roots.
  • Story: Admin can upload and browse assets.
    • Support upload, folder listing, preview/download, zip export, and delete/recycle with dependency checks.
  • Task: Enforce safe roots.
    • Allowed roots include overlays, backgrounds, fonts, branding, marquee, media, exports, and backups.

Overlays​

V1 overlays are transparent PNG assets with live/admin preview.

Tasks​

  • Task: Add OverlayService.
    • Model overlays as records with source type, output dimensions, asset id, assignment scope, and validation status.
  • Story: Admin can preview an overlay.
    • Preview PNG overlays over a fake/live preview frame or latest captured frame.
  • Story: Admin can assign overlays.
    • Assign overlays globally, per event/template, or per capture preset.
  • Task: Define overlay designer schema.
    • Document .pixelbooth-overlay.json with layer ids, output size, asset references, text/image/QR placeholder layers, transform data, and safe-area metadata.

Commander​

Commander replaces broad command buttons with safe, typed tasks.

Tasks​

  • Task: Add CommandService.
    • Define allowlisted tasks with id, label, category, risk level, input schema, timeout, executor, and confirmation requirement.
  • Story: Admin can run confirmed tasks.
    • Add Commander UI with confirmation, running status, output summary, and result history.
  • Task: Add fake executors for development.
    • Implement fake reload services, diagnostic bundle, camera test, remote test, and queue retry tasks.

Sync remote foundation​

Remote sync starts with a common model before provider adapters.

Tasks​

  • Task: Add remote model.
    • Store name, type, enabled flag, output selection, trigger policy, retry policy, status, and type-specific settings.
  • Story: Admin can create and test a fake/local remote.
    • Provide a fake/local adapter that exercises validation and queue status without external services.
  • Task: Prepare provider-specific adapters.
    • Leave clear extension points for USB, rclone, Immich, rsync/SFTP, and future webhook remotes.
