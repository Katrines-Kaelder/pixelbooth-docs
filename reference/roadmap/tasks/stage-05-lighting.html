Stage 5: Lighting

Goal​

Lighting is controllable, testable, and calibratable through explicit adapters and admin surfaces.

Context​

Bring lighting under explicit service/adapter control, with fake development support and real WLED/ESP32 calibration paths. Pixelbooth is local-first Raspberry Pi booth software; preserve fake/off-hardware paths and service boundaries.

Implementation Guidance​

  • Start with CLAUDE.md and the references below.
  • Keep scope limited to this epic and its child features.
  • Keep hardware and privileged operations behind backend services, adapters, or host helpers.
  • Reuse fake-adapter patterns so off-hardware development remains reliable.

Acceptance Checks​

  • Docs are updated before or alongside behavior changes, and the owning docs are listed in the PR/summary.
  • Hardware or privileged operations stay behind backend services, adapters, or host helpers.
  • Frontend work uses semantic tokens and respects kiosk no-scroll constraints for guest-facing screens.
  • Fake/off-hardware tests or verification steps cover the riskiest behavior; real-hardware checks are documented when required.

Features​

Lighting abstraction and fake development support​

Lighting behavior can be developed and tested without physical lights.

Tasks​

  • Task: Add fake lighting adapter for off-hardware development and tests.
    • Mirror the camera fake-adapter pattern and expose predictable state/events for UI and backend tests.

WLED setup and preset mapping​

Operators can discover, configure, and map WLED lights to booth presets.

Stories And Tasks​

  • Task: Implement WLED adapter.
    • Add WLED control behind the lighting service boundary with timeouts, diagnostics, and fake-compatible interfaces.
  • Story: Admin can discover or manually configure WLED devices.
    • As an admin, I can discover or manually configure WLED devices with clear validation and diagnostics.
  • Story: Admin can map lighting presets in the UI.
    • As an admin, I can map lighting presets in the UI and persist mappings through the backend service.

ESP32 control and exposure calibration​

Custom lights and camera exposure profiles can be tuned together for real booth conditions.

Stories And Tasks​

  • Task: Add custom ESP32 serial/TCP adapter.
    • Define serial/TCP communication, failure handling, and test doubles before wiring real hardware.
  • Story: Admin can configure exposure profiles in the UI.
    • As an admin, I can configure exposure profiles in a way that aligns with camera roles, lighting modes, and degraded hardware states.
  • Task: Add ring-light brightness and white-temperature calibration.
    • Provide calibration workflow/storage for brightness and warm/cold balance, with defaults and verification guidance.

Admin lighting frontend​

The admin UI should make lighting setup approachable for bar volunteers while retaining the power of the reference app WLED/admin configuration screens.

Stories And Tasks​

  • Story: Admin can test fake and WLED lighting from the browser.
    • As an admin, I can see connection/status feedback and run safe test actions for fake and WLED lighting without touching backend internals.
  • Task: Build preset mapping controls and validation.
    • Add UI for mapping booth session states and capture presets to lighting profiles, including validation for missing devices, unavailable channels, and unsaved changes.
  • Task: Build exposure and ring-light calibration screens.
    • Add admin screens for exposure profiles, brightness, and warm/cold ring-light calibration with clear before/after feedback and safe defaults.
