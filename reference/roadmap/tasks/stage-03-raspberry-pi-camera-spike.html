Stage 3: Raspberry Pi Camera Spike

Goal​

Real camera hardware can be exercised safely on target Raspberry Pi hardware while fake adapters keep local development reliable.

Design Source - Convert The Handoff​

Build the booth UI from the current handoff. The agreed visual and interaction reference is design-prototypes/booth-handoff/. The original concept and generation prompt remain in booth-touchscreen-design-brief.md.

This stage converts the Start / Attract preview-only surface (live-preview background, normal/wide camera-role switch, unavailable/incompatible-role states). Rebuild it in apps/web with Tailwind + semantic tokens (use the palette→token Do not import prototype code; re-express the handoff through Tailwind, semantic tokens, and reusable apps/web components while preserving the 1280x800 no-scroll guarantee.

Context​

Integrate real Raspberry Pi Camera Module 3 hardware while preserving fake/off-hardware development and every degraded camera state. Pixelbooth is local-first Raspberry Pi booth software; preserve fake/off-hardware paths and service boundaries.

This stage is a hardware spike plus a preview-only kiosk surface. It deliberately does not complete the guest capture loop: session orchestration and the full session-state kiosk flow (countdown → capture → review → processing → present) belong to Stage 3.5 (see ADR-0003), and capture quality is not tuned until Stage 5 lighting. "Stage 3 done" means cameras, degraded mode, preview/role selection, the capture trigger, and the exposure spike work — not a polished capture experience.

Hardware/decision prerequisites: 2× Raspberry Pi Camera Module 3 (normal + wide) on the target Pi (now on hand); the Pi OS/container strategy for Picamera2/libcamera access is resolved — bare-metal Pi OS with the system python3-picamera2 package (ADR-0009).

USB/V4L2 brought forward — ADR-0008. The first real camera adapter is USB/V4L2 (UVC) over OpenCV, ahead of Picamera2, because it can be verified on a single USB webcam today and sidesteps the (then-open) libcamera/container question. A single USB camera maps to one configurable role and reuses the existing normal_only/wide_only degraded path.

Picamera2/CSI implemented — ADR-0009. The dual CSI Camera Module 3 adapter now runs on bare-metal Pi OS using the system python3-picamera2 package (a --system-site-packages venv), mapping camera_num indices to the normal/wide roles. It is the first backend that drives true both/dual modes. On-Pi discovery/preview/capture verification is the remaining work. The exposure/ring-light items here stay design-only (no ESP32 hardware) until Stage 5.

Implementation Guidance​

  • Start with CLAUDE.md and the references below.
  • Keep scope limited to this epic and its child features.
  • Reuse current patterns in apps/web, services/api, adapters, events, and tests.
  • Do not pull future-stage work forward unless the docs make it necessary.

Acceptance Checks​

  • Docs are updated before or alongside behavior changes, and the owning docs are listed in the PR/summary.
  • Both-cameras, normal-only, wide-only, and no-real-camera states are handled or explicitly marked not applicable.
  • Hardware or privileged operations stay behind backend services, adapters, or host helpers.
  • Live preview meets a documented budget: 720p MJPEG at ~15fps with measured latency, CPU load, and bandwidth recorded on target hardware; still-capture-to-handoff time is measured and recorded.
  • Fake/off-hardware tests or verification steps cover the riskiest behavior; real-hardware checks are documented when required.

Features​

Camera hardware adapter and discovery​

Picamera2-backed capture and discovery are available behind the existing camera adapter boundary.

Tasks​

  • Task: Implement real Picamera2 adapter for two Camera Module 3 devices. (Done — ADR-0009.)
    • Added Picamera2Adapter behind the existing camera service (fake adapters intact), with a role → camera_num map and Picamera2's native JPEG path. Host/container assumption: bare-metal Pi OS + system python3-picamera2 in a --system-site-packages venv. On-Pi discovery/preview/capture verification remains (next task).
  • Task: Verify normal and wide camera discovery on target hardware.
    • Run on the real booth hardware and record how normal/wide roles are detected, named, and reported through the API.
  • Task: Verify 720p MJPEG preview at 15fps.
    • Exercise the live preview path on target hardware and confirm bandwidth, latency, CPU load, and browser compatibility are acceptable.
  • Task: Verify still capture from both camera roles.
    • Capture still images from normal and wide roles and confirm metadata, file handoff, and error reporting match the API contract.

Degraded camera availability and fallback policy​

The booth behaves predictably with both cameras, either single camera, or no real camera.

Stories And Tasks​

  • Story: Operator can run normal-only degraded camera mode.
    • As an operator, I can run degraded mode with only the normal camera connected so the booth remains usable when the wide camera is unavailable.
  • Story: Operator can run wide-only degraded camera mode.
    • As an operator, I can run degraded mode with only the wide camera connected so the booth remains usable when the normal camera is unavailable.
  • Task: Define preset-level camera fallback policies.
    • Document and implement how presets choose between required, preferred, fallback, and unavailable camera roles.
  • Story: Guest cannot select unavailable or preset-incompatible preview roles.
    • As a guest, unavailable or preset-incompatible preview roles are disabled so I only choose capture paths that can succeed.

Guest preview selection and capture trigger​

Guest-selected preview role drives the physical and UI capture path unless a preset explicitly overrides it.

Stories​

  • Story: Guest can switch the live preview camera before capture.
    • As a guest, I can switch the live preview camera before capture so I can choose the framing I prefer.
  • Story: Physical capture button uses selected preview camera role.
    • As a guest, the physical capture button uses my selected preview camera role so hardware and UI capture behavior match.

Exposure and pre-capture metering spike​

The team understands how camera exposure and dual-white lighting can be coordinated before full lighting automation.

Tasks​

  • Task: Spike exposure controls and metadata capture.
    • Prototype exposure configuration and persist useful metadata for diagnostics, tuning, and future render decisions.
  • Task: Spike dual-white ring-light control and warm/cold mix.
    • Validate how warm/cold ring light channels can be controlled and represented before productionizing lighting presets.
  • Task: Define pre-capture metering algorithm for normal lights and club mode.
    • Document the first metering approach for normal and club lighting modes, including fallback behavior and what data is needed from camera/lighting services.

Preview-only kiosk surface​

The guest-facing booth UI in this stage is preview-first, inspired by the reference app frontpage: live preview, camera-role switching, and clear unavailable/incompatible-role states, with no admin complexity in the guest path. It does not require an active session — the full session-state flow (countdown → capture → review → processing → present) lands in Stage 3.5.

Stories And Tasks​

  • Story: Guest can see live preview and choose a camera role.
    • As a guest, I can see the live preview and switch camera roles before capture so I can choose my framing.
  • Task: Build the camera role selector and unavailable-role UI.
    • Implement kiosk controls for normal/wide camera selection, unavailable roles, and preset-incompatible states without breaking the 1280x800 no-scroll guest constraint.
  • Task: Add kiosk E2E coverage for preview switching and unavailable-role states.
    • Cover preview switching and unavailable/incompatible-role behavior with fake camera states. Countdown/capture/session-loop E2E belongs to Stage 3.5.
