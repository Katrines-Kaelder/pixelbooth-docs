Stage 3.5: Capture Session Orchestration

Progress​

Delivered in incremental PRs:

  • Phase A — backend orchestration (done): async SessionService (state machine + single-active-session lock + 409), /sessions REST surface, session.created/state/completed/error events, capture off the event loop, and a RenderPort + FakeRenderAdapter render handoff, all on fake cameras. See system-architecture.md §Legal transitions / §Orchestration mechanism.
  • Phase B — session-state kiosk flow (done): attract → mode-select → countdown → capture flash → review (confirm/reject) → processing → present screens in apps/web, derived by a pure boothFlowReducer from the backend session.* WebSocket events (the single source of truth) plus a one-shot GET /sessions/active hydrate for reload recovery. Mode-select ships the still "Photo" preset only (framing selects selfie-normal/selfie-wide); Collage/GIF/Video are shown but disabled until their pipelines exist. Built on Tailwind + semantic tokens, 1280x800 no-scroll. Result media is not displayed yet (fake render URLs aren't servable until Stage 4); the screens are driven by session state, not result bytes. Superseded the Stage-3 TestCapture dev widget.
  • Phase C — kiosk E2E (done): Playwright drives the booth in a real browser against the fake-camera backend and the FakeRenderAdapter, proving the session.* WebSocket events move the guest screen through the full loop (attract → mode-select → countdown → capture → review → processing → present) plus retake (reject), an out-of-band abort, the single-active-session 409, and the recoverable error screen. apps/web/e2e/booth-session.spec.ts holds the loop; booth-preview.spec.ts was rewritten for the Phase B attract→mode-select split (the role selector moved into Mode Select). The 409 and session.error frames are forced deterministically (route + routeWebSocket) because the fake render/camera adapters expose no fault-injection seam yet — Stage 4's real render pipeline can replace those stand-ins with a genuine failure path.

Goal​

One service owns the capture session: the state machine, the single-active-session lock, and the session.* events that drive the guest-facing booth experience.

Design Source - Convert The Handoff​

Build the booth UI from the current handoff. The agreed visual and interaction reference is design-prototypes/booth-handoff/. The original concept and generation prompt remain in booth-touchscreen-design-brief.md.

This stage converts the session-driven screens — Mode Select, Countdown + capture flash, Processing, and Present/result hand-off — and wires START → session to real session.* events instead of the prototype's local state machine. Rebuild in apps/web with Tailwind + semantic tokens. Do not import prototype code; preserve the 1280x800 no-scroll guarantee.

Context​

SessionService is the connective tissue between camera capture (Stage 3) and media processing (Stage 4). The system architecture names it the owner of the active session and its state transitions, the API contract defines the /sessions surface and the ten-state session state machine, and the WebSocket contract defines session.created/state/completed/error. None of this had a scheduled owner, and the Stage 3 kiosk UI was written to consume session events that nothing emitted. This stage closes that gap. See ADR-0003.

This stage also takes ownership of the full session-state kiosk experience (countdown → capture → review → processing → present) that was previously bundled into Stage 3. Stage 3 keeps only the preview-only kiosk surface that does not need an active session.

Implementation Guidance​

  • Start with CLAUDE.md, system-architecture.md §Session State Machine, and the api-contract.md /sessions + session.* definitions.
  • SessionService owns state; CameraService, ExposureService, LightingService, and MarqueeService are called by it, not the other way round.
  • Keep scope limited to orchestration and the session-driven kiosk UI. Real rendering belongs to Stage 4; emit the processing transition and consume render progress from the shared job/queue primitive when it lands.
  • Preserve fake/off-hardware paths: the full session loop must run on fake cameras and fake render outputs.
  • Respect fast-capture-path: capture acknowledgement and the move to review/ instant preview must not wait on full-resolution rendering.

Acceptance Checks​

  • Docs are updated before or alongside behavior changes, and the owning docs are listed in the PR/summary.
  • Only one session can be active; a second POST /sessions returns 409 while one is in progress, verified by test.
  • Every documented session state and transition is exercised by fake/off-hardware tests, including reject/abort and error paths.
  • session.created, session.state, session.completed, and session.error events are emitted on the WebSocket contract and consumed by the kiosk UI.
  • Capture acknowledgement / transition into review is not blocked by rendering; the instant-preview-then-final handoff is observable in events.
  • The session loop runs end-to-end against fake cameras and a fake render adapter.

Features​

Session service and state machine​

The session lifecycle is owned, locked, and event-driven.

Stories And Tasks​

  • Task: Implement SessionService and the state machine.
    • Implement all allowed states (idle, attract, countdown, capture, record, review, processing, present, complete, error) and the legal transitions, with the transition table documented and tested.
  • Task: Enforce the single-active-session lock.
    • Guarantee one active session; return 409 on concurrent POST /sessions and cover it with tests.
  • Task: Wire the /sessions endpoints.
    • Implement POST /sessions, GET /sessions/active, confirm, reject, and abort against the API contract, including preset/camera_role selection from the active preview role.
  • Task: Emit session events.
    • Emit session.created/state/completed/error over the WebSocket contract as the single source of truth for booth state.

Session coordination across services​

The session drives cameras, exposure, lighting, and the render handoff without owning their internals.

Stories And Tasks​

  • Story: Session coordinates capture and the render handoff.
    • As the booth, a confirmed capture triggers CameraService capture, then moves to processing, hands work to the render/queue boundary, and advances to present when instant preview is ready — without blocking on full-resolution output.
  • Task: Coordinate exposure/lighting/marquee transitions.
    • Map session transitions to ExposureService metering, LightingService presets, and MarqueeService programs through their service boundaries, degrading safely when a subsystem is fake/unavailable.

Session-driven kiosk experience​

The guest booth UI reflects live session state at a glance, inspired by the reference app frontpage flow, within the 1280x800 no-scroll constraint.

Stories And Tasks​

  • Story: Guest can read live preview, countdown, capture, review, and result states.
    • As a guest, I can see the current booth state from session.* events so I know when to pose, wait, retry, or view the result.
  • Task: Build the session-state kiosk flow.
    • Implement countdown, capture feedback, review (confirm/reject), processing, and present screens driven by session events, reusing the Stage 3 preview surface.
  • Task: Add kiosk E2E coverage for the session loop.
    • Cover start → countdown → capture → review → processing → present, plus reject/abort, conflict (409), and error states against fake cameras and a fake render adapter.
