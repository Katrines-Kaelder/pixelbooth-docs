Feature Requests

This register captures product ideas before they are ready to become roadmap tasks, known issues, GitHub issues, or implementation plans. Use it for features that need light definition, triage, or product discussion.

The goal is to preserve the intent of an idea without pretending the design is finished. Each entry should be clear enough for a later planning session to understand the user, value, rough behavior, open questions, and next step.

How To Use This Register​

Add or update an entry when:

  • A user asks for a new feature or workflow.
  • A hardware/test session reveals a product improvement rather than a bug.
  • A known issue points at a new capability that should be designed separately.
  • A roadmap idea is not yet specific enough to become a task.

Move an entry out of the active table when:

  • It is accepted into a roadmap task or GitHub issue.
  • It is intentionally deferred or rejected.
  • It ships and has a concrete evidence link.

Intake Checklist​

Before adding a feature:

  • Check whether an existing FR-###, known issue, or roadmap task already covers the same idea.
  • Ask clarification questions only when the missing answer changes the meaning or scope of the request.
  • Keep product intent separate from implementation guesses.

After adding a feature:

  • Update the Request Index.
  • Keep the active and closed tables mutually consistent.
  • Tell the user which FR-### was added or updated, and list any remaining open questions.
