System Architecture

Runtime Shape​

Pixelbooth is a local-first appliance. The first implementation should use a Python/FastAPI local runtime and a built React frontend. In development, the frontend can run through Vite while the API runs separately.

The architecture should start simple but keep service boundaries explicit. Early Raspberry Pi bring-up may run mostly in one process, while render workers, sync jobs, network helpers, printer helpers, and marquee-facing services can split into separate processes or containers when useful.

Service Boundaries​

  • SessionService: owns the active capture session and state transitions.
  • CameraService: owns camera discovery, preview streams, and test captures.
  • LightingService: maps session events to lighting presets and talks to adapters.
  • ExposureService: meters scene brightness and coordinates camera exposure, EV, gain, white balance, and ring-light output.
  • MarqueeService: stores marquee designs, previews them in admin, maps booth events to marquee programs, and pushes programs to the Pi Zero 2 W agent.
  • MediaService: owns SQLite catalog and media file paths.
  • ConfigService: owns typed admin settings, settings templates, profile import/export, validation, secret handling, and revision history.
  • AssetService: owns operator-uploaded assets and safe file browser roots under data/assets, data/media, data/exports, and data/backups.
  • OverlayService: owns PNG overlay assets, overlay assignments, live/admin preview composition, render-pipeline overlay definitions, and the future overlay designer model.
  • RenderWorkerPool: creates previews, thumbnails, overlays, collages, GIFs, boomerangs, videos, chromakey renders, and AI/filter outputs without blocking capture.
  • ShareService: owns print/share integrations, quotas, and cooldowns.
  • SyncService: owns queue-backed sync remotes. Each remote has common enable/policy/status fields plus type-specific settings for local folder, USB, rclone/cloud, Immich, rsync/SFTP, and future webhook destinations.
  • NetworkService: owns Wi-Fi scan/connect and fallback access point support through a privileged helper.
  • PrintService: owns printer discovery, print preview/crop handling, queueing, CUPS/IPP adapters, and future payment gating.
  • CommandService: owns the admin "Commander" task runner: allowlisted system, service, diagnostics, queue, and hardware test tasks with confirmation, status, logs, and audit history.
  • DiagnosticsService: reports system health and service status.

Process Model​

The intended deployment direction is containerized or installer-managed, not a pile of hand-run local scripts. V1 may still run most services in one Python process for simplicity, but process boundaries must be respected. Heavy rendering, sync, network management, and printer work should be queue-backed and able to move into separate processes or containers without changing the public API.

Deployment targets:

  • local dev with fake hardware
  • single-process Pi appliance for early bring-up
  • Docker Compose on the Pi with API, render, sync, network, print, and marquee-facing services
  • installer-managed systemd services where containers are impractical for privileged hardware access

Local shell scripts, if present, are developer conveniences only. They are not the target deployment architecture and should eventually wrap or delegate to Docker Compose, the installer, or service-manager commands.

Shared job/queue primitive​

Heavy and deferrable work uses one durable, retryable, status-emitting job/queue primitive rather than per-feature queue implementations. The RenderWorkerPool (Stage 4) is its first consumer; the SyncService outbox (Stage 8) reuses the same primitive. Jobs are persisted in SQLite so they survive restarts, expose retry/status, and emit progress over the WebSocket contract. This keeps the fast-capture-path guarantee — capture acknowledgement is never blocked by queued work — consistent across render, sync, and future print/marquee jobs. The primitive is realized in Stage 4 (a generic jobs table on the SQLite foundation plus an asyncio worker; render.final is the first consumer) and the Stage 8 sync outbox reuses it; see ADR-0011.

Access Control​

Pixelbooth is a single-operator appliance, not a multi-tenant app. Guest-facing surfaces (booth, gallery, live preview, GET /config) are open. Everything under /admin/* and all mutating admin APIs (config, assets, Commander, remotes) sit behind a single API enforcement boundary, gated by an operator secret backed by environment bootstrap settings (not the SQLite settings it protects). This gate ships with the first admin surface (Stage 2.5); richer role-aware access (operator/rental/setup) extends the same boundary later (Stage 7). See ADR-0004.

Session State Machine​

Allowed states:

  • idle
  • attract
  • countdown
  • capture
  • record
  • review
  • processing
  • present
  • complete
  • error

Only one session may be active. New session requests return a conflict while another session is in progress.

Legal transitions​

SessionService is the single owner of the machine; the legal transitions are declared as pure data in pixelbooth/domain/sessions.py and exercised by tests.

FromTo
idlecountdown
attractcountdown
countdowncapture, record, complete, error
capturecountdown (next posed shot), review, complete, error
recordreview, complete, error
reviewprocessing (confirm), countdown (reject/retry), complete (abort), error
processingpresent, complete, error
presentcomplete, error
complete— (terminal)
error— (terminal)

Every non-terminal state may move to complete (graceful abort) or error (a fault at any point). Reaching a terminal state clears the active session so a new one may start; abort emits session.completed with reason: "aborted".

Orchestration mechanism (Stage 3.5)​

SessionService runs on the API event loop and drives the timed transitions (countdown → capture → review, and on confirm processing → present → complete) with an asyncio task, guarded by a single lock that enforces the single-active-session invariant. Blocking camera capture runs in a worker thread (asyncio.to_thread) so it never stalls the loop, and the move to review (instant preview) never waits on full-resolution rendering (.claude/rules/fast-capture-path.md).

Rendering is reached through a RenderPort seam (pixelbooth/adapters/render/): an instant preview on the capture path, then a full-resolution final during processing. Stage 3.5 ships only a deterministic FakeRenderAdapter; Stage 4 plugs the real LocalRenderAdapter (queue-backed render.final job) behind the same port without the session changing. The session passes the capture's preset_id and compose_format through the port so the final render can resolve a preset-scoped overlay and pick the right composer (additive seam changes only).

Capture choreography (per format). A composed preset captures a short frame sequence instead of a single still, but the choreography differs by capture mode (ADR-0013):

ModeChoreographyGuest feel
stillone countdown → capturesingle posed shot
collageposed: loop countdown → capture once per shot (default 4)re-pose between shots; kiosk shows "Shot k of N"
animation / boomerang / videorecord: one countdown, then the record state for record_seconds while frames are capturedfeels like recording; kiosk shows a recording indicator

The posed loop uses the capture → countdown transition; the record choreography drives the reserved record state. The active Session carries shot_index and shot_count (published on session.created and every session.state), which are 1/1 for non-posed modes. The three record formats share one capture path and differ only in the composer (GIF / forward-reverse GIF / MP4) — whether a real device grabs discrete stills or pulls frames from a live stream is an adapter detail behind CameraService.capture_sequence. The instant preview is always the first frame, so the fast path is unchanged. During render.final the LocalRenderAdapter picks a composer (adapters/render/composers.py) by compose_format and composes the frames into the final output. Composition is best-effort and backend-optional like overlays: collage/video use OpenCV and GIF/boomerang use Pillow, both optional (OpenCV is the usb extra, never on the dev/CI path), so when no backend is installed the format is recorded compose_status='skipped' and the final falls back to the first frame — a valid, servable output. See ADR-0012.

During the render.final job the LocalRenderAdapter also consumes overlay assignments from OverlayService (precedence preset > event > global, where the event scope is the currently active event resolved at render time) and composites the resolved transparent PNG into the final output for single-image and collage finals — the instant preview stays overlay-free on the fast path, and animation/video finals are not overlayable. Composition is best-effort: when the asset is missing, the PNG is invalid/has no alpha, the media type is incompatible, or no image backend is installed (OpenCV is the optional usb extra, never on the dev/CI path), the overlay is recorded as skipped with a logged diagnostic id and the un-overlaid bytes are kept. The render never fails on an overlay or composition problem; the outcome is persisted on the media record (overlay_status, compose_status). Exposure/lighting/marquee coordination goes through a no-op SessionCoordinator seam until those services exist (Stage 5/6); a subsystem fault there is logged and never breaks the session. See ADR-0003.

Storage​

Files are authoritative for image/video bytes and uploaded assets. SQLite is authoritative for metadata, settings, templates, queues, and task history:

  • session id, preset, state, selected camera role
  • media id, type, paths, timestamps, hidden flag
  • camera role and capture settings used
  • lighting preset used
  • exposure profile and metering decision used
  • marquee design/program triggered
  • active admin settings and settings schema version
  • settings templates for events, rentals, venues, and recovery defaults
  • setting revisions or backups for rollback where useful
  • asset records for overlays, backgrounds, fonts, branding, marquee assets, exports, and backups
  • overlay definitions, PNG overlay assignments, and future overlay-designer JSON
  • sync outbox jobs, remote definitions, remote-specific settings, and destination status
  • print jobs and printer status
  • Commander task definitions, run history, output summaries, and diagnostic ids
  • future challenge result metadata

JSON remains an interchange and backup format, not the only live state store. Use JSON for imported/exported setting templates, overlay definitions, marquee definitions, event/rental profiles, and backups. Runtime bootstrap settings that must exist before SQLite opens still belong in environment-backed process settings.

Engine and migrations​

The SQLite layer is a single Database object (pixelbooth/db/) that owns one shared sqlite3 connection opened at API startup and closed at shutdown, tied to the FastAPI lifespan. On connect it enables WAL, foreign keys, and a busy timeout; access is serialized by a lock so sync handlers running in Starlette's threadpool never share the connection concurrently. Schema changes are forward-only NNNN_name.sql files in pixelbooth/db/migrations/, applied once and tracked in a schema_migrations table; the runner validates ordering and fails the app start on a malformed or failing migration. A :memory: path gives tests an ephemeral database; the file path otherwise derives from data_dir (&lt;data_dir&gt;/pixelbooth.sqlite). The stdlib driver and hand-rolled runner (no ORM, no Alembic) are chosen in ADR-0007; every Stage 2.5–4 catalog adds its own migration and uses this one Database seam.

Storage capacity​

A busy event generates media continuously, so storage capacity is a first-class concern, not an afterthought. The system reports disk usage, defines capacity thresholds, and applies a retention/eviction policy so the disk cannot silently fill mid-event; capacity state surfaces on the dashboard and the booth degrades safely (e.g. blocks new captures with a clear operator message) when storage is critically low. This is owned by the media/storage layer in Stage 4.

Frontend Apps​

The React app is one frontend with multiple operator and guest surfaces:

  • /: booth/kiosk
  • /gallery: media gallery
  • /admin: admin control plane dashboard
  • /admin/settings: typed settings
  • /admin/assets: file browser/upload for safe asset roots
  • /admin/overlays: overlay preview and future designer
  • /admin/remotes: sync/upload remote configuration and queue status
  • /admin/commander: allowlisted system tasks
  • /admin/logs: logs and diagnostic ids
  • /admin/setup: first-run/rental setup
  • future /standalone/*: gallery/slideshow/display modes for secondary screens

The app consumes REST for commands and resources, and WebSocket events for live session/media/status updates.

Admin should be planned as a real product surface. The dashboard is the first admin view and should answer whether the booth is ready for an event. See admin-control-plane.md.

Frontend implementation standards are defined in frontend-development-standards.md. In short:

  • use Vite + React + TypeScript
  • use pnpm workspaces
  • use Tailwind CSS with semantic theme tokens for production styling
  • keep components small and reusable
  • use explicit utilities and proper state ownership
  • keep modules navigable enough for human review and optional code graph tools

Backend Runtime​

The first local backend runtime decision is documented in backend-runtime-decision.md. In short, Python/FastAPI is chosen because Pixelbooth's hardest local problems are Raspberry Pi camera control, media processing, device integration, and future CV/AI work. NestJS or Fastify may still be good choices for future cloud/fleet services or non-critical gateways.
