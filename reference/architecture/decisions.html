Architecture Decision Records (ADRs)

This folder records significant, hard-to-reverse decisions and the context behind them. Prose architecture docs (in docs/architecture/) describe how the system works now; ADRs capture why a choice was made and what was rejected, so the decision can be revisited without re-litigating from scratch.

When to write an ADR​

Write one when a change:

  • introduces, replaces, or retires a runtime, framework, protocol, or storage choice;
  • changes a service boundary, the public API/event contract, or the session state machine;
  • resolves one of the open questions listed in ../../README.md;
  • establishes a cross-cutting convention (e.g. the AI collaboration layer).

Small, reversible changes do not need an ADR — update the relevant prose doc instead. See rule docs-first-architecture.

Format​

One file per decision: NNNN-kebab-title.md, numbered sequentially. Use this template:

# ADR-NNNN: <title>

- **Status**: Proposed | Accepted | Superseded by ADR-XXXX | Deprecated
- **Date**: YYYY-MM-DD
- **Deciders**: <names/roles>

## Context
What forces are at play? What problem or constraint triggered this?

## Decision
The choice made, stated plainly.

## Consequences
What becomes easier, harder, or constrained. Follow-up work.

## Alternatives considered
Each option and why it was not chosen.

A decision is never edited away once Accepted; instead a new ADR supersedes it and the old one's status is updated with a link.
