Backend Runtime Decision

Decision​

Use Python with FastAPI as the first local booth runtime.

This is not a claim that FastAPI is universally better than NestJS or Fastify. It is the best fit for Pixelbooth's first hardware-heavy appliance runtime because the difficult parts of this system are camera control, media processing, Raspberry Pi integration, local files, workers, serial/network device control, and future computer vision.

Why FastAPI First​

FastAPI keeps the local API close to the Python hardware and media stack:

  • Raspberry Pi Camera Module 3, libcamera, and Picamera2 integration.
  • OpenCV, Pillow, ffmpeg/GStreamer orchestration, and future CV/AI tools.
  • GPIO, serial, USB, local filesystem, SQLite, and Linux device integration.
  • Pydantic models for typed API/config surfaces.
  • OpenAPI generation for TypeScript API types.
  • WebSocket support for live booth/session/status events.
  • Simple fake-hardware testing for non-Pi development.

The capture path should avoid unnecessary language and process boundaries until a specific service needs to split out.

Why Not NestJS First​

NestJS is strong for large structured TypeScript backends, especially business logic, auth-heavy APIs, multi-tenant systems, and teams that benefit from a strict module/decorator pattern.

For Pixelbooth v1, NestJS would likely become an orchestration layer around Python services, because camera capture, Picamera2, OpenCV, media processing, and future AI filters still belong in Python. That would add IPC, HTTP, gRPC, Redis, or queue boundaries around core capture behavior before the product needs them.

NestJS could make sense later for:

  • a hosted multi-booth management platform
  • account/admin/payment-heavy cloud services
  • a fleet dashboard separate from the local capture appliance
  • a non-critical API gateway over already-separated Python workers

Why Not Fastify First​

Fastify is fast, lightweight, and a good Node runtime choice. If Pixelbooth's backend were mostly web/API work, Fastify would be a serious candidate.

The same core concern remains: Node is not the natural home for Raspberry Pi camera control and image-processing work. Fastify could be useful later for:

  • a cloud companion API
  • a thin local gateway if the Python services are already separate
  • upload/webhook integrations where the Node ecosystem has a clear advantage

Architecture Guardrail​

FastAPI is the first runtime choice, not a permanent monolith promise.

The codebase should be organized around clear service boundaries:

  • ControlApi
  • SessionService
  • CameraService
  • RenderWorkerPool
  • MediaService
  • LightingService
  • ExposureService
  • SyncService
  • NetworkService
  • PrintService
  • MarqueeService
  • ObservabilityService

If a boundary becomes easier, faster, or safer in another runtime, it can move behind the same public API/events without rewriting the booth experience.

Acceptance Criteria​

The backend choice is working if:

  • capture commands acknowledge quickly
  • preview and capture are reliable on the Pi
  • render work cannot block the booth UI
  • fake adapters make development possible away from hardware
  • the frontend receives generated or strongly aligned TypeScript API types
  • future workers/services can split without breaking the public API contract
