Admin Control Plane

Pixelbooth has one frontend application. The guest booth, gallery, admin, setup, and future companion display views live in the same React app and share one API/event model. Admin is not a separate product or an afterthought; it is the operator control plane for the appliance.

The reference photobooth frontend is useful because it shows a mature single-app shape: a kiosk route, gallery routes, standalone display routes, an authenticated admin layout, a real dashboard as the admin landing page, file upload/browser tools, schema-driven config, logs, first-start setup, multicam tools, and system commands behind confirmations. Pixelbooth should keep those concepts, improve the UX, and avoid copying the implementation directly.

Admin Information Architecture​

Initial admin routes should be organized around operator jobs:

  • /admin: dashboard with opening-night readiness, system health, recent activity, storage, temperature, camera/lighting/sync/print status, and safe quick actions.
  • /admin/settings: typed settings editor grouped by booth, capture presets, cameras, exposure, lighting, overlays, sharing, remotes, printer, network, and advanced system settings.
  • /admin/assets: file browser and upload surface for user-managed assets such as overlays, backgrounds, fonts, logos, QR branding, marquee assets, and imported/exported setting templates.
  • /admin/overlays: overlay library, PNG overlay preview, and future overlay designer.
  • /admin/media: operator gallery with hide/delete/download/recycle controls.
  • /admin/remotes: upload/sync remotes, queue status, test actions, and per-remote credentials/settings.
  • /admin/commander: allowlisted system tasks, service tasks, diagnostics tasks, and hardware test commands with confirmation and live output.
  • /admin/logs: structured logs, filters, diagnostic ids, and downloadable bundles.
  • /admin/setup: guided first-run/rental setup flow.

The first implementation can combine some routes into tabs or panels, but the conceptual model should stay stable so later work has a place to land.

Dashboard First​

The admin landing view should answer "can we open the booth now?" without forcing the operator to inspect every panel.

Dashboard sections:

  • readiness summary: cameras, storage, disk space, temperature, API version, frontend version, network, printer, sync, lighting, marquee
  • active session/state summary
  • recent media count and latest capture
  • render/sync/print queue health
  • warnings that need attention
  • quick tests: camera test capture, lighting test, overlay preview, sync remote test, printer test
  • safe system actions with confirmation: reload services, restart app, reboot host, shutdown host

Avoid burying critical warnings in diagnostics. Diagnostics can be detailed; dashboard should be decisive.

Settings And Templates​

All admin-set values must persist. Pixelbooth should use:

  • SQLite as the authoritative store for active settings, templates, queue state, media metadata, command history, and remote status.
  • Pydantic models as the typed validation boundary for settings groups.
  • JSON import/export for setting templates, event/rental profiles, overlay definitions, marquee definitions, and backup/restore.
  • File assets under data/assets/*, referenced from SQLite by stable ids and relative paths.
  • Environment variables only for process/runtime bootstrap values such as data directory, environment, bind host/port, and fake-hardware flags.

This gives the speed and inspectability of JSON where it helps, while avoiding a growing pile of mutable JSON files for transactional app state.

The capture settings section also carries the per-format render output sizes — still/video/gif max width/height, where 0 keeps native resolution (ADR-0015/0016). These are admin-editable at runtime (ADR-0017): defaults come from the typed CaptureSettings model (seeded into SQLite on first run like any setting — not env), and the render reads them live, so a change applies to the next capture's render without a restart. GIF defaults to a 720px box because native GIF is very large and slow.

Settings model:

  • active_settings: current appliance settings, versioned by schema.
  • settings_templates: named snapshots for event profiles, rental profiles, venue profiles, and recovery defaults.
  • settings_revisions: optional history for rollback and audit.
  • settings_secrets: encrypted or protected fields where needed; do not expose secrets through guest config endpoints.

The admin UI can be partly schema-assisted, but high-risk or high-value settings should get purpose-built controls instead of a raw generated form. Schema-driven forms are good for advanced/system sections; capture presets, overlays, remotes, and first-run setup deserve custom UX.

Assets And File Browser​

The file browser should be an operator asset manager, not unrestricted filesystem access.

Allowed roots:

  • data/assets/overlays
  • data/assets/backgrounds
  • data/assets/fonts
  • data/assets/branding
  • data/assets/marquee
  • data/media
  • data/exports
  • data/backups

The API should enforce path sandboxing, MIME/type allowlists, file-size limits, and destructive confirmations. Uploading a transparent PNG overlay should make it selectable in overlay settings without requiring the operator to remember a path string.

Overlay Concept​

Overlays are first-class media assets and render instructions.

V1 overlay behavior:

  • transparent PNG overlays at output image size
  • assigned globally, per capture preset, or per event/template
  • previewed in admin against a live preview stream or latest fake/real camera frame
  • composited into final render outputs by the render pipeline
  • optionally shown over the kiosk live preview during idle/countdown

Future overlay designer:

  • live preview stream or still frame as the background
  • drag-and-drop layers for image, text, logo, QR placeholder, safe areas, and decorative elements
  • output-size canvas with responsive fit preview
  • layer ordering, snapping, position/size controls, opacity, rotation, lock/hide, and validation
  • export to a .pixelbooth-overlay.json definition plus generated preview/render assets

The designer should not be required for V1, but its data model should be anticipated so V1 PNG overlays do not become a dead end.

Commander​

Commander is the admin-facing task runner. It should improve on the reference app's broad command buttons by using typed, allowlisted tasks instead of arbitrary shell commands from UI config.

Task examples:

  • reload Pixelbooth services
  • restart API/frontend bundle
  • reboot/shutdown host
  • run camera test capture
  • run lighting test preset
  • run sync remote test
  • retry failed queues
  • create diagnostic bundle
  • check disk/media integrity
  • export settings/media bundle

Each task should define:

  • id, label, category, risk level, and confirmation requirement
  • required role/auth/local-only constraint
  • input schema for safe parameters
  • executor adapter
  • timeout, log handling, and cancellation support where possible
  • audit record with status, timestamps, output summary, and diagnostic id

Do not expose free-form shell execution in the admin UI. A future expert mode can be considered, but the appliance default should be safe and predictable.

Remotes And Synchronizer​

Sync is remote-driven. Operators configure one or more remotes, each with common queue behavior and type-specific settings.

Common remote fields:

  • name
  • type
  • enabled
  • sync policy: originals, final renders, print-ready, mobile/share, metadata, or all
  • trigger policy: on capture complete, scheduled, manual, on USB insert
  • concurrency and retry policy
  • destination path/album/folder semantics
  • last success/failure and queue depth

Remote types should be adapters, not separate one-off services. Recommended first adapters:

  • local folder/export
  • USB export
  • rclone remote for Google Drive and other cloud providers
  • Immich upload/import
  • rsync/SFTP for simple server targets
  • webhook later

Transport engines are decided in ADR-0005: rclone is the default for cloud and SFTP (many providers, auth flows, retry/bandwidth/parallelism/checksum control), rsync/SSH serves LAN/self-hosted copies, and Immich uses its native upload path (endpoint verified against current official docs at implementation time). All remotes share one common model with type-specific settings, and the durable SQLite outbox reuses the shared job/queue primitive.

rclone backend settings​

An rclone remote selects a backend and carries that backend's settings keys (they map mechanically to rclone's own config; secrets travel via env, never the command line):

  • s3 (default, MinIO-first): endpoint, access_key_id, secret_access_key, bucket, optional provider/region; the bucket is the first path segment.
  • webdav: url, optional vendor/user/pass (password obscured via rclone obscure).
  • ftp / sftp: host, optional user/port/pass/tls/key_file (password obscured).
  • drive (Google Drive): token (required), optional client_id/client_secret/scope/root_folder_id/team_drive.

Google Drive auth is OAuth2, so it can't be a plain credential paste. The operator runs rclone authorize "drive" once on any machine with a browser, completes Google consent, and pastes the resulting token JSON blob into the remote's settings (alongside an optional own client_id/client_secret to avoid rclone's shared quota, and a scope such as drive.file). The token carries a refresh token, so it is treated as a secret — stored in the remote's settings and passed to rclone via the env config, never argv.

Immich, rsync/SSH, and USB settings​

Beyond rclone, three more transports cover LAN, self-hosted, and removable-drive targets. They share the one common remote model; only their type-specific settings differ.

  • immich — uploads each media file through Immich's native asset API (POST /api/assets, authenticated with x-api-key; endpoint verified against current official Immich docs at implementation time). Settings: server_url (required, e.g. https://immich.example.com), api_key (required, treated as a secret — redacted on read, sent only in the request header, never on a command line). Optional device_id labels the booth as the source device. Immich dedupes by checksum, so re-uploads are idempotent. Immich is a backup/sync remote, not the recommended QR backend (it has no dumb public-URL surface) — same posture as Google Drive.
  • rsync_sftp — rsync over SSH for LAN/self-hosted server copies (incremental, resumable). Settings: host (required), path (required, the remote destination directory), optional user/port/ssh_key_file/rsync_options. Authentication is SSH key only — rsync cannot take a password on argv safely, so the operator installs a key (path in ssh_key_file, or the agent/default key) and the adapter runs SSH in BatchMode (no interactive prompts). The ssh_key_file path is a filesystem reference, not a pasted secret.
  • usb — copies media to a configured USB drive mount path. Settings: mount_path (required, where the host has mounted the drive), optional subdir. The privileged mount/unmount itself stays in a host-side helper (hardware-boundaries); the adapter only copies to an already-mounted path and degrades with a user-safe "drive not mounted" message when the path is absent. Pairs with the on_usb_insert trigger policy.

Share remote and QR sharing​

One enabled remote is designated the share remote that backs guest QR sharing (ADR-0018). Because guest phones cannot reach the Pi at the venue, the booth uploads each finished render to this remote and the QR encodes a public URL the guest opens over their own mobile data. The share/sync services (upload engine, durable outbox, admin remote config) are built first and are independently valuable for off-site backup/export; QR is a follow-on slice on top. The first reference transport is S3-compatible (MinIO, self-hosted); the same model carries self-hosted WebDAV/FTP/SFTP and Google Drive (the latter is a backup/sync remote, not the recommended QR backend).

The Sharing settings (in the sharing settings section) configure the QR public link:

  • share_remote_id — which remote backs the QR.
  • link_mode — sharepage (default, the themed arcade grab page) or direct.
  • public_link_template — placeholders {key}, {event}, {filename}.
  • sharepage_url — base URL of the deployed static sharepage.
  • link_expiry_days — privacy/lifecycle hint (the host enforces expiry).
  • share_variant — final (default) or original.

Admin actions: designate the share remote, edit the link template / sharepage toggle / expiry, run "test remote", and show a live share-link preview for the active event. The host stays a dumb static/file server — no server-side script — so admin only supplies the upload target plus the link template; credentials stay in the remote's type-specific settings, redacted on read.

The scripts/share-host-setup helper (a local operator script — never host-side) deploys the static apps/sharepage to the upload target via rclone and prints the exact Remotes + Sharing values to paste here, so first-time setup is copy-paste rather than hand-derived.

Reference Frontend Ideas To Keep​

  • One frontend application with kiosk, gallery, admin, and standalone display routes.
  • Admin dashboard as the first admin view.
  • Admin layout with persistent navigation.
  • File browser/upload/download/delete concepts.
  • Schema-derived settings as an accelerator, not the only UX.
  • Live preview rendered efficiently in the browser.
  • PNG frame overlays for live preview and capture output.
  • Confirmation dialogs for dangerous system actions.
  • Status and logs close to the operator workflow.

Reference Frontend Ideas To Improve​

  • Use explicit HTTP methods for side effects.
  • Prefer typed task/remotes/settings models over command strings.
  • Restrict file browser roots and destructive actions.
  • Keep localization out of the critical path for V1.
  • Use one React design system and custom UX for high-value controls instead of relying on generic generated forms everywhere.
  • Store transactional state in SQLite, with JSON import/export for portability.
