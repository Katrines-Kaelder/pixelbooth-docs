ADR-0007: SQLite foundation — stdlib driver and SQL-file migrations

  • Status: Accepted
  • Date: 2026-06-12
  • Deciders: Jonas Landsfeldt, Claude

Context​

Stage 2.5 (Unified Frontend And Admin Control Plane Foundation) establishes the persistence substrate that every later catalog builds on: config/active settings, settings templates, assets, overlays, remotes, command history, and — from Stage 4 — the media catalog and the durable job/queue outbox. system-architecture.md (§Storage) already fixes the split: files are authoritative for image/video bytes and uploaded assets; SQLite is authoritative for metadata, settings, templates, queues, and task history. What was undecided is how the SQLite layer is opened, migrated, and accessed.

Forces at play:

  • The appliance is local-first and single-process on a Raspberry Pi. Concurrency is low (one booth, a handful of operators), durability across restarts matters, and field inspectability of the schema matters for offline diagnosis.
  • The project ethos is deliberate and minimal (ADR-0002 rejects premature structure; no-premature-scaffold). Existing services are synchronous and small.
  • Off-hardware development and tests must run without touching real disk state (the task requires a fake/in-memory path).
  • This is a hard-to-reverse foundation: every Stage 2.5–4 catalog inherits it.

Decision​

Use the Python standard-library sqlite3 driver with a hand-rolled SQL-file migration runner. No ORM and no external migration framework at this stage.

  • Connection lifecycle. A single Database object owns one shared sqlite3.Connection (check_same_thread=False) opened at API startup and closed at shutdown, tied to the FastAPI lifespan. On connect it sets PRAGMA journal_mode=WAL, PRAGMA foreign_keys=ON, and PRAGMA busy_timeout. Access is serialized by a threading.Lock so concurrent request threads (Starlette runs sync handlers in a threadpool) cannot use the connection simultaneously. The object exposes a read connection() context manager and a transaction() context manager that commits on success and rolls back on error.
  • Migrations. Ordered NNNN_name.sql files in pixelbooth/db/migrations/ are applied exactly once and tracked in a schema_migrations table. The runner discovers files, validates names and contiguous ordering, applies any unapplied migrations on startup, and fails fast (the app does not start) if a migration is malformed or errors. Migrations are forward-only.
  • In-memory path. A :memory: database path yields an ephemeral in-process DB for tests; the single-shared-connection design makes :memory: behave correctly for the app's lifetime. The database file path is derived from data_dir (<data_dir>/pixelbooth.sqlite) and overridable via settings.
  • Scope. This task ships only the engine, the migration mechanism, the connection lifecycle, and one foundation migration (app_meta). Catalog tables are added by their own migrations in their own Stage 2.5 / Stage 4 tasks.

Consequences​

  • Easier: zero added dependencies; an explicit, inspectable schema; trivial off-hardware and CI runs via :memory:; a single seam (Database) every later catalog/service depends on; durable state across restarts via WAL.
  • Constrained: writes are serialized through one connection+lock — correct and ample for an appliance, but not a high-concurrency design. Migrations are forward-only with no autogenerated diffing; each schema change is a hand-written SQL file reviewed like code.
  • Follow-up: catalog migrations (settings, assets, overlays, remotes, command history) land with their services this stage; the media catalog and shared job/queue primitive reuse this foundation in Stage 4. If query/schema complexity later outgrows hand-rolled SQL, revisit with a superseding ADR (an ORM/Alembic is a deliberate future step, not a default).

Alternatives considered​

  • aiosqlite + SQL-file migrations. Async-native and idiomatic with FastAPI's async handlers, at the cost of an added dependency and async plumbing through every catalog. Rejected: the appliance's low concurrency does not need it, and the existing services are synchronous; the stdlib driver behind a lock is simpler and sufficient. Reconsider if a future workload shows real contention on the DB thread.
  • SQLAlchemy + Alembic. Battle-tested ORM and migration autogeneration, strong for large relational schemas. Rejected as premature (YAGNI): a heavyweight dependency and abstraction layer for a handful of appliance tables, against the project's minimalism ethos. Revisit only if schema/query complexity clearly justifies it.
  • JSON files for live state. Simple and inspectable, but loses transactional integrity, concurrent-write safety, and query power for queues/history/metadata. JSON remains an interchange/backup format (templates, overlay/marquee definitions, profiles, backups), not the live store — consistent with §Storage.
