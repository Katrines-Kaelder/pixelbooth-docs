ADR-0004: Admin access control lands with the first admin surface

  • Status: Accepted
  • Date: 2026-06-11
  • Deciders: Jonas Landsfeldt, Claude

Context​

Stage 2.5 introduces the admin control plane: config mutation (PUT /admin/config), asset delete/recycle, and Commander task execution including dangerous-risk tasks. In the original roadmap, any form of access control did not appear until Stage 7 ("protected admin navigation", "role-aware access"). That left several stages in which destructive, unauthenticated endpoints are reachable by anyone on the booth's LAN — a busy bar network. This contradicts the project security rules.

Pixelbooth is a local-first appliance, not a multi-tenant web app. It does not need full identity/IdP infrastructure, but it does need a gate so guests, curious patrons, or anyone on the same network cannot reconfigure or wipe the booth.

Decision​

A minimal admin access gate ships in Stage 2.5, with the first admin surface. The guest-facing booth and gallery remain open; everything under /admin/* and the admin/Commander/config/asset-mutating API routes sit behind the gate.

  • V1 is a single shared admin secret (operator passphrase/PIN) plus a signed session/token boundary, configured through environment-backed bootstrap settings (not stored only in the SQLite settings it protects).
  • Guest-safe read endpoints (GET /config, media/gallery reads, live preview) stay unauthenticated so the booth and gallery work without login.
  • The gate is a single enforcement boundary in the API layer, so richer role-aware access (per the Stage 7 rental/operator roles) extends it later without re-securing each route.

Stage 7 keeps only the richer role/rental access work; it no longer owns the basic existence of authentication.

Consequences​

  • Easier: destructive admin power is never exposed unauthenticated; one place to reason about who can mutate state; rental role logic later is an extension, not a retrofit.
  • Constrained: Stage 2.5 must define the auth boundary and a guest/admin route split before admin feature panels land; fake/dev flows need a documented default credential or bypass for off-hardware development.
  • Follow-up: Stage 7 adds role-aware access (operator vs. rental vs. setup), confirmations, and destructive-action guardrails on top of this gate.

Alternatives considered​

  • Defer all auth to Stage 7 (original plan). Rejected: leaves destructive endpoints open across multiple stages on an untrusted LAN.
  • Full identity provider / multi-user accounts now. Rejected: massive overkill for a single-operator appliance; a shared operator secret matches the deployment reality and keeps the gate simple and reliable.
  • Network-only protection (firewall/AP isolation). Rejected as the sole control: the booth deliberately exposes Wi-Fi/AP modes and a guest gallery on the LAN, so the application boundary must protect admin regardless of network posture.
