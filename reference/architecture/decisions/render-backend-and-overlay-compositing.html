ADR-0014: Render backend dependency policy and overlay compositing

  • Status: Accepted
  • Date: 2026-06-15
  • Amends: ADR-0012
  • Related: ADR-0008, ADR-0011, ADR-0013

Context​

The render pipeline (overlays, thumbnails, collage, GIF/boomerang, video) was built backend-optional: OpenCV (cv2) and Pillow were both optional, behind the usb extra. When a backend was absent, composers and overlay compositing returned None and the render degraded to the first captured frame (compose_status='skipped', overlay_status='skipped(no_backend)'). The dev/CI image installed neither library.

This produced three problems:

  1. No real render path in CI. Off-hardware everything degraded to a frame-0 still, so overlays, GIF/boomerang, collage, and thumbnails were never exercised by automated tests — only on the Pi.
  2. Overlays and video were observed broken on hardware (Raspberry Pi, usb extra installed). So these are real render-logic defects, not just the off-hardware degrade behavior (see KI-008, KI-014):
    • Overlay compositing force-resized the overlay to the base dimensions and rejected any overlay PNG without an alpha channel (shape[2] != 4 → skipped), and overlays were only applied to the final render, never the instant preview — so the guest never saw the overlay during the flow.
    • Video used cv2.VideoWriter with the mp4v FourCC, producing MPEG-4 Part 2, which most browsers cannot play. The file rendered but would not play back.
  3. Rendering is a core product capability, not an optional peripheral. The booth's value is composed media. Degrading to a single still is a poor default for the appliance's primary output.

Decision​

Split the dependency policy by what each backend is actually for:

  1. Pillow becomes a core runtime dependency. All raster composition Pillow can do runs on the core path and in CI: overlay alpha-compositing, thumbnail downscaling, collage grids, and GIF/boomerang encoding. These always render and are CI-testable. The skipped(no_backend) path is removed for them.
  2. OpenCV (cv2) stays the optional usb extra, scoped to camera capture / CV work (its original ADR-0008 purpose). It is no longer required for raster composition.
  3. Video (MP4) is encoded with ffmpeg (bundled binary via imageio-ffmpeg, a core dependency) using H.264 / yuv420p / +faststart for browser-compatible playback. This replaces the OpenCV mp4v FourCC (MPEG-4 Part 2), which browsers reject — the KI-008 video defect. A defensive compose_status='skipped' fallback to a still remains if the binary ever fails to load. Real motion video + audio remains deferred (ADR-0011/0012); v1 video encodes the captured still-sequence to a playable H.264 MP4.

Overlay coordinate space and compositing rules​

  • Overlays are authored against a normalized full-frame canvas matching the configured capture aspect ratio (the capture-size contract — see the companion ADR). The compositor scales the overlay to the actual rendered frame dimensions so preview and final scale identically.
  • Preview and final use the same compositing so the guest sees the overlay during the flow exactly as it lands in the final. Overlay compositing on the instant preview is a single cheap alpha-composite and does not block the fast-capture path.
  • The overlay alpha channel is honored when present; an overlay without an alpha channel is treated as fully opaque (composited, not rejected). Only an undecodable overlay raises an error and records a precise skip reason.

Consequences​

  • Overlays (KI-014), thumbnails, collage, and GIF/boomerang (KI-009) render reliably off-hardware and are covered by automated tests; their bugs can be reproduced and fixed in CI.
  • compose_status / overlay_status of skipped now means a real reason (incompatible media, undecodable/invalid asset, or — for video only — no ffmpeg), not "no library installed."
  • The API image gains Pillow (small, wheels for mac/linux/arm) and imageio-ffmpeg (bundled ffmpeg binary). cv2 stays out of the core image, keeping it lean.
  • Video remains Partial until re-verified on the Pi (browser playback of the H.264 output), per the known-issues register.

Alternatives considered​

  • Make both Pillow and OpenCV core. Rejected: cv2 is ~60MB+ with slower installs/CI and is only needed for camera/CV, not raster composition. ffmpeg gives better, browser-compatible video than OpenCV's available encoders.
  • Keep everything optional (status quo). Rejected: leaves the primary product output untestable off-hardware and the real overlay/video defects unreproducible.
