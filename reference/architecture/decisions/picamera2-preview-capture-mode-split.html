ADR-0024: Low-latency Picamera2 preview via a light video config + hardware MJPEG encoder, switching to the native still config only for capture

  • Status: Accepted
  • Date: 2026-06-21
  • Amends: ADR-0022
  • Related: ADR-0009, ADR-0008

Context​

ADR-0022 made the Picamera2 (CSI) adapter run one create_still_configuration with a native-resolution main stream and a small lores stream, serving live preview from lores and stills from main off the same continuously-running still configuration. It explicitly deferred the reference app's switch_mode design as a "thermal optimisation follow-up" if continuous full-res preview proved costly.

Hardware validation (KI-003) showed the live preview is noticeably delayed and the frame rate feels far below what a local feed should manage. Two root causes, both on the producer:

  1. The sensor mode is pinned to full resolution by the still config. On Camera Module 3 (IMX708) the native 4608x2592 readout tops out around ~14 fps, and main + lores are derived from that one sensor mode — so the lores preview inherits the ~14 fps ceiling no matter how cheap the lores stream itself is. Running native res continuously just to preview also loads the ISP, which is what makes the feed trail live motion.
  2. Preview pulled frames with per-frame capture_file(name="lores", format="jpeg"). Each frame re-entered the capture pipeline in Python instead of letting a hardware encoder push frames, adding latency and capping throughput.

The touchscreen input is instant; only the video feed lags. So this is purely the camera pipeline — not the browser compositor and not the CRT/scanline overlay.

The native-resolution ORIGINAL contract from ADR-0022 (and the runtime capture-size setting) must be preserved: this changes how the sensor runs while previewing, not the captured resolution.

Decision​

Two configurations, switched per capture (supersedes ADR-0022's single-still-config)​

Each camera builds two configurations at open time:

  • a preview config — create_video_configuration with a main (preview_main_width/height, default 2304x1296, the full-FoV 2x2-binned IMX708 mode) and a lores stream sized to the preview budget (preview_width/height), encode="lores", controls={"FrameRate": preview_fps} (default 30); and
  • a capture config — the ADR-0022 create_still_configuration with the native (or runtime-capped) main and a matching lores.

These preview knobs are runtime, admin-editable settings, not env config. They live in the SQLite-backed camera settings section (domain/settings.py), read live by the adapter through a PreviewTuning provider on every open/reconfigure — mirroring how the per-format render output sizes live in CaptureSettings (ADR-0017). The env Settings keep only the USB/fake bootstrap preview budget. A successful patch to the camera section calls refresh_preview_tuning, which switch_modes any open camera to a freshly built preview config in place, so a change applies on the next preview reconnect rather than only on a cold open.

Image flip (orientation)​

The camera section also carries an image flip applied via a libcamera Transform: flip_horizontal (mirror), flip_vertical, and a flip_scope. The flip always applies to the preview config; for the capture config it applies only when flip_scope == "all" (the camera is physically mounted mirrored/rotated, so the saved still must match), and is omitted under preview_only (selfie-mirror UX — the live preview is mirrored but the saved photo is not, so text stays readable). Transform is imported lazily (same system python3-picamera2 package), so the fake/dev/CI paths never require libcamera.

The camera idles in the preview config, so the sensor runs a fast low-resolution mode and the lores preview is unthrottled. For a still (or a burst), the adapter takes the per-role lock, stops the encoder, switch_modes to the capture config, captures main at native resolution, switch_modes back to the preview config, and restarts the encoder. The captured ORIGINAL is therefore still native — only the idle sensor mode changed.

Field-of-view must match. Preview and capture use different sensor modes, so the preview main must select a mode with the same field of view as the native still, or the preview looks cropped/zoomed relative to the saved photo (KI-003). On the IMX708 the 2x2-binned 2304x1296 mode is full-FoV (~56fps, well above 30) and is the default; the higher-fps 1536x864 mode is a cropped readout and is intentionally avoided despite its headroom. Lores inherits the main's FoV (it is an ISP downscale of the same field), so matching the main mode is sufficient.

Continuous hardware MJPEG encoder feeding a latest-frame buffer​

While previewing, a Picamera2 MJPEGEncoder runs continuously on the lores stream, writing each hardware-encoded JPEG into a small BufferedIOBase sink (FileOutput) that keeps only the newest frame and bumps a sequence counter (the reference app's PicamLoresData pattern). stream_frames blocks on that buffer's condition and yields the freshest frame, so a slow or reconnecting client never drains a backlog — the feed stays realtime. This replaces per-frame capture_file for preview; stills still use capture_file(name="main").

Preview no longer holds the per-role lock per frame (it waits on the buffer), so capture and preview still serialize through the lock, but preview only pauses for the brief switch_mode window around a capture (an accepted trade-off, as in ADR-0022).

Scope of the protocol​

The core CameraAdapter surface — discover / is_available / capture_test / capture_sequence / stream_frames / close — keeps the same signatures, so CameraService and the MJPEG endpoint are untouched. The protocol gains two small methods: measured_fps(role) (the real producer fps the live encoder is delivering, surfaced on the dashboard via GET /system/metrics; None when idle) and refresh_preview_tuning() (apply an updated camera section to any open camera). Both are no-ops on the fake and USB adapters — only the Picamera2 adapter measures fps and live-reconfigures.

Consequences​

  • Live preview runs a fast sensor mode at the configured preview_fps with hardware JPEG and latest-frame delivery, removing the ~14 fps ceiling and the per-frame Python encode — the intended fix for KI-003.
  • The captured ORIGINAL stays native (ADR-0022 contract intact); capture resolution remains the runtime, admin-editable setting. Final image quality is unchanged.
  • Each capture incurs one switch_mode round-trip (brief preview pause), pre-switchable during the countdown later if the pause is visible; bursts switch once for the whole sequence.
  • The encoder runs whenever a camera is open; it is stopped and the camera released on close(), and the buffer is signalled closed so in-flight preview generators end promptly (KI-012).
  • Exact IMX708 preview-mode framerates and the chosen preview_main size are to be verified on the target Pi (KI-003 next action); the default full-FoV 2304x1296 is the starting point. Because the knobs are now runtime camera settings with the measured fps on the dashboard, this tuning is done live in admin (no restart) instead of by editing env and redeploying.

Alternatives considered​

  • Keep ADR-0022's single continuous still config. Rejected: it pins the sensor to the slow full-res mode, which is the KI-003 latency itself.
  • Lower the capture resolution to speed preview. Rejected: it degrades the ORIGINAL and every downstream output/overlay, and is unnecessary — the mode split keeps capture native.
  • Switch transport to WebRTC / libcamera-native. Deferred: far more complex, and unnecessary for a same-device kiosk where MJPEG over loopback is effectively free; the preview provider stays abstract so this remains open if tuned MJPEG is ever insufficient.
  • A dedicated worker thread pumping capture_metadata (full reference-app shape). Not adopted: the lock-plus-encoder model delivers the same latency win with less machinery and no bespoke per-role arbitration thread; the worker's exposure-stats side benefit is unused.
