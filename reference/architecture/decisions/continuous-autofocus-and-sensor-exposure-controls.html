ADR-0025: Continuous autofocus and sensor-side exposure controls (camera-only; light tuning deferred)

  • Status: Accepted
  • Date: 2026-06-22
  • Related: ADR-0024, ADR-0022, ADR-0009

Context​

The CSI cameras (Camera Module 3 / IMX708) supported preview and native-resolution capture (ADR-0022/0024), but autofocus was never enabled and no camera controls were set, so on hardware the cameras focused at whatever the lens powered up at and the captures were often soft (KI-023). Exposure was left entirely implicit.

The full exposure story in camera-lighting.md §Exposure And Ring Light is a coordinated ExposureService that meters the scene and drives a ring light, ambient/colour light sensors, and per-event lighting profiles (normal_lights, club_mode, …). That is a hardware spike (KI-006, Stage 5) and stays deferred — it needs the real lights and sensors to design against.

What the cameras need now is the camera-only slice: focus the lens, and let the sensor's own auto-exposure/auto-white-balance ("what the sensor sees") produce a normal exposure, with a few standard libcamera knobs an operator can adjust at an event. The reference photobooth proves the minimal shape — _init_autofocus sets AfMode=Continuous + AfSpeed=Fast, and a worker reads exposure/focus metadata for diagnostics — but hardcodes the focus controls and never surfaces them. ADR-0024 explicitly noted the reference worker's "exposure-stats side benefit is unused"; this ADR puts that metadata to use.

Decision​

Scope: camera-side controls only​

This ADR lands autofocus and sensor-side exposure/white-balance controls on the Picamera2 adapter. It explicitly does not add ring-light coordination, ambient/colour light sensors, pre-capture metering ramps, custom AGC tuning files, or the normal_lights/club_mode profiles — those remain the deferred ExposureService (KI-006, Stage 5). The camera controls here are the inputs that service will later coordinate; landing them now is forward-compatible.

A CameraControls value object (mirrors PreviewTuning)​

Focus and exposure controls become an immutable CameraControls value object in adapters/camera/base.py, read live by the adapter through a controls_provider callable — the same settings→value-object seam ADR-0024 used for PreviewTuning. The provider is built in main.py from the live, admin-editable camera settings section, so an operator edit applies on the next open, and a successful patch re-applies controls to any open camera in place (no restart). The fake and USB adapters take a no-op provider.

Controls carried (all optional / defaulted, applied per-group so a sensor lacking one degrades):

  • Focus: af_mode (continuous default | auto | manual), af_range (normal | macro | full), af_speed (normal | fast), and lens_position (dioptres, 1/m) used only when af_mode == manual.
  • Exposure (sensor AEC/AGC): ae_enable (default on), ae_metering (centre | spot | matrix), ae_exposure_mode (normal | short | long), exposure_value (EV compensation, -8..8), and an optional manual pair exposure_time_us + analogue_gain used only when ae_enable is off.
  • White balance: awb_enable (default on) and awb_mode (auto | tungsten | fluorescent | indoor | daylight | cloudy).

Autofocus default + the manual-lock escape hatch​

The default is AfMode=Continuous + AfSpeed=Fast — the lens tracks the guest as they move into frame, zero config, matching the reference. Because a busy club is a low-contrast environment where continuous AF can hunt (the lens visibly breathes searching for focus, and may be mid-hunt at capture), the operator can switch the camera to AfMode=Manual with a fixed LensPosition dialled in for the booth's known standing distance. The motor then never moves — no hunting, no focus latency at capture. This is an operator setup choice (applied live like the other camera knobs), not an automatic trigger. An auto + AF-trigger-at-countdown middle mode is intentionally left for a follow-up.

Each control group is applied in its own try/except after camera.start(), so a camera/role that lacks a given control logs and continues rather than failing the open (the reference's per-control resilience; degraded-camera-mode rule).

Focus/exposure metadata on the dashboard​

The adapter gains camera_metadata(role) returning an immutable snapshot (lens_position, focus_fom, exposure_time_us, analogue_gain, digital_gain, lux, colour_temperature) read from Picamera2 capture_metadata(). It is surfaced on the admin dashboard next to the measured preview fps (ADR-0024) via GET /system/metrics, so an operator can see the lens settle and the exposure values while tuning on hardware — which is the acceptance step for both KI-023 and the camera-only part of KI-006. The read does not take the capture lock (mirrors measured_fps), so the dashboard poll never contends with a capture. Fake/USB return None.

Scope of the protocol​

The core CameraAdapter surface is unchanged. The protocol gains camera_metadata(role) and the Picamera2 adapter accepts a controls_provider; refresh_preview_tuning is extended (or paired with refresh_controls) to re-push controls on a live camera-section edit. Both additions are no-ops/None on fake and USB.

Consequences​

  • CSI captures are focused (continuous by default) and normally exposed by the sensor's own AEC/AGC/AWB, with a small admin-editable control set — the intended fix for KI-023 and the camera-only slice of KI-006.
  • The manual LensPosition lock gives a deterministic path to sharp captures even under club lighting where continuous AF hunts, without waiting on the deferred ring-light/metering work.
  • Exposure/focus metadata is observable in admin, which is how AF/AE behaviour gets verified on the Pi and how the future ExposureService design gets its real-world numbers.
  • The CameraControls value object is the forward-compatible input surface the Stage 5 ExposureService will coordinate; this ADR does not pre-empt that service's lighting decisions.
  • Bumping the settings schema is handled by the existing ensure_seeded backfill (KI-003), so older databases gain the new fields without a migration.

Alternatives considered​

  • Hardcode AfMode=Continuous like the reference and stop there. Rejected: continuous AF hunts under club lighting with no escape hatch, and the captures would still be at the mercy of the lighting; the manual-lock option is cheap and de-risks the event.
  • Land the full ExposureService (ring light + sensors + metering) now. Rejected: it is a hardware spike (KI-006) that needs the real lights/sensors to design against; the camera-only controls are independently useful and forward-compatible.
  • A dedicated metadata worker thread (full reference-app shape). Not adopted: a single capture_metadata() read on the existing dashboard-poll cadence delivers the diagnostic values without a bespoke per-role thread (consistent with ADR-0024's rejection of the same worker).
  • Trigger an AF cycle just before each capture (Auto + AfTrigger). Deferred to a follow-up: it needs session/countdown coordination; Continuous default + Manual lock covers the booth's fixed-distance case now.
