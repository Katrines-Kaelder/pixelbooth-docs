ADR-0018: Cloud/self-hosted media sharing services (rclone engine + durable outbox), with QR layered on top

  • Status: Accepted
  • Date: 2026-06-16
  • Deciders: Jonas Landsfeldt, Claude

Context​

Guests must be able to scan a QR on the booth and get their own photo/GIF/video. The earlier implicit model was local-network serving — the guest's phone reaches the Pi directly and downloads the file. At the real venue (a busy Friday bar at Aarhus University) that is infeasible:

  • The bar Wi-Fi carries payment terminals; guests can't and shouldn't join it.
  • The venue Wi-Fi is not usable either.
  • The Pi is unreliable joining venue Wi-Fi and poor at hosting its own access point.

So guest phones cannot reach the Pi. The booth must instead push each finished render to a remote host and put a public URL in the QR, so the guest downloads over their own mobile data. Booth connectivity and guest connectivity become fully decoupled.

The reference app (photobooth-app.org) already does exactly this: rclone uploads each file to a remote (FTP/WebDAV/NextCloud/S3), and a single static "sharepage" HTML loader renders the media; the QR points at a link template with a {filename} placeholder. Pixelbooth adopts this shape but makes it more robust, more configurable, and seamless, reusing infrastructure we already have:

  • domain/sync.py RemoteRecord (type: rclone, output_selection incl. "share", trigger: "on_capture_complete", RetryPolicy, per-remote status).
  • services/sync.py RemoteService (CRUD/test/enqueue/jobs) and _process_job.
  • services/jobs.py JobQueue — the durable, retryable outbox (ADR-0011).
  • domain/settings.py SharingSettings and EventSettings.slug (scopes QR links); credential redaction via ConfigService secret-suffix rules.
  • services/media.py shared timestamp stem per item → deterministic object keys.
  • rclone is already the chosen cloud/SFTP transport (ADR-0005).

This supersedes the implicit local-serve assumption and brings the Stage 8 share path forward, so it is a significant, hard-to-reverse decision and gets an ADR.

Decision​

Share/sync services are built first; QR is a follow-on slice layered on top. The upload engine + durable outbox + admin config is the real, independently-valuable work (it gives off-site backup/export before any QR exists) and is QR's hard dependency. QR itself is thin: an image of a URL, a pending→ready state, and a static sharepage.

  1. Upload-on-render via rclone to a designated share remote. When a render reaches final, the booth enqueues a share-upload job for each enabled on_capture_complete remote whose output_selection matches — in particular the share remote. Upload runs behind the RemoteService/adapter boundary on the JobQueue outbox, never on the capture path (fast-capture-path). The first reference transport is S3-compatible, concretely MinIO (rclone s3 backend, provider = Minio, self-hosted on the operator's own server); the same generalized remote model then carries self-hosted WebDAV/FTP/SFTP and Google Drive (ADR-0005, one rclone engine).

  2. The QR encodes a configurable public-link template. link_mode is sharepage (default) or direct; the template resolves placeholders ({key}, {event}, {filename}) against the uploaded object's deterministic key. Object keys are {event_slug}/{stem}-{token}.{ext} with an unguessable token so URLs are non-enumerable yet computable without a round-trip.

  3. A themed static sharepage renders the media. The QR points at an arcade grab page (our sharepage), not a raw file: it reads the key, fetches the bytes, and shows a hero (image/GIF/<video>) with a big SAVE button and event branding.

  4. The host stays a dumb static/file server — NO server-side script. No PHP/dl.php, no Node/Python runtime on the host. Sharing is static sharepage + uploaded bytes only. Upload is rclone; any setup helper is a local operator shell/Python script, not host-side code.

  5. Robustness beyond the reference (the "better than reference" rules):

    • A QR becomes scannable only once the upload is verified live (per-media share status pending → ready, surfaced over the media.shared event).
    • The durable retry outbox drains on reconnect; uploads survive restarts and offline periods.
    • Unguessable object keys plus host-side lifecycle/expiry give privacy.
    • One rclone engine serves all transports, so adding WebDAV/NextCloud, FTP/SFTP, or S3+CDN later is configuration, not new integrations.

Consequences​

  • Easier: guest sharing works on any network because it is decoupled from the Pi; off-site backup/export is available before QR exists; one engine and one outbox cover every transport; the sharepage deploys once to any static host.
  • Constrained: QR requires a configured, network-reachable share remote and a deployed sharepage — until both are wired and verified against a real remote, end-to-end QR is Unverified (tracked in known-issues.md). Credentials live behind the ConfigService secret-redaction boundary. The no-server-side-script rule means all grab-page logic is client-side and all expiry/lifecycle is the host's responsibility.
  • Sequencing: Phase 1 (share/sync services + admin remotes, S3/MinIO end-to-end first) ships independently; Phase 2 (per-media share status, media.shared, GET /media/{id}/qr.svg, the sharepage, kiosk/gallery QR UI, share-host setup helper) builds on top. See roadmap.md Stage 8.

Alternatives considered​

  • Local AP / same-LAN serve (phone reaches the Pi). Rejected: infeasible at the venue — payment-terminal Wi-Fi is off-limits, venue Wi-Fi unusable, Pi AP/Wi-Fi unreliable.
  • Public Immich instance as the QR backend. Rejected: requires exposing/operating a server with its own auth; Immich stays a share/sync remote (ADR-0005), not the QR public-link backend.
  • Server-side download script on the host (PHP dl.php or similar). Rejected by user constraint: the host must stay a dumb static/file server; a runtime adds attack surface, hosting requirements, and per-host divergence.
  • Google Drive as the recommended QR backend. Rejected: its public per-file link UX for "scan and save" is poor and quota-bound. Google Drive is kept as a share/sync remote (backup/export), not the QR public-link backend.
  • Per-provider SDKs (S3 SDK, Drive SDK, …). Rejected (ADR-0005): multiplies integrations and failure modes that one controllable rclone binary already abstracts.
