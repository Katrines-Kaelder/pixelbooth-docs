ADR-0016: Media file-store layout, naming, and frame lifecycle

  • Status: Accepted
  • Date: 2026-06-16
  • Amends: ADR-0015
  • Related: ADR-0007, ADR-0012

Context​

MediaService stored every item under an opaque per-id folder with variant filenames: <data_dir>/media/<media_id>/{preview,final,thumb,frame-N}.<ext>. Two problems (KI-015):

  1. Operator-hostile. An operator browsing the data dir (or an SD card) sees med_… folders with no idea what each holds. They want human-readable timestamp filenames and clear top-level folders.
  2. Source frames leak. Composed formats (collage/GIF/boomerang/video) wrote frame-0…frame-N files that are only render inputs but were kept forever next to the gallery media.

Confirming scope also surfaced that the ADR-0015 capture-size defaults (1920×1440, 4:3) silently downscaled the 12 MP (4608×2592, 16:9) sensor to ~2 MP — a placeholder, not a deliberate quality choice.

Decision​

On-disk layout (operator-facing)​

Media bytes live in three human-readable folders under <data_dir>/media/:

  • ORIGINAL/<ts>.<ext> — the raw full-resolution capture, archived for reprocessing.
  • PREVIEW/<ts>.<ext> — the downscaled instant preview the kiosk paints immediately.
  • PROCESSED/<ts>.<ext> and PROCESSED/<ts>-thumb.<ext> — the rendered final + its thumb.

<ts> is the capture time YYYY-MM-DD-HH.MM.SS (UTC) — dashes separate the date, dots the time (e.g. 2026-06-16-14.30.05.jpg). Same-second captures get a -2, -3, … suffix. One shared base stem ties an item's files together across the three folders; the trailing .<ext> stays the file extension. SQLite remains authoritative for metadata and stores the relative path of each artifact (original_path is added; preview_path/final_path/ thumb_path are reused). The HTTP API is unchanged — it still serves ?variant=preview|final|thumb; the reorg is internal to MediaService.

Frame lifecycle​

Composed-format source frames are transient render inputs, staged under media/.tmp/<media_id>/frame-N.<ext>, read by the render.final job, and deleted after a successful render. They are never gallery/media files. delete(force=True) also sweeps any leftover temp frames; failed renders leave them for debugging.

Capture-size defaults (amends ADR-0015)​

Each capture-size axis defaults to 0 = "no cap / keep native", so PROCESSED keeps the camera's native resolution by default (full 12 MP, correct aspect). All formats are still per-format configurable (PIXELBOOTH_CAPTURE_{STILL,VIDEO,GIF}_{WIDTH,HEIGHT}). The downscale-to-fit machinery (fit_within, ADR-0015) stays; 0 simply makes it a no-op.

Consequences​

  • The data dir is self-describing: originals to keep/reprocess, processed outputs to share/print, and a transient .tmp/ that empties as renders complete.
  • No more leaked source frames; storage for composed formats drops to original + final + thumb + preview.
  • PROCESSED stills keep native resolution by default — good for prints. GIF at native is very large/slow; operators should cap it (PIXELBOOTH_CAPTURE_GIF_WIDTH/HEIGHT).
  • Stills are stored twice when no overlay is applied (an ORIGINAL and an identical native PROCESSED); acceptable given retention/capacity controls. Dedup is a possible later optimization.
  • No data migration: the layout applies to new captures only; the PR is unmerged/unverified and dev data is disposable. Old rows keep resolving via their stored paths.
  • ORIGINAL is tracked and deletable but not API-served yet (no ?variant=original); exposing it for operator/photographer download is a possible follow-up.

Alternatives considered​

  • Keep per-id folders, just rename files. Rejected — still opaque to operators.
  • Retain source frames as media. Rejected — they are render inputs, not outputs, and bloat the gallery/store.
  • Migrate existing dev media. Rejected — disposable pre-release data; fresh-forward is simpler and breaks nothing.
