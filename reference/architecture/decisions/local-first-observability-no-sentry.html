ADR-0006: Local-first observability; no Sentry

  • Status: Accepted
  • Date: 2026-06-11
  • Deciders: Jonas Landsfeldt, Claude

Context​

The roadmap and expanded-capabilities.md planned an "optional Sentry integration" for external error reporting. For a single local-first booth appliance run by bar volunteers, a hosted error-tracking SaaS is overkill: it adds an external account, network/privacy considerations, and a dependency that does nothing when the booth is offline — which is its normal operating condition. The actual need is plain, correctable logging: when something breaks, an operator or developer can find out what and fix it.

Observability building blocks (structured logs, diagnostic IDs) are also used as a cross-cutting requirement from Stage 4 onward — hardware-boundaries requires every hardware/service operation to return a user-safe message, a recovery suggestion, and a developer diagnostic ID. Parking the entire observability stack in the final stage (Stage 10) left those IDs referenced before their substrate existed.

Decision​

Pixelbooth's observability is local-first and Sentry is dropped.

  • The logging + diagnostic-ID foundation moves early, into Stage 2.5: structured JSON logs with event/context fields, stable diagnostic-ID generation, and persisted, rotated local log storage that survives restarts and is queryable.
  • Stage 10 keeps only the surfaces, not the foundation: the admin log viewer with filtering, diagnostic-ID lookup, Commander run history, the safe diagnostic bundle generator (version, config summary, recent logs, queue/hardware status, task history, secrets redacted), and frontend error boundaries.
  • No Sentry task. External error reporting is removed as a planned feature. If a remote sink is ever wanted, it enters through a generic, opt-in log/error export extension point — not a hardcoded Sentry dependency — and only via a future ADR.

Consequences​

  • Easier: the booth is fully diagnosable offline; diagnostic IDs have a real backing store from the moment they are first referenced (Stage 4); no external account, SDK, or privacy surface to manage.
  • Constrained: there is no automatic crash aggregation across booths; field diagnosis relies on the local log store and the diagnostic bundle an operator exports. Acceptable for the current single-booth scope.
  • Follow-up: Stage 2.5 owns the logging/diagnostic-ID foundation; Stage 10 builds the admin-facing viewer/bundle/error-boundary surfaces on top of it.

Alternatives considered​

  • Keep optional Sentry (original plan). Rejected as overkill for a local-first, usually-offline single appliance; the value (hosted aggregation) does not justify the dependency and privacy surface at this scope.
  • Logs only, no diagnostic IDs. Rejected: hardware-boundaries requires correlatable diagnostic IDs on user-visible errors, so IDs are part of the foundation, not optional.
  • Full self-hosted telemetry stack (Loki/Grafana/OTEL collector). Rejected: heavy for a Pi appliance whose primary need is "read recent logs and fix the error"; structured local logs plus the diagnostic bundle meet that need.
