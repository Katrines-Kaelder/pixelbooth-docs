ADR-0002: Compose-first development shape for the Stage 2 skeleton

  • Status: Accepted
  • Date: 2026-06-11
  • Deciders: Jonas Landsfeldt, Claude

Context​

Stage 2 (Deliberate Project Skeleton) needs a concrete development/runtime shape. The architecture (system-architecture.md §Process Model) calls for explicit service boundaries and a containerized direction, while keeping V1 simple. The documentation index lists "first version of the Docker Compose layout and which functions need host-side helpers" as an open question.

This ADR resolves the development layout only. Privileged Pi hardware access (Picamera2/libcamera, GPIO) and the production Pi deployment are explicitly out of scope here — they belong to Stage 3 and Stage 7.

Decision​

The repository keeps both deployables under a single apps/ directory plus shared dev tooling:

  • apps/api — Python/FastAPI runtime (the control API). The project root holds packaging/ops (pyproject.toml, Dockerfile, tests/); the importable code is the pixelbooth package inside it. It owns the service graph as in-process modules with explicit boundaries, fronted by ControlApi. The HTTP/WebSocket handler layer is pixelbooth/http/ (named http, not api, to avoid a redundant apps/api/.../api/ path).
  • apps/web — React/TypeScript/Vite kiosk/gallery/admin frontend.

There is intentionally no top-level services/ directory yet. Service orientation lives in the module boundaries inside the pixelbooth package and the stable /api/v1 contract, not in the folder layout — a boundary can move to its own process/container without changing the public API regardless of directory depth. The first genuinely separate deployable (the Pi Zero 2 W marquee agent, Stage 6; or privileged host-side helpers, Stage 7–8) introduces a sibling directory at the moment it actually lands, when its real name and shape are known.

Development runs through docker-compose.yml at the repo root: an api service and a web service on a shared network, with the Vite dev server proxying /api to the API container so the kiosk and API are same-origin (matching the production appliance, where the built UI is served behind the API). The root package.json scripts (pnpm dev, pnpm test, pnpm lint, ...) wrap the common commands; they are conveniences that delegate to Compose and the per-package tooling, never the target deployment.

Hardware is faked via adapters selected at the composition root (main.create_app), driven by PIXELBOOTH_USE_FAKE_HARDWARE / PIXELBOOTH_FAKE_CAMERA_ROLES. No real-camera code ships in this stage.

Consequences​

  • Easier: off-Pi development and CI; swapping a real adapter in at one seam; later splitting render/sync/network/print/marquee services into their own Compose services without changing the public API.
  • Constrained: the two-service Compose file is a dev shape. Production Pi Compose files, host-side privileged helpers, and systemd units are deferred (Stage 7) and must be added deliberately, not by promoting this file as-is.
  • Follow-up: real Picamera2 adapter and Pi container/host strategy (Stage 3); worker pool + media pipeline services (Stage 4+).

Alternatives considered​

  • Single combined container (API serves built UI). Simpler to run, but collapses the API/UI boundary during development and slows the frontend iteration loop (no Vite HMR). Kept as the likely production packaging, not dev.
  • No containers; bare pnpm + uvicorn scripts. Lowest friction to start, but drifts from the documented containerized direction and reproduces environment differences the appliance must avoid. Rejected as the primary shape; the root package.json scripts still offer direct local commands for convenience.
  • Polyrepo (separate API and web repos). Premature; the contract is co-evolving and a monorepo keeps the API/types/contract changes atomic.
  • Top-level services/ wrapper for the backend (services/api). Anticipates multiple backend services, but today it would hold a single occupant with a speculative future, and adds a services/api/.../api/ path redundancy. Rejected as premature (YAGNI); reintroduce services/ by a cheap git mv when a real second deployable arrives. Until then both deployables live under apps/.
