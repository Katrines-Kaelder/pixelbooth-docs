ADR-0028: Custom ESP32 lighting adapter (newline-JSON over serial/TCP)

  • Status: Accepted
  • Date: 2026-06-23
  • Deciders: Mads, Claude

Context​

Besides WLED (ADR-0027), Pixelbooth supports a custom ESP32 firmware for booths whose ring light / decorative LEDs run Pixelbooth's own controller rather than WLED. Unlike WLED, this is a Pixelbooth-defined contract — the firmware contract was already documented in .docs/architecture/camera-lighting.md §Custom ESP32 Adapter: newline-delimited JSON over serial or TCP, with a {"command":"preset",…} request and a {"ok":true,…} response.

The adapter must implement the same LightingAdapter protocol, be testable without ESP32 hardware (none on hand), and keep the serial dependency off the default/CI install.

Decision​

Add an Esp32LightingAdapter driving the documented newline-JSON contract behind an injectable transport, with stdlib-TCP and optional-pyserial-serial transports.

  • Wire contract (unchanged from the docs). One JSON line per request, one per response:
    • request {"command":"preset","preset":"<preset>","brightness":<1-255>}, plus an optional "cct" (0 warm … 255 cold) for ring-light calibration that firmware may ignore.
    • response {"ok":true,...,"firmware":"..."}; ok:false (or no line / parse error) is a fault. The firmware string becomes the device label.
  • Injectable transport. Esp32Transport is a small Protocol (request(line)->line, close()). The adapter logic (command framing, calibration scaling, reachability, firmware capture) is unit-tested against a fake transport; failures raise a plain exception that LightingService wraps as ServiceError.
  • Two real transports in-module. TcpEsp32Transport uses the stdlib socket (no dependency) and is verified against a loopback socket server; SerialEsp32Transport uses pyserial, shipped as an optional esp32 extra and lazy-imported (mirroring the usb/opencv pattern) so the dev/CI path and the TCP path never need the wheel. Both connect lazily, so a missing device degrades on first apply_preset, not at startup.
  • Config-driven. The transport choice (serial/TCP) and connection live in the same admin-editable lighting settings section as WLED; the factory builds the transport + adapter from the resolved LightingConfig (ADR-0027), so the service rebuilds it live on an edit.

Consequences​

  • Easier: a real ESP32 path developed/tested off-hardware (fake transport for logic, a real loopback socket for TCP framing); no new core dependency (TCP is stdlib; serial is opt-in); the same config/factory/calibration seam as WLED.
  • Constrained: real serial/TCP timing and the firmware's own behaviour are unverifiable here — real-device verification is a new known-issue. The serial path needs the esp32 extra installed on the wired host.
  • Follow-up: exposure profiles feed brightness_ceiling/cct (Stage 5 commit 3); the admin lighting frontend exposes the transport/port fields and a test action.

Alternatives considered​

  • Reuse WLED for everything (drop the custom firmware). Simpler, but the booth's bespoke controller is a real target the docs already committed to; some builds are not WLED. Rejected — both adapters coexist behind one protocol.
  • Make pyserial a core dependency. Simpler packaging, but forces a serial wheel onto every dev/CI/TCP environment that never opens a serial port. Rejected for the optional esp32 extra + lazy import.
  • A third-party serial-over-TCP/asyncio framework. Overkill for one line-oriented request/response; stdlib socket + pyserial cover both transports with no extra moving parts. Rejected.
