ADR-0023: Graceful shutdown that releases camera handles on the first interrupt

  • Status: Accepted
  • Date: 2026-06-21
  • Related: ADR-0008, ADR-0009, ADR-0022

Context​

The booth keeps two endless responses open: the live preview MJPEG response (GET /cameras/{id}/stream.mjpg, whose adapter stream_frames loop runs while True) and the /events WebSocket fan-out (blocked on queue.get() forever). The API was launched on the Pi with a bare python -m uvicorn pixelbooth.main:app.

Hardware validation (KI-012) found that stopping the app did not reliably shut everything down. uvicorn's default graceful shutdown waits for in-flight responses to finish, but the open endless responses never end, so the first Ctrl+C hung. A forced second Ctrl+C exited uvicorn without running the FastAPI lifespan, so the lifespan finally that calls CameraService.close() never ran. The Picamera2/V4L2 handle was leaked and the device stayed locked — the app could not restart without rebooting the Pi (Pipeline handler in use by another process, Camera __init__ sequence did not complete). This is the documented uvicorn behaviour in encode/uvicorn#1579; the project's reference photobooth hit the same wall and worked around it with server.force_exit = True.

Revised after on-Pi validation (2026-06-21). This ADR first tried to keep uvicorn's graceful drain and end the streams cooperatively (a one-way threading.Event per endless response, polled by each loop, signalled from handle_exit, with a bounded timeout_graceful_shutdown backstop). On hardware that proved fragile and made shutdown worse: a second endless response (/events) was easy to miss, a preview blocked inside a driver capture_file call cannot check the flag until it returns (so the bounded timeout still fired and force-cancelled tasks with noisy tracebacks), and — fatally — even after the lifespan released the cameras the process hung after Application shutdown complete and never exited. The decision below replaces the cooperative mechanism with the reference photobooth's approach, which is simpler and proven on the same hardware class.

Decision​

A dedicated process entrypoint owns shutdown​

Run the API as python -m pixelbooth (a new pixelbooth/__main__.py) instead of the bare uvicorn CLI. It builds the app and a uvicorn.Server subclass, sets server.force_exit = True, and runs it. The bind address comes from new PIXELBOOTH_HOST / PIXELBOOTH_PORT bootstrap settings.

force_exit abandons the endless responses on the first signal​

server.force_exit = True (the reference photobooth's documented workaround for uvicorn #1579) makes uvicorn abandon in-flight connections on the first signal instead of waiting for them: its shutdown skips the "wait for connections / background tasks" loops entirely. So the open preview stream and /events socket no longer block the first Ctrl+C, and no timeout_graceful_shutdown wait or task force-cancellation happens. We deliberately do not try to end the streams cooperatively — abandoning them is what makes shutdown robust regardless of where a stream is blocked.

Cameras are released synchronously in the signal handler​

In uvicorn 0.49, force_exit = True also skips the FastAPI lifespan shutdown (if not self.force_exit: await self.lifespan.shutdown()), so the lifespan finally that releases the cameras no longer runs. The uvicorn.Server subclass therefore overrides handle_exit to call CameraService.close() synchronously, right there in the signal handler, before delegating to super().handle_exit() — mirroring how the reference app releases resources from its own signal handler. close() is idempotent, never raises, and stops the preview loops before releasing the device handles.

A hard process-exit so the process actually dies​

Releasing the camera handles is necessary but not sufficient: on the Pi a libcamera/V4L2 worker thread can outlive Picamera2.close(), and because it is non-daemon, the interpreter's own shutdown joins it forever — so the process prints its last log line and then hangs, never freeing the camera fds, so the restart this ADR is about still fails. (The reference app sidesteps this because its camera worker thread is daemon=True; our single-process design runs Picamera2 in-process.) After the server's run() returns, main() flushes logging and calls os._exit(0). The cameras were already released in handle_exit, so this only skips joining a stuck C-extension thread.

Consequences​

  • A single Ctrl+C / SIGTERM stops the app, releases the cameras, and the process actually exits — so the app restarts without rebooting the Pi, the change that makes on-hardware iteration practical (KI-012).
  • force_exit abandons in-flight connections at signal time (clients of the preview / event stream simply reconnect on the next launch), which is acceptable for a kiosk appliance and far more robust than chasing every endless response cooperatively.
  • force_exit skips the lifespan finally, so cleanup other than the camera release (DB close, job-queue stop) does not run on shutdown. This is safe: SQLite is crash-consistent (WAL) and the job outbox is durable, both recover on next start. The camera release — the only cleanup that affects restart — is done explicitly in handle_exit.
  • The docker-compose dev path keeps uvicorn --reload (fake hardware holds no device lock), so backend hot-reload is unaffected.

Alternatives considered​

  • Cooperative stream stop + bounded timeout_graceful_shutdown (first version of this ADR). Keeps shutdown "graceful" and the lifespan intact, but proved fragile on hardware: it must track every endless response, a stream blocked in a driver call can't honour the stop flag in time, and it still left the process hanging on the non-daemon libcamera thread. Replaced by force_exit + os._exit.
  • os._exit(0) alone, no force_exit. Would defeat the post-shutdown hang but not the first-signal hang: without force_exit uvicorn still waits on the open endless responses before run() ever returns, so os._exit is never reached on the first Ctrl+C.
  • Make the camera worker daemon and rely on the lifespan (reference-app shape). Our adapter has no long-lived worker thread of its own to mark daemon; the blocking thread is inside libcamera/Picamera2, which we do not control. Releasing in handle_exit and hard-exiting is the equivalent that works for an in-process camera.
