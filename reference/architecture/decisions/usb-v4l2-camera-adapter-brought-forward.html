ADR-0008: USB/V4L2 camera adapter brought forward into Stage 3

  • Status: Accepted
  • Date: 2026-06-12
  • Deciders: Jonas Landsfeldt, Claude

Context​

Stage 3 (Raspberry Pi Camera Spike) exists to exercise real camera hardware safely while the fake adapters keep off-Pi development reliable. Its task list assumes the booth's two CSI cameras — 2× Raspberry Pi Camera Module 3 (normal + wide) — and a Picamera2/libcamera adapter.

Two forces collide with that assumption:

  • The host/container strategy for Picamera2/libcamera access is still an open question (.docs/README.md §Open Questions). Standing it up is itself unscoped work.
  • The hardware on hand for this spike is a single USB (UVC) webcam, not the CSI modules. Waiting for 2× Camera Module 3 to verify any real-camera path would block the spike's whole purpose.

USB/V4L2 support was already planned, just sequenced later: expanded-capabilities.md records "CameraService: Raspberry Pi Camera Module 3 first, USB/V4L2 later." Promoting it is a change to a documented hardware contract (a new real adapter type and how a single device maps onto the normal/wide role model), which is why this is recorded as an ADR rather than a prose tweak (docs-first-architecture).

The existing seam makes this cheap: CameraAdapter is a small Protocol (discover / is_available / capture_test) that CameraService depends on abstractly, and the service's availability table already covers both / normal_only / wide_only / none.

Decision​

Bring the USB/V4L2 camera adapter forward into Stage 3 as the first real camera adapter, ahead of Picamera2.

  • Backend selection. A camera_backend setting selects the adapter: fake (default, off-Pi) | usb (this ADR) | picamera2 (later this stage, still blocked on CSI hardware + the libcamera/container question). The fake adapter stays the default and is never removed.
  • Single-device role mapping. A USB UVC camera is one physical device. It maps to one configurable role (usb_camera_role, default normal) addressing a single /dev/video* device (usb_camera_device). This deliberately reuses the existing normal_only (or wide_only) degraded-mode path — no new availability state, no changes to CameraService's classification. Multiple USB cameras mapping to both roles is a later extension, not V1.
  • Capture path. OpenCV VideoCapture over V4L2 for discovery, still capture, and the MJPEG preview stream. cv2 is lazy-imported and ships as an optional dependency extra (usb → opencv-python-headless), so the fake/dev path, CI, and tests never require the native imaging stack.
  • Preview-frame contract. The CameraAdapter Protocol gains a stream_jpeg(role) frame generator. The previously-stubbed GET /cameras/{id}/stream.mjpg endpoint becomes a real multipart/x-mixed-replace MJPEG stream, served identically by the fake adapter (animated placeholder) and the USB adapter (live V4L2 frames), targeting ~720p @ ~15fps.

Consequences​

  • Easier: a real-camera path the spike can verify today on commodity USB hardware; the still-open Picamera2/libcamera container question no longer blocks Stage 3 progress; the fake path and tests stay dependency-light because cv2 is optional and lazy; degraded-mode behaviour gets exercised for real because one USB camera is a single-role booth.
  • Constrained: a single USB camera cannot exercise true dual-camera (both / dual) modes — those still need the CSI modules or a second USB device. opencv-python-headless is a heavy native dependency; it is intentionally isolated to the usb extra and the USB adapter module. Real frame-rate/latency/CPU numbers must be measured on the Pi, not the dev laptop.
  • Follow-up: the Picamera2/CSI adapter and its host/container strategy were the remaining Stage 3 gate; both are now resolved in ADR-0009 (bare-metal Pi OS + system python3-picamera2). Preset-level camera fallback (required / prefer / any / dual) gets a pure resolver here but full preset-catalog wiring lands with the session/preset work (Stage 3.5). Capture quality and lighting are not tuned until Stage 5.

Alternatives considered​

  • Wait for 2× Camera Module 3 and implement Picamera2 first (as originally sequenced). Truest to the documented booth hardware, but blocked on hardware not on hand and on the unresolved libcamera/container question — it would stall the spike indefinitely. Rejected for now; Picamera2 remains the next real adapter once that hardware/question is resolved.
  • Map the USB camera to both normal and wide from one device. Would let a single webcam appear to satisfy dual-camera modes, but it fakes capability the hardware does not have and muddies degraded-mode semantics. Rejected: one device = one role; both/dual require real second optics.
  • pyuvc / raw V4L2 (v4l2-python) instead of OpenCV. Lighter and more direct for UVC, but OpenCV is already the project's planned imaging/CV stack (render pipeline, future filters/face landmarks) and gives JPEG encode + capture in one well-supported dependency. Rejected to avoid a second imaging library; revisit only if OpenCV proves too heavy on the Pi.
  • Make opencv-python-headless a core dependency. Simpler packaging, but forces a heavy native wheel onto every dev/CI environment that only ever uses fakes. Rejected in favour of the optional usb extra + lazy import.
