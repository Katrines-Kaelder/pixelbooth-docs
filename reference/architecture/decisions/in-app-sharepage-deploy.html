ADR-0021: In-app sharepage deploy (reuse the stored share-remote credentials)

  • Status: Accepted
  • Date: 2026-06-20
  • Amends: the Remotes/Sharing posture in admin-control-plane.md §Remotes And Synchronizer and the /sync/remotes contract in api-contract.md
  • Related: ADR-0018 (cloud share services + QR, the standalone share-host-setup helper), ADR-0020 (write-only remote secrets), ADR-0005 (one remote model, transports as adapters)

Context​

Guest QR sharing needs a static index.html (the arcade grab page, apps/sharepage/) sitting at the root of the same public host the booth uploads photos into. ADR-0018 shipped a standalone operator script, scripts/share-host-setup/sharehost.py, to push that page and print the values to paste into admin.

In practice the operator has, by the time they deploy the page, already created the share remote in admin: endpoint, bucket, access key, and the write-only secret (ADR-0020) are persisted server-side. The CLI helper asks for all of it again — re-typing the endpoint, bucket, access key, and re-supplying the secret on the command line (shown once, easy to get wrong, and a second place credentials are handled). That redundancy is the friction: the booth runs from the cloned repo tree on the Pi (pi-pixelbooth-setup.sh), so the API process already has both the sharepage bytes on disk and the remote's credentials in SQLite.

Decision​

Let the API deploy the sharepage, reusing the share remote's stored settings.

  1. New endpoint. POST /sync/remotes/{id}/deploy-sharepage → RemoteService.deploy_sharepage. It loads the remote, and uploads the sharepage to the remote's root via the adapter's single-file upload_file(settings, index, "index.html") — the same privileged boundary and deterministic path uploads already use. The page lands at index.html under the remote's path, i.e. the same root photos go under.

  2. Reuse stored credentials. The upload uses the remote's raw stored settings (secret included). The secret never leaves the server — the response carries only ok, a user-safe message, the public url (when sharepage_url is set, for one-click verify), and a diagnostic_id on failure.

  3. Static-host remotes only. Deploy is accepted for rclone (the real QR backends: S3/MinIO, WebDAV, FTP/SFTP) and local_folder (the testable, no-network path). Other transports (immich/usb/rsync) are not static hosts and return 422.

  4. Sharepage location is configuration, not bundling. The API resolves the page via PIXELBOOTH_SHAREPAGE_PATH, defaulting to the repo-tree apps/sharepage/index.html next to the running app. apps/sharepage stays the single canonical source — no copy into the API package, no Docker build-context change. (The Pi runs from the cloned tree, so the file is already on disk beside the API.)

  5. CLI helper demoted to fallback. scripts/share-host-setup keeps working and remains the way to stand up a host before a remote exists in admin, or in CI. The in-app Deploy sharepage button (on the share remote's row) is the default path.

  6. Force HTTP/1.1 for the rclone transport. Every rclone invocation (test/sync/upload) runs with GODEBUG=http2client=0, disabling Go's client-side HTTP/2. Some S3-compatible reverse proxies (observed: MinIO behind Caddy) drop the HTTP/2 request body on an upload PUT and return a 502 with a 0-byte body, while a bodyless GET/list succeeds — so "Test passes but Deploy/sync fails" looks like a credentials problem when it is really the proxy mangling h2 uploads. HTTP/1.1 sidesteps the whole class; S3 needs no h2 and rclone still parallelises across connections. Operators no longer have to disable h2 on their own proxy. The value merges with any operator-set GODEBUG rather than clobbering it.

Consequences​

  • One-time setup is: create the remote → Test → Deploy sharepage → enable QR. No second tool, no re-typed endpoint/bucket/key, and the secret is handled in exactly one place (admin, write-only per ADR-0020).
  • The deploy reuses the existing adapter boundary, so no new privileged code path and no new credential surface; it inherits the adapters' upload_file behavior and error reporting.
  • The page's bytes must be reachable by the API process. The default suits the repo-tree Pi layout; other layouts set PIXELBOOTH_SHAREPAGE_PATH. A missing file returns a clear sync.sharepage_missing error rather than failing silently.
  • apps/sharepage remains canonical for both the in-app deploy and the CLI helper, so there is no duplicated copy to keep in sync.
  • For object stores the operator must still make the uploaded objects publicly readable; the deploy confirms the page uploaded, not that the host serves it publicly (the QR troubleshooting in the operator guide covers the public-read check).

Operational notes (rollout findings)​

Shaking this out against a real MinIO AIStor host behind Caddy surfaced three environment issues, now handled so the next operator doesn't repeat the hunt:

  • rclone must be current. The Debian apt rclone is years old and fails to transmit the upload body to modern S3 servers (truncated-body 502/499 IncompleteBody). pi-pixelbooth-setup.sh and the API Dockerfile now install the official rclone binary, not the apt package.
  • Addressing + region default to MinIO. A signed request that uses virtual-host addressing (rclone/aws-sdk-go-v2's default) is 308-redirected by MinIO/AIStor unless MINIO_DOMAIN + wildcard DNS are set, and a signed request whose region scope doesn't match is rejected — both surface as PermanentRedirect (anonymous requests are unaffected, which is why an unsigned curl succeeds while rclone fails). The S3 backend now defaults to path-style addressing (force_path_style=true) and region us-east-1, both overridable and exposed on the remote form, so MinIO works without per-operator tuning.
  • HTTP/1.1 is forced for the transport (decision #6) — h2 request bodies were dropped by the proxy on upload.
  • The QR targets index.html explicitly ({sharepage_url}/index.html?id={key}, not a bare {sharepage_url}/?id=). Object stores don't serve a directory index, so a bare directory link hits the bucket root (a listing / the MinIO console login) instead of the page; index.html is a real object everywhere. Consequently sharepage_url must point at the root that holds both the page and the photos — on S3/MinIO that includes the bucket (https://host/bucket), since the object key carries only {event}/{file}.
