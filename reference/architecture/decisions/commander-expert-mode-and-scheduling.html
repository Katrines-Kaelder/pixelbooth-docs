ADR-0019: Commander expert mode (gated host shell) and task scheduling

  • Status: Accepted
  • Date: 2026-06-17
  • Amends: the Commander posture in admin-control-plane.md §Commander ("Do not expose free-form shell execution in the admin UI")
  • Related: ADR-0004 (admin access control), ADR-0006 (audit/diagnostic ids), ADR-0011 (shared job queue)

Context​

Commander shipped as a fixed registry of typed, allowlisted tasks with the explicit rule: never arbitrary shell from the UI. That posture is right for the default appliance — a busy bar booth operated by non-engineers — where a free-form shell is a remote-code-execution surface behind a single shared operator secret.

But the booth owner is also its developer/maintainer, who legitimately needs to:

  1. Run a one-off command on the Pi during setup/debugging without SSHing in.
  2. Schedule routine maintenance (e.g. a nightly diagnostic bundle or queue retry) instead of remembering to click it.

These are real operator needs, so we extend Commander rather than route owners to a separate, unaudited channel (SSH) where nothing is recorded.

Decision​

1. An off-by-default, gated expert mode that runs a host shell command​

We add a single dangerous task, shell_exec, that runs a command on the host and captures its output — but behind multiple gates so the safe default is preserved:

  • Off by default. A new system.expert_mode_enabled setting (default false) governs it. When off, the task is not listed and run rejects it. Turning it on is itself an admin-gated settings change (ADR-0004) and is visible in the dashboard.
  • Dangerous + confirm. risk_level="dangerous", requires_confirmation=true; the UI forces an explicit confirm step before sending confirm: true.
  • Bounded + audited. The executor runs the command with a hard timeout, captures combined stdout/stderr (truncated), and — like every Commander run — records a CommandRun with status, output summary, diagnostic id, and timestamps (ADR-0006). No new audit path; it reuses the existing one.
  • Still a typed task. It flows through the same allowlist/executor seam — the allowlist now contains one task whose parameter is a command, rather than the UI gaining a generic shell. There is no second execution path.

This is deliberately a security trade-off the owner opts into, not a default. The appliance ships safe; an expert turns the key. The risk (a leaked operator secret now implies host command execution) is accepted and documented here and in the setting's admin copy.

Privileged-boundary note (hardware-boundaries): the shell executor is itself the privileged boundary — a single, audited, timeout-bounded executor — rather than shell calls scattered through the app. A future hardening step can move execution to a separate host helper with its own allowlist; the seam (one executor behind the task registry) does not change.

2. Scheduling allowlisted tasks ("run when")​

Operators can schedule any allowlisted task (including shell_exec only while expert mode is on) to run on an interval. Schedules are persisted in SQLite and a lightweight scheduler enqueues due runs through the existing Commander run path, so a scheduled run is audited identically to a manual one. Scheduling never widens what can run — it only automates the existing allowlist under the same gates and confirmation semantics (a scheduled dangerous task is created from an already-confirmed schedule).

Consequences​

  • The documented "no free-form shell" rule is narrowed, not deleted: the default remains no shell; expert mode is an explicit, off-by-default, audited exception.
  • admin-control-plane.md §Commander is updated to describe expert mode and scheduling and to keep the "safe by default" framing.
  • Anyone reviewing security must treat expert_mode_enabled as a high-value setting and the operator secret as guarding host command execution when it is on.
