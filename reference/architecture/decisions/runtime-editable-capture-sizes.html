ADR-0017: Runtime-editable capture sizes, GIF default cap, and ?variant=original

  • Status: Accepted
  • Date: 2026-06-16
  • Amends: ADR-0015, ADR-0016
  • Related: ADR-0004, ADR-0007, ADR-0014

Context​

ADR-0015 introduced a per-format capture-size contract and ADR-0016 set the defaults to "keep native" (0 = no cap). Both left three things open, captured as KI-015 follow-ups:

  1. Capture sizes were startup-frozen, modelled as env config. They were Settings (env) fields read once at process start and frozen into the render adapter. In practice nobody set PIXELBOOTH_CAPTURE_* — there is no .env for them — so the values were just Python field defaults wearing an env override that went unused. Worse, env is reserved for process/runtime bootstrap (admin-control-plane.md §Settings And Templates: data dir, environment, host/port, fake-hardware) — render output sizes are appliance settings, so they belonged in the SQLite settings the admin control plane edits live (ADR-0004), not env.
  2. GIF shipped at native by default. ADR-0016 acknowledged native GIF is "very large/slow" but still defaulted GIF to 0 (native), pushing the cap onto operators who would only discover the problem after a slow, oversized first GIF.
  3. ORIGINAL was archived but not served. ADR-0016 added original_path and the ORIGINAL/ folder but exposed no API to download the raw capture.

Decision​

Capture sizes are runtime, admin-editable settings (SQLite authoritative)​

The per-format output boxes move out of env entirely and into the typed capture settings section (CaptureSettings) that ConfigService owns and the admin control plane edits, joining the existing capture fields. Six axes are added, each a pixel maximum where 0 = "keep native": still_max_width/height, video_max_width/height, gif_max_width/height.

  • One source of defaults. The shipped defaults are the CaptureSettings model field defaults — nothing else. On first run ConfigService.ensure_seeded() writes the AppSettings() document, which already includes them; SQLite is authoritative thereafter. No env vars, no separate seed bridge, no duplicated default.
  • The render reads live. LocalRenderAdapter resolves sizes through a capture_sizes_provider at render time (mirroring the existing event_id_provider), so an operator edit applies to the next render with no restart. The field→box mapping lives in one place — CaptureSettings.size_axes() produces the axes, CaptureSizes.from_caps applies the 0 → unbounded rule.
  • The admin Settings panel renders the new numeric fields automatically (it reflects the typed section), so no bespoke UI is required; values are validated (ge=0) like any section.
  • Operators who want to ship file-based defaults (e.g. a venue profile) use the existing settings templates / JSON import-export (/admin/config/export, /import, save_template) — the project's validated config-as-data surface — rather than a new config file competing with SQLite.

GIF defaults to a 720px box​

The shipped default for gif_max_width/height is 720 (the CaptureSettings model default). still/video keep 0 (native) per ADR-0016. Native GIF is palettized, very large, and slow to encode on the Pi; 720px is a sensible sharing-quality default that operators can raise or lower.

?variant=original is served​

GET /media/{id}/file?variant=original now serves the archived raw capture (original_path). Because a composed item's content_type becomes the final's type (e.g. video/mp4) while the original is always a still, the original response derives its MIME from the stored file extension rather than the record's composed content_type.

Consequences​

  • Operators tune output sizes (including the GIF cap) from the admin UI during an event; changes take effect immediately on the next capture's render.
  • One source of truth for the defaults (the model), so no env/SQLite drift and no duplicated GIF=720. Existing installs pick up the new fields at their model defaults on read.
  • The PIXELBOOTH_CAPTURE_* env knobs are removed; env stays limited to bootstrap, matching the documented posture. (ADR-0015/0016 mention those vars; they are superseded here.)
  • Photographers/operators can download the untouched original via the API.

Alternatives considered​

  • Keep capture sizes as env / startup config. Rejected — they are appliance settings an operator wants to change per venue/event; env is bootstrap-only, and there is no .env for them, so they were really just frozen code defaults.
  • A separate config.json / config.toml for defaults. Rejected — it would be a third config surface beside env and the SQLite settings document. The project already has a validated config-as-data layer (AppSettings in SQLite + JSON templates/import-export); the right home for defaults is the Pydantic model, and for shippable file defaults, a settings template.
  • Env overrides SQLite per-field at runtime. Rejected — surprising precedence; the admin UI is the source of truth once the appliance is running.
  • Leave GIF at native and rely on the operator to cap it. Rejected — the failure mode (huge, slow GIFs) only shows up after the fact; a safe default is better.
