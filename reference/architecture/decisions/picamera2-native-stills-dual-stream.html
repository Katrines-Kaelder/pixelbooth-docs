ADR-0022: Picamera2 native-resolution stills via a lores-preview / native-main dual stream, with capture resolution as a runtime setting

  • Status: Accepted (preview path amended by ADR-0024)
  • Date: 2026-06-21
  • Amends: ADR-0009
  • Amended by: ADR-0024 — the single continuous still config is replaced by a light video preview config + switch_mode for capture; the native-ORIGINAL contract and runtime capture-size setting below are unchanged.
  • Related: ADR-0015, ADR-0016, ADR-0017

Context​

ADR-0016 set the captured ORIGINAL to "keep native" (no downscale) and ADR-0017 made the per-format output sizes runtime, admin-editable settings that downscale after capture. Both assumed the camera hands the render pipeline a full-resolution original.

The Picamera2 (CSI) adapter (ADR-0009) did not. It opened one shared video configuration with main sized to the preview budget (1280x720) and used that same stream for both live preview and still capture. Hardware validation on the Camera Module 3 therefore captured stills at 1280x720 instead of the native 4608x2592, contradicting the native/no-downscale ORIGINAL contract (KI-022). Because every output and overlay is derived from that original, the low capture resolution also degraded video quality (KI-008) and overlay sizing (KI-014).

Two things had to change: the still must be captured at the sensor's native resolution, and the capture resolution must be controllable (an operator may want to cap it for thermal or storage headroom) and persisted like every other appliance setting.

Decision​

A single still configuration with two streams (lores preview, native main)​

Each Picamera2 instance is configured with create_still_configuration exposing two streams:

  • a main stream carrying the full-resolution ORIGINAL — the configured capture size, or the native full sensor size when unset; and
  • a small lores stream sized to the preview budget (1280x720).

Stills are read from main (capture_file(..., name="main", format="jpeg")) so the ORIGINAL is native; preview frames are read from lores (name="lores") so live view stays cheap. A buffer_count of 3 keeps the lores preview smooth while the same camera serves native stills on demand. The per-role lock (ADR-0009) is unchanged, so preview and capture on one device still serialize. This mirrors the project's reference photobooth's lores/highres technique (inspiration only).

Capture resolution is a runtime, admin-editable setting​

CaptureSettings gains capture_max_width / capture_max_height (0 = native full sensor), joining the runtime capture section that ConfigService owns and the admin control plane edits live (ADR-0017). The Picamera2 adapter reads them through a capture_size_provider (mirroring the render's capture_sizes_provider), so an operator edit applies to the next opened camera with no restart. The admin Settings panel renders the new numeric fields automatically. This is the sensor capture size and is distinct from, and upstream of, the per-format output boxes (*_max_*) from ADR-0017, which still downscale the rendered outputs on top.

Consequences​

  • The captured ORIGINAL is full sensor resolution by default, restoring the ADR-0016/0017 contract; downstream output sizes and overlays scale from a true native original.
  • Operators can cap capture resolution from the admin UI for thermal/storage reasons without touching code or env.
  • Running the sensor at full resolution continuously to also serve preview, and lores YUV420→JPEG behaviour, were verified on the target Pi while closing KI-022. If continuous full-res preview proves costly, the follow-up is the reference app's switch_mode pre-switch during the countdown (preview on a light video config, switch to the still config only for the capture) — a thermal optimisation that does not change this contract.
  • Burst capture (collage/GIF/video) still issues sequential native stills; per-format burst timing tuning remains a hardware follow-up (KI-008/009).

Alternatives considered​

  • Keep a single shared config, just raise its size to native. Rejected: a native-sized preview stream is wasteful and slow for live view, and conflates two concerns (cheap preview vs. quality still) the dual stream separates cleanly.
  • Two configurations with switch_mode_and_capture per shot (a light video preview config
    • a separate still config, switched only for the capture). This is the lower-thermal design and the reference app's choice, but it stutters preview at each capture and adds switch latency on every still. Deferred as the thermal-optimisation follow-up rather than the baseline fix; the dual-stream config is simpler and keeps preview continuous.
  • Make capture resolution an env/bootstrap setting. Rejected for the same reasons as ADR-0017: appliance output/quality knobs belong in the SQLite settings the admin edits live, not in process bootstrap env.
