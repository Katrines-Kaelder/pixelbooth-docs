ADR-0003: Capture session orchestration is an explicit stage and service

  • Status: Accepted
  • Date: 2026-06-11
  • Deciders: Jonas Landsfeldt, Claude

Context​

The system architecture centers the runtime on SessionService — it "owns the active capture session and state transitions" — and the API contract defines the full /sessions surface plus a ten-state session state machine (idle, attract, countdown, capture, record, review, processing, present, complete, error) and the session.created/state/completed/error WebSocket events. Only one session may be active; concurrent starts return 409.

Despite being the connective tissue between camera capture (Stage 3) and media processing (Stage 4), session orchestration appeared in no roadmap stage. The Stage 3 kiosk UI was written to consume session.* events, but nothing was scheduled to emit them, own the state machine, or enforce the single-session lock. This is a gap, not a deliberate omission.

Decision​

Schedule capture session orchestration as an explicit stage, Stage 3.5: Capture Session Orchestration, sequenced between the camera spike (Stage 3) and media processing (Stage 4). SessionService is the single owner of:

  • the session state machine and all allowed transitions;
  • the single-active-session lock and the 409 conflict behavior;
  • emission of session.created, session.state, session.completed, and session.error events over the WebSocket contract;
  • coordination calls out to CameraService, ExposureService, LightingService, MarqueeService, and the render/queue boundary, without those services owning session state.

Stage 3 is scoped to camera hardware, degraded-mode behavior, preview/role selection, the capture trigger, and the exposure spike, plus a preview-only kiosk surface that does not require an active session. The full session-state kiosk experience (countdown → capture → review → processing → present feedback) and its E2E coverage move into Stage 3.5, where the events they depend on exist.

Consequences​

  • Easier: the camera→session→media seam has one clear owner; the kiosk UI is built against a real event source instead of a stub; the single-session invariant is enforced in one place.
  • Constrained: Stage 3 no longer "completes" the guest capture loop on its own; guest capture flow is only end-to-end after Stage 3.5.
  • Follow-up: the render/queue handoff (processing → present) depends on the shared job/queue primitive established in Stage 4; Stage 3.5 emits the processing transition and consumes render-progress events from that primitive.

Alternatives considered​

  • Fold orchestration silently into Stage 3 or Stage 4. Rejected: it is the largest single behavior in the booth and changing it touches the documented state machine, so it deserves explicit scheduling and an ADR per the ADR triggers.
  • Let each service own its slice of session state. Rejected: distributes the state machine, makes the single-session lock ambiguous, and breaks the system-architecture boundary that names SessionService as the owner.
