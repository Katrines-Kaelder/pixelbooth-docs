Expanded Capabilities Architecture

This document captures feature and architecture decisions that are intentionally broader than the first implementation. The guiding principle is to build Pixelbooth around capabilities, adapters, queues, and profiles so features can be added, replaced, or disabled without tangling capture flow, UI, and hardware code.

Service-Oriented Appliance​

Pixelbooth may start as one deployable app, but the boundaries must support splitting into services later.

  • ControlApi: REST, WebSocket, auth, admin UI API, session orchestration.
  • CameraService: USB/V4L2 (UVC) and Raspberry Pi Camera Module 3. USB came first in the Stage 3 spike (ADR-0008); the Picamera2/CSI dual-camera adapter follows on bare-metal Pi OS (ADR-0009).
  • RenderWorkerPool: overlays, collages, GIFs, boomerangs, video renders, chromakey, AI filters.
  • MediaService: SQLite metadata and file storage.
  • ConfigService: active settings, templates, revisions, import/export, validation, and secret handling.
  • AssetService: safe file browser roots, upload/download, asset metadata, and dependency checks.
  • OverlayService: PNG overlays, overlay preview, preset/event assignments, and future overlay designer definitions.
  • SyncService: local/USB/rclone/Immich/rsync-SFTP remotes and queue-backed upload/export jobs.
  • NetworkService: Wi-Fi scan/connect and fallback access point mode.
  • LightingService: decorative WLED/custom ESP32 lighting.
  • ExposureService: camera exposure, ring-light brightness, white temperature, and pre-capture metering.
  • MarqueeService: designs, preview, triggers, and push/status for the Pi Zero 2 W marquee agent.
  • PrintService: CUPS/IPP printer jobs, preview/crop handling, and future payment gating.
  • CommandService: allowlisted admin task runner for service/system commands, diagnostics bundles, queue retries, and hardware tests.
  • ObservabilityService: structured logs, diagnostic ids, and local diagnostics (local-first; no Sentry — see ADR-0006).

The same source tree should support:

  • Single-process development with fake hardware.
  • Raspberry Pi deployment with local workers and privileged hardware helpers.
  • Docker Compose deployment with separate API, render, sync, network, and marquee/agent-facing services where useful.
  • A future installer that configures services, permissions, boot mode, and kiosk startup.

Containerized deployment is the preferred operational model where Raspberry Pi hardware permissions allow it. Some privileged capabilities, such as Wi-Fi management, camera access, GPIO, printer access, and USB mount/sync, may require host permissions, device mounts, or small host-side helper services. The architecture should keep those privilege boundaries explicit instead of hiding them inside the UI.

Kiosk UI Constraints​

The primary physical screen is a 1280x800 touchscreen. This is the main design target for guest-facing booth experiences.

Hard UI rules:

  • Guest-facing capture screens must not scroll.
  • The gallery is the only guest-facing screen allowed to have scroll bars.
  • Countdown, capture, review, result, QR/share, print confirmation, challenge prompts, onboarding steps, and user-safe error/recovery screens must fit in one 1280x800 viewport.
  • Admin/operator screens may scroll.
  • The UI should still be responsive, but responsiveness should preserve the kiosk contract instead of simply shrinking a desktop layout.

Implementation implications:

  • Use stable viewport-sized scene layouts for kiosk flows.
  • Reserve scroll containers only for gallery and admin surfaces.
  • Use container queries or equivalent responsive constraints for preview, action buttons, and status/marquee areas.
  • Validate kiosk screens at 1280x800 during UI review.

Performance Model​

Capture must feel immediate. Heavy render work must not block camera control or kiosk feedback.

  1. Capture original as quickly as possible.
  2. Immediately show an instant preview.
  3. Queue full-resolution render work in RenderWorkerPool.
  4. Emit render progress over WebSocket.
  5. Replace preview with final/share/print-ready output when available.

RPi 5 multicore CPU should be used through multiprocessing or external accelerated tools, not CPU-bound Python threads. GIF, boomerang, and video work should prefer ffmpeg/GStreamer-style pipelines where possible.

Outputs should be separate:

  • original capture
  • instant preview
  • share/mobile render
  • print-ready render
  • archive render

Admin Control Plane​

Admin is a first-class appliance surface inside the same frontend as kiosk and gallery. It should be designed around operator jobs, not backend modules alone.

Initial admin surfaces:

  • dashboard: readiness, system health, queue health, recent captures, warnings, and safe quick actions
  • settings: typed settings grouped into understandable domains
  • assets: upload and file browser for safe roots
  • overlays: PNG overlay library and preview
  • media: operator gallery and recycle/destructive controls
  • remotes: sync/upload remote setup, queue state, and tests
  • Commander: allowlisted tasks with confirmation and live status
  • logs: structured logs and diagnostic ids
  • setup: first-run/rental/event profile flow

The dashboard should be the first admin view and should answer whether the booth is ready for opening night. Deep diagnostics still exist, but they should not be the only admin landing experience.

Settings, Templates, And Assets​

All settings changed in admin must persist. Use SQLite for active settings, templates, revisions, queue state, media metadata, remotes, task history, and asset records. Use Pydantic models for validation and JSON schema where schema-assisted forms are useful.

Use JSON import/export for:

  • event templates
  • rental/venue profiles
  • settings backup/restore
  • overlay designer definitions
  • marquee design definitions

Use safe asset roots under data/assets/* for operator-uploaded files. The UI should offer file picking from known asset kinds rather than asking the operator to type arbitrary paths.

Capture Modes​

Capture modes are registered capabilities, not hardcoded branching.

  • single_picture
  • collage
  • collage_wall_canvas
  • boomerang
  • gif
  • video
  • video_with_audio
  • future wigglegram

Each mode defines:

  • required camera roles
  • default camera role
  • user-selectable camera roles before capture
  • camera fallback policy: required, prefer, any, or dual
  • capture sequence
  • render pipeline
  • UI flow
  • output types
  • share/print compatibility
  • lighting, exposure, and marquee events

Camera selection is part of capture mode capability, not a one-off UI shortcut. Before capture, the guest may switch the live preview between available compatible roles. The selected preview role becomes the session camera role for any and prefer presets. Presets using required or dual can constrain or override the selector, but this must be visible before the guest presses capture.

Overlays​

Overlays are render assets and live-preview affordances. They are not just one-off file paths in capture settings.

V1 should support:

  • transparent PNG overlays at output image size
  • global, preset, event/template, and capture-mode assignments
  • admin overlay preview against the live camera stream or a still frame
  • kiosk preview overlay during idle/countdown when enabled
  • final-output composition in the render pipeline
  • validation for dimensions, alpha channel, file type, and output compatibility

Future overlay designer:

  • live preview stream or still frame as background
  • drag-and-drop layers for image, text, logos, QR placeholders, safe areas, and simple shapes
  • canvas set to output dimensions with responsive preview scaling
  • layer ordering, snap guides, opacity, rotation, lock/hide, and validation
  • exportable .pixelbooth-overlay.json plus generated preview/render assets

Design V1 so a PNG overlay is represented as an overlay definition, not merely as a string path. That keeps the future designer compatible with the same assignment and render pipeline.

Exposure And Ring-Light Control​

The booth will run in changing conditions: normal room lighting, full club mode, dancefloor lights, and mixed event lighting. Exposure control must therefore be a closed-loop capture capability, not just "autoexposure on/off".

Hardware Plan​

  • Main cameras: Raspberry Pi Camera Module 3.
  • Ring light: high-CRI dual-white LED strips, with warm and cold white channels, likely 2700K to 6500K.
  • Optional future sensor: ambient light/color sensor to help estimate room brightness and color temperature.

Services​

  • ExposureService: owns exposure profiles, metering decisions, and capture-time recommendations.
  • LightMeterService: provides scene brightness, highlight clipping, contrast, and estimated color temperature.
  • RingLightController: controls brightness and warm/cold mix independently from decorative RGB lighting.

Profiles​

Initial exposure profiles:

  • normal_lights: lower ring fill, normal autoexposure, mild center/face weighting.
  • club_mode: stronger ring fill, capped shutter time to avoid motion blur, higher gain allowed.
  • manual_event_locked: superadmin-calibrated settings saved for a venue/event.
  • calibrated_auto: uses pre-capture metering and bounded adjustments.

Pre-Capture Metering​

During countdown, Pixelbooth should sample preview frames and camera metadata before final capture.

The metering step can:

  • inspect preview histogram/brightness
  • estimate highlight clipping
  • optionally test or ramp the ring light
  • choose EV bias, exposure time, gain ceiling, and ring-light brightness
  • choose warm/cold LED mix or white balance mode
  • lock settings before final capture

Important default: in dark party mode, prefer more ring-light brightness and moderate gain before allowing shutter time to get long enough to blur moving people.

Marquee Display​

The marquee is a remote display target driven by a Raspberry Pi Zero 2 W. The main app owns design, storage, preview, trigger mapping, and push/status. The Pi Zero 2 W owns HUB75 panel driving, local render timing, GIF/video decoding later, brightness enforcement, and fail-safe display behavior.

Physical Resolution​

The planned hardware is two P2.5 LED matrix panels side by side:

  • one panel: 64x32
  • two panels: 128x32 physical pixels

Grouping is optional and must not be hardcoded into assets.

Render modes:

  • native: logical pixels map directly to physical pixels.
  • hard-2x2: logical pixels map to 2x2 blocks, yielding a 64x16 logical canvas on 128x32 physical hardware.
  • soft-2x2: text/sprites snap to the 2x2 pixel-art grid, while gradients/transitions may use native physical pixels.

Main App Responsibilities​

  • Store marquee designs as JSON in SQLite and exportable files.
  • Validate canvas and effect support.
  • Preview designs in the admin UI.
  • Assign designs to booth events.
  • Push designs/programs to the Pi Zero 2 W marquee agent.
  • Track agent connection, firmware, canvas size, brightness, and status.

Agent Responsibilities​

  • Drive the physical panels.
  • Render programs continuously.
  • Keep showing the last safe design if the main app restarts.
  • Support GIF and ideally video later.
  • Enforce brightness limits and panel refresh timing.

Marquee Design Format​

Saved files should use JSON, for example .pixelbooth-marquee.json.

{
  "schema_version": 1,
  "id": "player-ready",
  "name": "Player Ready",
  "canvas": {
    "physical_width": 128,
    "physical_height": 32,
    "logical_width": 64,
    "logical_height": 16,
    "grouping": "soft-2x2"
  },
  "duration_ms": 6000,
  "brightness": 120,
  "background": {
    "type": "solid",
    "color": "#05030a"
  },
  "layers": [
    {
      "id": "text-main",
      "type": "text",
      "text": "PLAYER 1 READY!",
      "font": "pixel-7",
      "x": 0,
      "y": 3,
      "color": "#ff2aa3",
      "effect": {
        "type": "scroll",
        "direction": "left",
        "speed": 28
      }
    }
  ],
  "triggers": ["idle", "attract"]
}

Initial layer types:

  • text
  • sprite
  • bitmap
  • gradient
  • timer

Future layer types:

  • gif
  • video
  • camera_preview
  • scoreboard
  • audio_reactive

Initial effects:

  • static
  • scroll
  • bounce
  • blink
  • breathe
  • countdown
  • wipe
  • flash

Sync And Uploads​

Uploads are remote-driven and queue-backed. Operators can configure one or more remotes. Each remote has common settings plus type-specific settings.

Common remote settings:

  • name
  • type
  • enabled flag
  • output selection: originals, final renders, print-ready, share/mobile, metadata, or all
  • trigger policy: on capture complete, scheduled, manual, or on USB insert
  • retry policy, concurrency, and backoff
  • last success/failure, queue depth, and test status

Recommended remote adapters:

  • local_folder: simple export to a local path under an allowed root
  • usb: removable media export through a helper/privileged boundary
  • rclone: Google Drive and other cloud providers, and SFTP targets, through rclone
  • immich: Immich's native upload path (endpoint confirmed against current official docs at implementation time)
  • rsync_sftp: LAN/self-hosted server copies over SSH
  • future webhook: custom event/upload integrations

Sync transport engines are decided in ADR-0005: rclone is the default for cloud and SFTP (parallel transfers, bandwidth limits, checksums, retries, resumable uploads), rsync/SSH serves LAN/self-hosted copies, and Immich uses its native upload path so dedup/albums work. Because booth media is write-once, parallelism, integrity verification, resumability, and a durable retry queue matter more than delta transfer. The SQLite-backed outbox (built on the shared job/queue primitive) makes sync jobs survive restarts, and remote tests are explicit admin actions.

Commander​

Commander is the admin task runner for system and operational commands. It should provide the power of the reference app's system-control buttons while avoiding arbitrary shell execution from UI settings.

Tasks must be allowlisted and typed:

  • id, label, category, risk level, input schema, and confirmation requirement
  • local-only/admin-only constraints where needed
  • executor adapter with timeout and safe output capture
  • run history with status, timestamps, diagnostic id, and output summary

Initial commands:

  • reload services
  • restart Pixelbooth app
  • reboot host
  • shutdown host
  • camera test capture
  • lighting test preset
  • sync remote test
  • retry failed sync/print/render queues
  • create diagnostic bundle
  • export settings and media bundle

Wi-Fi And Rental Onboarding​

Pixelbooth needs a UI flow for Wi-Fi because private-event borrowers should not need terminal access.

  • NetworkService handles Wi-Fi scan, connect, disconnect, saved networks, and fallback AP mode.
  • The UI talks to the API only; privileged network changes live in a dedicated helper/service.
  • Fallback AP mode is required so the booth is recoverable if it cannot join a venue network.

Rental mode is a guided event setup flow:

  1. Wi-Fi.
  2. Event name and media naming.
  3. Overlay/theme pack.
  4. Enabled modes.
  5. Upload remotes.
  6. QR/share settings.
  7. Printer test.
  8. Camera, exposure, lighting, and marquee test.

Superadmins can reset internal media, archive/export first, lock down advanced settings, and switch the booth into rental onboarding mode.

QR Sharing​

QR sharing is first-class, but the guest's phone cannot reach the Pi at the venue (payment-terminal Wi-Fi is off-limits, venue Wi-Fi is unusable, and the Pi is unreliable joining or hosting Wi-Fi). So the booth uploads each finished render to a remote host and the QR encodes a public URL the guest opens over their own mobile data. Booth connectivity and guest connectivity are fully decoupled. See ADR-0018.

Sequencing — share/sync services first, QR on top. The upload engine, durable outbox, and admin remote config are the real work (they also give off-site backup/export before any QR exists) and are QR's hard dependency. QR is the thin layer: an image of a URL, a pending→ready state, and a static sharepage.

Upload + link model:

  • Upload-on-render via rclone (ADR-0005): when a render reaches final, a share-upload job is enqueued for the designated share remote on the durable outbox — never on the capture path (fast-capture-path). First reference transport is S3-compatible (MinIO, self-hosted); the same model carries self-hosted WebDAV/FTP/SFTP and Google Drive.
  • Configurable public-link template (link_mode: sharepage | direct) with placeholders {key}, {event}, {filename} resolved against a deterministic, unguessable object key ({event_slug}/{stem}-{token}.{ext}) so URLs are non-enumerable yet computable without a round-trip.
  • Themed static sharepage is the QR target — an arcade grab page that fetches the bytes and shows the media with a big SAVE button and event branding, not a raw file.

Hard constraint — the host stays a dumb static/file server: NO server-side script (no PHP/dl.php, no Node/Python on the host). Sharing is static sharepage + uploaded bytes only; upload is rclone; any setup helper is a local operator script, not host-side code.

Robustness and policy:

  • A QR becomes scannable only once the upload is verified live (per-media share status pending → ready, surfaced over the media.shared event); the kiosk shows "SAVING…" until ready and a friendly offline state when no remote is configured.
  • Durable retry outbox drains on reconnect; uploads survive restarts/offline periods.
  • Unguessable keys + host-side lifecycle/expiry for privacy.
  • Per-media QR and event gallery QR; arcade QR framing (QR modules stay black-on-white for scan reliability — the arcade feel comes from the frame, caption, and event name).
  • Download size policy: original, final, or mobile/share render.

Printing And Payment​

Printer support should be adapter-based, with CUPS/IPP as the Linux/Pi-first backend and Canon Selphy 1500 as an important target.

Print flow:

  1. Select media.
  2. Choose print format.
  3. Show crop/safe-area preview.
  4. Confirm.
  5. Queue print job.
  6. Track printer/job status.

Future payment gateway support should gate release of the print job, not be baked into media or print rendering.

AI, Background Removal, And Filters​

Image effects are render pipeline stages:

  • ChromakeyStage
  • BackgroundRemovalStage
  • FaceLandmarkStage
  • StickerOverlayStage
  • PoseChallengeStage
  • ColorFilterStage

These stages are optional, worker-based, and must never block the capture experience. Snapchat-like filters should build on face landmarks and layered render stages.

Observability​

Pixelbooth needs both user-safe feedback and developer diagnostics.

  • Structured JSON logs (persisted and rotated locally).
  • Local admin log viewer.
  • Health/status events.
  • Diagnostic ids that connect UI errors to logs.
  • Safe diagnostic bundle export (secrets redacted).

Observability is local-first; there is no Sentry. The logging and diagnostic-id foundation ships in Stage 2.5 so diagnostic ids referenced from Stage 4 onward have a backing store; the admin viewer and bundle come in Stage 10. See ADR-0006.

Every hardware/service operation should return:

  • user-safe message
  • retry/recovery suggestion where possible
  • developer diagnostic id
  • structured underlying error details in logs
