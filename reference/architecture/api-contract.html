API Contract

Base path: /api/v1.

REST​

Sessions​

  • POST /sessions
    • Starts a session.
    • Body: { "preset_id": "selfie-normal", "camera_role": "normal" }
    • camera_role is usually the current live preview role selected by the guest before pressing capture.
    • The booth ModeSelect maps a chosen mode to a built-in preset: Photo → selfie-normal / selfie-wide (by role), Collage → collage-4, GIF → gif-loop (Boomerang is folded under the GIF tile → boomerang), Video → video-clip.
    • Returns 201 with a Session (carries capture_mode, shot_index, shot_count).
    • Returns 409 when another session is active.
  • GET /sessions/active
    • Returns the active session or null.
  • POST /sessions/{id}/confirm
    • Confirms the review step.
  • POST /sessions/{id}/reject
    • Rejects current capture and retries or ends according to preset.
  • POST /sessions/{id}/abort
    • Aborts the active session.

Cameras​

  • GET /cameras
    • Returns configured camera roles and status.
  • GET /cameras/{id}/stream.mjpg
    • Returns a multipart MJPEG stream.
  • GET /cameras/active-preview
    • Returns the kiosk's active preview role and stream status.
  • PUT /cameras/active-preview
    • Body: { "camera_role": "wide" }
    • Switches the guest-facing live preview role without starting a capture session.
  • POST /cameras/{id}/capture-test
    • Captures a test image and catalogs it as test media.

Media​

Guest-readable. Booth and gallery are open surfaces (see system-architecture.md §Access Control), so media listing and byte serving are not behind the admin gate. Files are authoritative for bytes; the SQLite media catalog is authoritative for metadata (system-architecture.md §Storage).

  • GET /media
    • Lists newest media first. Supports ?limit= and ?offset= and excludes hidden items by default. Returns { "items": [Media], "total": <int> }; total backs the booth "PICS TAKEN" counter.
  • GET /media/{id}
    • Gets a single media record (metadata, not bytes). A record carries overlay_id and overlay_status (none | applied | skipped) so admin and gallery surfaces can show whether an assigned overlay was composited into the final render or skipped (and a developer diagnostic id is logged on skip).
    • A record also carries the composition fields compose_format (still | collage | animation | boomerang | video, or null for a plain still), compose_status (none | applied | skipped), and frame_count (the number of captured source frames). Composed formats capture a short frame sequence; the background render composes the frames into the final output. Like overlays, composition is best-effort: when no image/video backend is installed (OpenCV is the optional usb extra; Pillow is optional too) the format is recorded skipped and the final falls back to the first captured frame — a valid, servable output. See ADR-0012.
  • GET /media/{id}/file
    • Serves the media bytes for a variant: ?variant=final (default), preview (instant-preview frame), or thumb. Returns 404 when the variant is absent. A capture's instant preview is available here before the full-resolution final render completes (fast-capture-path).
  • DELETE /media/{id}
    • Deletes or recycles a media item.
  • POST /media/{id}/share
    • Starts configured share/print action. (Stage 8.)

Render and other deferrable work runs on the shared job/queue primitive (system-architecture.md §Shared job/queue primitive, ADR-0011) and reports progress over the WebSocket job.* events; capture acknowledgement is never blocked by it.

Assets​

Admin-managed files live behind safe asset/media/export roots rather than arbitrary filesystem paths.

  • GET /assets
    • Lists asset records, filterable by kind such as overlay, background, font, branding, marquee, export, or backup.
  • POST /assets
    • Uploads one or more files to an allowed asset root.
  • GET /assets/{id}
    • Gets asset metadata.
  • GET /assets/{id}/file
    • Downloads or previews the asset file.
  • DELETE /assets/{id}
    • Deletes or recycles an asset if no active setting depends on it.
  • GET /files/browse
    • Lists allowed roots and folders for the admin file browser.
    • Query: ?root=assets.overlays&path=/event-a
  • POST /files/folders
    • Creates a folder in an allowed root.
  • POST /files/zip
    • Creates a zip download for selected files under allowed roots.

Overlays​

  • GET /overlays
    • Lists overlay definitions and PNG overlay assets.
  • POST /overlays
    • Creates an overlay from an uploaded transparent PNG or future designer definition.
  • GET /overlays/{id}
    • Gets overlay metadata and render settings.
  • PUT /overlays/{id}
    • Updates overlay metadata, assignment, or definition.
  • DELETE /overlays/{id}
    • Deletes or archives an overlay.
  • POST /overlays/{id}/preview
    • Renders a preview using a live camera frame, latest media frame, or uploaded sample.
  • GET /overlays/assignments
    • Returns global, preset, and event/template overlay assignments.
  • PUT /overlays/assignments
    • Updates overlay assignments.

Overlay composition: the render pipeline resolves the overlay for a capture by scope precedence preset > event > global (most specific wins) and composites a valid transparent PNG into the final output during background render — the instant preview stays overlay-free on the fast path. The event scope is the currently active event entity, resolved at render time, so per-event overlays follow whichever event is live. Single-image and collage finals are overlayable; animation/video finals are not (the overlay is recorded skipped as incompatible media). Composition is best-effort: an assigned overlay is recorded as skipped (with a logged diagnostic id) when the asset is missing, the PNG is invalid or has no alpha channel, the media type is incompatible, or no image backend is installed on the host (OpenCV is the optional usb extra). The render never fails on overlay problems — it keeps the un-overlaid bytes and records the outcome on the media record (overlay_status).

Lighting​

  • GET /lighting/devices
    • Lists configured/discovered lighting adapters.
  • GET /lighting/presets
    • Lists event-to-preset mappings.
  • POST /lighting/test
    • Body: { "preset": "countdown" }
    • Sends a test preset.

Exposure​

  • GET /exposure/profiles
    • Lists exposure/ring-light profiles such as normal_lights, club_mode, and manual_event_locked.
  • PUT /exposure/profiles/{id}
    • Updates a profile.
  • POST /exposure/meter
    • Runs a pre-capture metering pass and returns a recommended camera/ring-light decision.
  • POST /exposure/test
    • Applies a profile temporarily for camera and ring-light testing.

Marquee​

  • GET /marquee/status
    • Returns Pi Zero 2 W agent status, canvas size, firmware, brightness, and connection health.
  • GET /marquee/designs
    • Lists saved marquee designs.
  • POST /marquee/designs
    • Creates a saved JSON design.
  • PUT /marquee/designs/{id}
    • Updates a saved design.
  • DELETE /marquee/designs/{id}
    • Deletes a saved design.
  • POST /marquee/preview
    • Renders a browser/admin preview without pushing to hardware.
  • POST /marquee/push
    • Pushes a design/program to the Pi Zero 2 W agent.
  • POST /marquee/test
    • Triggers a design or event mapping for testing.

Sync​

  • GET /sync/remotes
    • Lists configured remotes and common status.
  • POST /sync/remotes
    • Creates a remote with common settings and type-specific settings.
  • GET /sync/remotes/{id}
  • PUT /sync/remotes/{id}
  • DELETE /sync/remotes/{id}
  • POST /sync/remotes/{id}/test
    • Runs a connection/config test without syncing all media.
  • POST /sync/remotes/{id}/sync
    • Manually queues sync for the selected scope.
  • GET /sync/jobs
    • Lists queued/running/completed/failed sync jobs.
  • POST /sync/jobs/{id}/retry

Remote types:

  • local_folder
  • usb
  • rclone
  • immich
  • rsync_sftp
  • future webhook

Network​

  • GET /network/wifi
  • POST /network/wifi/scan
  • POST /network/wifi/connect
  • POST /network/wifi/disconnect
  • POST /network/fallback-ap

Print​

  • GET /printers
  • POST /printers/{id}/test
  • POST /media/{id}/print-preview
  • POST /media/{id}/print

Config and Diagnostics​

  • GET /config
    • Returns guest-safe active config needed by booth/gallery UI.
  • GET /admin/config
    • Returns admin-visible active settings with secrets redacted unless explicitly allowed by role.
  • PUT /admin/config
    • Replaces active settings after typed validation.
  • PATCH /admin/config/{section}
    • Updates one settings section.
  • GET /admin/config/schema
    • Returns typed JSON schema for settings sections where schema-assisted UI is useful.
  • GET /admin/config/templates
    • Lists saved settings templates.
  • POST /admin/config/templates
    • Saves a named template from active settings or uploaded JSON.
  • POST /admin/config/templates/{id}/apply
    • Applies a template to active settings after validation and confirmation.
  • GET /admin/config/export
    • Exports active settings or a template as JSON.
  • POST /admin/config/import
    • Imports a settings JSON file as a template or active settings draft.
  • GET /diagnostics

Commander​

Commander exposes allowlisted operator tasks, not arbitrary shell commands.

  • GET /commands
    • Lists available command tasks with id, category, risk level, input schema, and confirmation requirement.
  • POST /commands/{id}/runs
    • Starts a command task.
  • GET /commands/runs
    • Lists recent task runs.
  • GET /commands/runs/{id}
    • Gets task status, timestamps, output summary, diagnostic id, and safe logs.
  • POST /commands/runs/{id}/cancel
    • Attempts to cancel a running task when the executor supports it.

WebSocket​

Endpoint: /api/v1/events.

Events are JSON objects:

{
  "type": "session.state",
  "payload": {
    "id": "session-id",
    "state": "countdown",
    "capture_mode": "collage",
    "shot_index": 2,
    "shot_count": 4,
    "result": {
      "instant_preview_url": null,
      "final_url": null,
      "final_content_type": null,
      "media_id": null
    }
  },
  "created_at": "2026-06-10T00:00:00Z"
}

The session.state payload carries the session result so the kiosk can show the instant preview at review and the rendered photo at present without a second fetch. The URLs point at guest-open media bytes (GET /media/{id}/file?variant=). result.final_content_type is the MIME type of the final output (e.g. image/jpeg, image/gif, video/mp4) so the kiosk can choose <video> vs <img> without a refetch — and correctly handles the off-hardware case where a video session's final is a skipped-compose still.

The payload also carries capture_mode and the posed-shot progress shot_index / shot_count (both 1 for non-posed modes). Per the per-format capture choreography (ADR-0013), collage loops countdown → capture once per shot (so shot_index advances and the kiosk shows "Shot k of N"), while animation / boomerang / video run a single countdown then the record state for a timed window. These fields ride on session.created and session.state alike (same Session shape).

Event types:

  • session.created
  • session.state
  • session.completed
  • session.error
  • camera.status
  • camera.preview.selected
  • lighting.status
  • exposure.metered
  • marquee.status
  • marquee.program
  • media.created
  • media.deleted
  • job.progress
  • job.completed
  • job.failed
  • asset.created
  • asset.deleted
  • overlay.preview
  • overlay.updated

  • sync.remote.status

Core Types​

  • CameraRole: normal, wide
  • CameraFallbackPolicy: required, prefer, any, dual
  • CaptureMode: still, collage, animation, boomerang, video
  • SessionState: idle, attract, countdown, capture, record, review, processing, present, complete, error
  • LightingPreset: idle, attract, countdown, capture, review, success, error, challenge
  • MediaType: image, collage, animation, video, test
  • MediaVariant: final, preview, thumb
  • OverlayStatus: none, applied, skipped
  • ComposeFormat: still, collage, animation, boomerang, video
  • ComposeStatus: none, applied, skipped
  • JobStatus: queued, running, completed, failed
  • ExposureProfile: normal_lights, club_mode, manual_event_locked, calibrated_auto
  • MarqueeGrouping: native, hard-2x2, soft-2x2
  • AssetKind: overlay, background, font, branding, marquee, media, export, backup
  • OverlaySourceType: png, designer_json
  • SyncRemoteType: local_folder, usb, rclone, immich, rsync_sftp, webhook
  • CommandRisk: safe, confirm, dangerous

The frontend consumes these types from the generated OpenAPI schema rather than hand-written mirrors: pnpm gen:api exports the FastAPI document (apps/web/src/api/generated/openapi.json) and runs openapi-typescript to produce schema.ts; apps/web/src/api/{types,admin/types}.ts re-export thin aliases onto it. WebSocket event payloads and the user-safe error envelope are not modelled in OpenAPI and stay hand-written. Regenerate after any request/response model change.
