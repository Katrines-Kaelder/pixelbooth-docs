Frontend And Development Standards

This document defines implementation standards for the React/TypeScript frontend and shared codebase. It supersedes any rough prototype/scaffold choices made before the architecture was complete.

Package Manager Decision​

Use pnpm as the package manager for the JavaScript/TypeScript workspace.

  • pnpm is the default and documented package manager for the monorepo.
  • Frontend scripts, CI, install docs, and lockfile expectations should use pnpm.
  • Do not add alternative package-manager instructions unless the project explicitly changes this decision.
  • Vite remains the frontend build/dev-server choice unless a later architecture review replaces it.

Why pnpm:

  • mature workspace and lockfile behavior
  • strong monorepo ergonomics
  • predictable CI/deployment setup
  • lower onboarding risk for future contributors

Package Version Policy​

Do not assume package versions from memory.

When adding or upgrading frontend packages:

  • check current stable versions from the package registry or official release notes at implementation time
  • prefer current stable releases over stale examples
  • avoid preview/RC versions unless explicitly chosen
  • commit lockfiles
  • document major-version choices when they affect architecture

Package/version checks should include at least:

  • React
  • Vite
  • TypeScript
  • pnpm
  • Tailwind CSS
  • clsx
  • tailwind-merge
  • class-variance-authority or chosen variant helper
  • TanStack Query
  • router/state libraries
  • test tooling

Styling Decision​

Do not build the production UI with loose global/default CSS.

Preferred styling stack:

  • Tailwind CSS as the default implementation layer.
  • Semantic CSS variables and project-level Tailwind theme tokens.
  • clsx, tailwind-merge, and a variant helper such as class-variance-authority for reusable component variants.
  • Plain CSS only for truly global reset/base rules and token definitions.

Why Tailwind:

  • Better fit for agentic AI coding and rapid UI iteration.
  • Easier for multiple agents to generate, inspect, and refactor consistently.
  • Strong ecosystem for variants, composition, and responsive/layout work.

Tailwind does not replace the design system. It is the utility engine used to consume semantic tokens. Do not add a second production styling system; Pixelbooth uses Tailwind for frontend styling.

Theme Tokens​

The system theme must use semantic tokens, not random component-local colors.

Examples:

  • color.background.kiosk
  • color.background.admin
  • color.text.primary
  • color.text.muted
  • color.action.primary
  • color.action.danger
  • color.status.success
  • color.status.warning
  • color.status.error
  • color.arcade.marqueeText
  • color.arcade.portalLight
  • space.panel
  • space.controlGap
  • size.touchTarget
  • radius.control
  • font.display
  • font.arcade
  • font.body
  • font.brand
  • shadow.portalGlow

Rules:

  • Components consume semantic tokens through Tailwind theme names, CSS variables, and reusable component variants.
  • Raw hex/oklch/color literals live in theme files only.
  • Avoid default palette utilities such as bg-pink-500 in production components unless they are wrapped in project tokens or prototypes clearly marked for replacement.
  • Avoid arbitrary values in component markup unless they represent measured layout constraints and have a short comment or extracted constant.
  • Hardware-related colors, such as marquee, ring-light preview, or RGB presets, live in device/profile config, not arbitrary component styles.
  • The admin UI and kiosk UI may use different token themes while sharing component primitives.
  • Typography follows the same token-theme split. The booth/gallery (kiosk theme) use the pixel fonts — font.display (Press Start 2P) for headings and font.arcade (VT323) for arcade labels. The admin theme (.theme-admin) prioritises fast overview and legibility, so it overrides font.display and font.arcade to the legible font.body system stack at the theme scope (no per-component changes). font.brand always resolves to the pixel display face and is not overridden by the admin theme; it is reserved for the small Pixelbooth wordmark accents (admin sidebar wordmark, admin login title) so the control plane keeps a touch of arcade identity while staying scannable.

Component Structure​

Prefer small reusable components over long files.

Rules:

  • Large screens are composed from smaller feature components.
  • Components should usually be under about 150 lines unless there is a clear reason.
  • Extract repeated layout, controls, status badges, previews, and editor primitives.
  • Keep route files thin: fetch data, compose sections, and delegate behavior.
  • Prefer composition and clear props over giant mode-switching components.

Suggested frontend shape:

apps/web/src/
  app/
  features/
    booth/
    gallery/
    admin/
    assets/
    overlays/
    marquee/
    exposure/
    sync/
    commander/
    network/
    print/
    settings/
    setup/
  components/
  theme/
  api/
  state/
  utils/
  test/

Utilities​

Use utility modules deliberately.

Good candidates:

  • date/time formatting
  • media path and URL helpers
  • API error normalization
  • color conversion
  • marquee canvas math
  • exposure histogram helpers
  • print crop/safe-area calculations
  • file-size and duration formatting

Rules:

  • Utilities should be pure where possible.
  • Utilities should have focused tests.
  • Avoid dumping unrelated helpers into a single catch-all file.

State Management​

Use the right state for the job:

  • TanStack Query for server/cache state.
  • A small client store such as Zustand for kiosk/session UI state, WebSocket event projection, and ephemeral editor state.
  • Local component state for simple form fields and transient UI.
  • URL state for shareable admin/gallery filters where useful.

Rules:

  • Do not duplicate server state into global client state without a reason.
  • WebSocket events update query caches or dedicated event projections.
  • Long-running jobs, render status, sync status, print status, marquee status, and exposure status are modeled as server state.
  • Forms use explicit validation schemas where the shape is non-trivial.

File And Code Style​

  • TypeScript strict mode is required.
  • Avoid any; prefer discriminated unions for modes, events, and capabilities.
  • Use generated API types from FastAPI/OpenAPI when the API stabilizes.
  • Keep domain types separate from UI component props.
  • Prefer explicit named exports.
  • Prefer immutable updates.
  • Handle loading, empty, success, and error states for every async surface.
  • Keep user-facing errors safe and pair them with developer diagnostic ids.

Review And Codebase Navigability​

Write modular, readable code because it makes the product easier to maintain, test, review, and extend. A useful side effect is that code-intelligence tools can inspect the system more accurately, but tool compatibility is not the reason for the coding style.

Architecture expectations:

  • clear module boundaries
  • small files with focused exports
  • stable names for services, capabilities, adapters, events, and route modules
  • avoid hidden global state
  • avoid giant components or service files that make symbol graphing noisy
  • add architecture docs close to the relevant feature area
  • keep dependency direction obvious

Before major implementation milestones, run a review pass, optionally assisted by code graph/navigation tools such as Serena, that checks:

  • module dependency graph
  • oversized files/components
  • unclear ownership boundaries
  • duplicated logic
  • missing tests around service contracts
  • unsafe hardware/network/print operations

Serena or a similar tool can be added to the development workflow once the repo structure stabilizes, but the acceptance criteria are human engineering qualities: clear ownership, small modules, low coupling, good tests, and obvious dependency direction.
