Camera And Lighting Architecture

Camera Roles​

Pixelbooth targets Raspberry Pi 5/CM5-class hardware with two camera connectors:

  • normal: default portrait/selfie and high-quality stills.
  • wide: group shots, bar context, and future pose/challenge modes.

Each camera adapter exposes:

  • status
  • preview frames as MJPEG bytes
  • still capture
  • video recording hooks
  • role-specific settings such as transform, exposure, focus, and resolution

V1 does not require hard synchronization between the two cameras. Presets choose one role per capture unless a future preset explicitly captures both.

Camera Backends And USB / V4L2 Cameras​

The camera adapter is selected by a camera_backend setting so the same CameraService and API serve any source:

  • fake (default off-Pi): deterministic placeholder frames; reaches every availability state.
  • usb: a single USB (UVC) camera over V4L2/OpenCV — see below.
  • picamera2: the dual CSI Camera Module 3 adapter on bare-metal Pi OS — see below (ADR-0009).

Picamera2 / CSI dual-camera role mapping. The Pi exposes up to two CSI cameras, so the adapter manages one Picamera2 instance per role via a role → camera_num map (picamera2_normal_index=0, picamera2_wide_index=1 by default; camera_num indexes Picamera2.global_camera_info()). A role is available only when its index is actually connected, so all four availability states — both, normal_only, wide_only, none — follow directly from the connected hardware, and this is the first backend that can drive the real both/dual preset modes. Enumeration is open-free (it does not grab a streaming handle), so status polling never collides with an active preview; each camera has its own lock so preview and still capture on the same device serialize. Each camera builds two configurations and idles in the cheap one (ADR-0024): a light create_video_configuration with a main (a full-FoV IMX708 sensor mode, default 2304x1296) plus a lores stream for live preview, and a create_still_configuration with the native (or runtime-capped) main for the full-resolution ORIGINAL. Idling in the video config keeps the sensor in a fast mode, so the lores preview is not pinned to the slow native-resolution readout (~14 fps on Camera Module 3). The preview main must select a full-field-of-view mode (the binned 2304x1296, ~56fps), not the cropped 1536x864 high-fps mode, or the preview looks zoomed-in versus the saved still (KI-003). These preview knobs + the image flip are runtime, admin-editable camera settings. A still takes the role lock, stops the encoder, switch_modes to the still config, captures main at the native full sensor resolution (4608x2592 by default), switches back, and restarts the encoder — so the captured ORIGINAL stays native while idle preview runs fast. The capture resolution is an admin-editable capture setting (capture_max_width/height, 0 = native full sensor) read when a camera opens, so it applies on the next open. Live preview is served by a continuous hardware MJPEGEncoder on the lores stream writing each JPEG into a latest-frame buffer; the stream yields the freshest frame so a slow client never drains a backlog (realtime feed, not per-frame capture_file). Stills still use capture_file(name="main"). Picamera2 is the system python3-picamera2 package (it is not pip-installable), so the API runs on bare-metal Pi OS with a --system-site-packages venv; it is lazy-imported, so fake/dev/CI paths never require it. A CSI-only booth needs no OpenCV. Rationale and trade-offs: ADR-0009, ADR-0022, ADR-0024.

USB / V4L2 single-device role mapping. A USB UVC webcam is one physical device, so it maps to one configurable role (usb_camera_role, default normal) addressing one /dev/video* device (usb_camera_device). This reuses the existing degraded-mode path unchanged: a single USB camera is simply a normal_only (or wide_only) booth. Exercising both/dual modes still requires the CSI modules or a second USB device. A UVC webcam allows only one open V4L2 handle at a time, so availability/discovery is checked open-free (device node presence) — the device is opened only for the live stream and still capture, never just to report status, so status polling can't collide with an active preview. The OpenCV imaging stack is an optional dependency (usb extra) and is lazy-imported, so fake/dev/CI paths never pull in the native wheel. Rationale and trade-offs: ADR-0008.

Lifecycle and restart recovery. Camera devices and streams must be released on app shutdown so a restart can reacquire them instead of finding the device locked (a real risk on the Pi, where a UVC webcam allows only one open V4L2 handle and CSI cameras hold a per-role lock). The CameraAdapter protocol therefore exposes an idempotent close(): usb/picamera2 release their capture handles, the fake adapter is a no-op. CameraService.close() never raises — if a device cannot be released (e.g. it is still locked), it logs a user-safe diagnostic with a diagnostic_id and returns it, so a stuck device is actionable rather than silent.

Reaching that release on the first interrupt needs help, because the booth keeps two endless responses open — the live preview MJPEG response and the /events WebSocket fan-out — and uvicorn's graceful shutdown waits for in-flight responses to finish. So a bare uvicorn launch hangs on the first Ctrl+C and only a forced second interrupt exits, skipping the lifespan and leaking the locked camera (uvicorn #1579). The Pi therefore runs the API via the python -m pixelbooth entrypoint, which (following the reference photobooth's proven workaround) sets server.force_exit = True so uvicorn abandons those endless responses on the first signal instead of waiting for them. force_exit also skips the FastAPI lifespan, so the entrypoint's handle_exit calls CameraService.close() synchronously in the signal handler, before delegating to uvicorn — releasing the cameras even though the lifespan no longer runs.

One more hazard is specific to the camera: a libcamera/V4L2 worker thread can outlive Picamera2.close(), and being non-daemon it blocks the interpreter's own shutdown forever — so the process prints its last log line and then hangs, never freeing the camera fds and defeating the very restart this is about. Once the server returns, the entrypoint therefore forces the process to exit (os._exit(0)); the cameras were already released in handle_exit, so this only skips joining the stuck thread. See ADR-0023.

Autofocus​

The CSI Camera Module 3 (IMX708) cameras have a focus motor, so the booth enables autofocus (ADR-0025). Focus is an admin-editable camera setting, read live by the Picamera2 adapter and applied after the camera starts (and re-applied in place on an admin edit, like the preview tuning):

  • af_mode — continuous (default), auto, or manual. Continuous tracks the guest as they move into frame (zero config). The lens is set to AfSpeed=Fast so it settles quickly.
  • af_range — normal (default), macro, or full.
  • lens_position — a fixed focus distance in dioptres (1/metres), used only under af_mode = manual. This is the manual-lock escape hatch: a busy club is a low-contrast scene where continuous AF can hunt (the lens visibly breathes searching for focus, and may be mid-hunt at capture), so an operator can pin focus at the booth's known standing distance and the motor then never moves — no hunting, no focus latency at capture. It is an operator setup choice, not an automatic trigger. (An auto + AF-trigger-at-countdown middle mode is a planned follow-up.)

Each control is applied defensively (a camera/role lacking it logs and continues), so a non-AF device degrades instead of failing to open.

Camera Availability And Degraded Mode​

The booth should remain usable if one camera is unavailable.

  • If both normal and wide cameras are available, all compatible modes may be offered.
  • If only normal is available, normal selfie/photo modes continue and wide-only/group modes are disabled or remapped by preset policy.
  • If only wide is available, wide/group modes continue and normal-only modes are disabled or remapped by preset policy.
  • If no real camera is available, the app enters hardware fault mode on the Pi, while development can still use fake cameras.
  • Camera status must be visible in admin and reflected on the kiosk only as a user-safe message.
  • Capture presets declare required camera roles and fallback behavior.
  • Rental/onboarding mode must test cameras and warn before the event starts.

Fallback behavior is explicit per preset:

  • required: disable the preset if the camera role is unavailable.
  • prefer: use the preferred camera, but fall back to another compatible role.
  • any: use whichever camera is available.
  • dual: requires both cameras and is disabled unless both are available.

Development Cameras​

Non-Pi development uses fake cameras that generate deterministic placeholder images. This keeps tests and frontend work usable on laptops.

Preview​

V1 uses MJPEG streams because they are simple, local-network friendly, and match the appliance deployment model. The preview provider remains abstract so it can later move to WebRTC or libcamera-native streaming if needed.

Each camera adapter exposes a stream_frames(role) generator; the API serves it as a multipart/x-mixed-replace MJPEG response at GET /cameras/{id}/stream.mjpg. The fake, USB, and CSI adapters serve this identically (placeholder vs. live frames). The CSI adapter produces frames from a continuous hardware MJPEGEncoder on a fast-mode lores stream (ADR-0024) rather than re-encoding per frame, to keep the local feed realtime. The documented budget is ~720p at ~30fps; measured fps, latency, CPU, and bandwidth are recorded on target hardware (KI-003).

Live Preview Camera Selection​

Guest-facing booth screens must let users switch the live preview camera before pressing the capture button. The common case is a simple normal/wide selector, but the selector must be driven by camera availability and the chosen capture preset.

Rules:

  • Switching preview camera is a pre-capture action and must not start a session.
  • The current preview role is visible in the UI, but presented in friendly language such as "Portrait" and "Wide".
  • Unavailable or incompatible camera roles are disabled.
  • If the selected preset supports any or prefer camera behavior, the session starts with the currently selected preview role.
  • If the selected preset has required behavior, the UI should switch to or lock the required role before capture.
  • If the selected preset has dual behavior, the UI should communicate that both cameras will be used.
  • Physical capture buttons use the current preset plus the current preview camera role.
  • Admin diagnostics must include a test that switching preview roles also changes the camera used for a real capture.

The preview selector must also work in degraded mode. If only one real camera is available, the selector collapses to that role and compatible modes still work.

Lighting Events​

Lighting is event-driven. Session state changes map to presets:

  • idle
  • attract
  • countdown
  • capture
  • review
  • success
  • error
  • challenge

Exposure And Ring Light​

Exposure control is separate from decorative lighting. The booth must handle both bright room lights and dark club mode with dancefloor lighting.

This area lands in two parts:

Sensor-side camera exposure (implemented — ADR-0025)​

The camera-only slice runs now: the sensor's own auto-exposure/auto-gain (AEC/AGC) and auto-white-balance produce a normal exposure from "what the sensor sees", with a small set of admin-editable camera controls applied live by the Picamera2 adapter (ADR-0025):

  • ae_enable (auto exposure on by default), ae_metering (centre/spot/matrix), ae_exposure_mode (normal/short/long — short favours faster shutter / higher gain to reduce motion blur of moving guests), and exposure_value (EV compensation).
  • an optional manual exposure_time_us + analogue_gain pair, used only when ae_enable is off.
  • awb_enable (on by default) and awb_mode (auto/tungsten/fluorescent/indoor/ daylight/cloudy).

Focus/exposure metadata (lens position, focus figure-of-merit, exposure time, analogue/digital gain, lux, colour temperature) is read from camera metadata and surfaced on the admin dashboard next to the measured preview fps, so an operator can verify AE/AF behaviour while tuning on hardware.

Coordinated ExposureService (deferred — KI-006, Stage 5)​

The full picture below — coordinating the camera controls with the ring light, ambient/colour sensors, and per-event lighting profiles — is the deferred ExposureService hardware spike (KI-006). It needs the real lights and sensors to design against; the sensor-side controls above are the forward-compatible inputs it will later coordinate.

The planned ring light uses high-CRI dual-white LED strips with warm and cold channels, likely covering about 2700K to 6500K. The app should control both total brightness and warm/cold mix.

ExposureService should coordinate:

  • camera exposure mode
  • exposure time
  • analogue gain
  • EV compensation
  • white balance mode or manual color temperature
  • ring-light brightness
  • warm/cold LED mix
  • optional ambient light/color sensor input

Pre-capture metering should run during countdown. It should sample preview frames and camera metadata, estimate brightness/highlight clipping, optionally ramp or test the ring light, then lock a camera and ring-light decision before capture.

Initial profiles:

  • normal_lights
  • club_mode
  • manual_event_locked
  • calibrated_auto

In club mode, the default should prefer stronger ring light and moderate gain before allowing shutter times long enough to blur people.

WLED Adapter​

The first real adapter targets WLED-compatible ESP32 devices. It should support host/IP setup, brightness, preset ids, and test commands.

Custom ESP32 Adapter​

The custom adapter contract is newline-delimited JSON over serial or TCP. Example command:

{"command":"preset","preset":"countdown","brightness":180}

Example response:

{"ok":true,"preset":"countdown","firmware":"pixelbooth-esp32"}

The API should not expose WLED-specific details outside the adapter-specific configuration.

Marquee Display​

The marquee is not part of the camera or lighting path, but it subscribes to the same session events. It is driven by a Raspberry Pi Zero 2 W running a future marquee agent.

Hardware plan:

  • two P2.5 LED matrix panels side by side
  • one panel is 64x32
  • combined physical canvas is 128x32
  • optional grouping can create a 64x16 logical canvas for a 2x2 pixel-art look

Grouping must be a render setting, not a destructive asset format:

  • native
  • hard-2x2
  • soft-2x2

The main app owns design storage, admin preview, trigger mapping, and push/status. The Pi Zero agent owns HUB75 output, local timing, panel refresh, brightness enforcement, and later GIF/video playback.
