Inspiration Photobooth App Analysis

What It Provides​

The inspiration app is a Python FastAPI service with a Vue/Quasar single-page frontend. It includes camera capture, live preview, still image capture, collages, GIF animations, video capture, media gallery, sharing/printing commands, QR sharing, GPIO button triggers, system diagnostics, admin configuration, file browsing/upload, confirmed system commands, and WLED lighting presets.

The strongest reusable ideas are the service split around acquisition, processing, media collection, sharing, GPIO, WLED, diagnostics, and server-sent events. Its capture workflow is also useful: countdown, capture or record, optional approval, final processing, presentation, and return to idle.

After adding the frontend source, the strongest reusable UI idea is the single-app shape: kiosk, gallery, standalone/slideshow views, auth, and admin live in one frontend. Admin is a real center with dashboard, config, files, multicam, logs, help, first-start setup, and confirmed system-control actions.

Main Workflows​

  • Guest opens the booth screen, sees the live preview, presses a capture action, waits through a countdown, reviews or confirms if configured, then sees the final media.
  • Admin opens the admin area, lands on a dashboard, configures camera backends, UI options, WLED, GPIO, sharing, and media processing.
  • Admin uploads/browses files and uses those files as assets for frames/backgrounds/configured outputs.
  • Gallery lists newest media first, supports detail view and deletion.
  • Share service executes configured shell commands for print/upload-style integrations.
  • QR share service talks to a separate PHP upload helper.
  • System status reports platform, camera backend stats, CPU, memory, temperature, and counters.

Useful Patterns​

  • Python is a good fit for Raspberry Pi camera and media processing.
  • Hardware adapters behind service classes make fake and real cameras possible.
  • MJPEG preview is simple and works well on appliance-style local networks.
  • Pydantic settings create a typed configuration surface that can drive an admin UI.
  • JSON schema generated from Pydantic settings can accelerate advanced admin forms.
  • A file browser and upload UI makes operator-managed assets practical.
  • Transparent PNG frame overlays can be displayed over the live preview and applied during processing.
  • The capture state machine gives the frontend a clear state to render.
  • File-based media storage keeps the system easy to inspect and recover.

Issues To Improve​

  • Many command endpoints use GET, which makes side effects easy to trigger accidentally.
  • The global service container couples hardware, processing, config, and web serving tightly.
  • Server-sent events are fine, but WebSockets give cleaner bidirectional control for setup and future challenges.
  • The media catalog is rebuilt into memory from folders instead of using durable queryable metadata.
  • Lighting is WLED-specific, while Pixelbooth should support WLED first and custom ESP32 firmware later.
  • Admin setup is powerful but dense; Pixelbooth needs guided setup for bar volunteers.
  • File browsing should be sandboxed to safe roots and dependency-aware before delete.
  • Command-style share/system actions are powerful but should become typed, allowlisted tasks with audit/status rather than arbitrary shell strings.
  • Schema-generated config is useful for advanced settings, but high-value flows such as presets, remotes, overlays, and first-run setup need custom UX.
  • The frontend is prebuilt Vue assets in the Python package; Pixelbooth needs first-class React/TypeScript source.

Features Pixelbooth Should Keep/Improve​

  • Live preview.
  • Still photo, collage, GIF/animation, and short video capture.
  • Capture countdown and approval flow.
  • Gallery with preview, detail, delete, and share.
  • Admin camera, lighting, GPIO, sharing, and diagnostics screens.
  • Admin dashboard as the first view when logged in as admin.
  • Admin asset/file browser with upload/download/delete.
  • Overlay preview and PNG frame overlay configuration.
  • Confirmed system actions, improved into a Commander task runner.
  • Physical button hooks.
  • Lighting presets connected to capture state.
  • Fake camera and fake lighting for development away from the Pi.

Architecture Direction​

Pixelbooth should keep the appliance-friendly local model but split responsibilities more deliberately:

  • A FastAPI API process owns REST commands, WebSocket events, and static frontend serving in production.
  • Python domain services own camera control, sessions, media, processing, lighting, sharing, and diagnostics.
  • SQLite stores media/session/config metadata while files store original and rendered assets.
  • React/TypeScript owns kiosk, gallery, and admin UI.
  • ESP32 lighting sits behind a protocol-neutral lighting adapter.
