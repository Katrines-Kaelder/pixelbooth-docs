Design Direction

Identity​

Pixelbooth is for one of the biggest Friday bars at Aarhus University on an engineering campus. The bar is known for 8-bit pixel art on the walls, so the booth should feel like it belongs there: bright, playful, technical, social, and a little competitive.

Working concept: Pixel Portal: High Score Selfie Booth.

Visual Language​

  • Dark venue backdrop with saturated pixel color: hot pink, electric cyan, coin yellow, arcade green, and deep violet.
  • Pixel marquee language: PLAYER 1 READY, PRESS START, INSERT POSE, NICE SHOT, HIGH SCORE.
  • Strong rectangular composition inspired by arcade cabinets and LED matrix panels.
  • Big touch targets, high contrast, and instantly readable states for a noisy bar.
  • White "portal" light motif for capture readiness, with RGB side glow for session state.
  • The formal moodboard and design brief lives in moodboard.md.

Booth UI Tone​

  • Fast, confident, celebratory.
  • Uses short arcade copy rather than explanatory text.
  • Treats capture like a game round: start, countdown, score/result, share the win.
  • Challenge-ready without requiring challenge logic in v1.

Admin UI Tone​

  • Calm, practical, and scan-friendly.
  • Uses conventional controls, status indicators, and test buttons.
  • Still visually related to the booth through color accents and pixel motifs, but avoids decorative overload.

Design Constraints​

  • The booth screen must work as a first screen, not a marketing page.
  • The kiosk UI must support touch and physical button use.
  • The admin UI must be usable by tired volunteers during a busy bar night.
  • Text must never be the only feedback for capture or lighting state.
  • Production styling should use Tailwind CSS through semantic system tokens, not arbitrary component-local colors.
  • Components must not introduce arbitrary one-off colors; raw color values belong in theme/profile definitions.

Screen And Layout Targets​

  • Primary target display: 1280x800 touchscreen in the physical Pixelbooth.
  • Kiosk-facing screens should be designed first for this resolution, then adapted to other sizes.
  • User-facing booth flows must fit within one viewport without scroll bars.
  • The gallery is the only user-facing screen that may scroll.
  • Capture, countdown, review, print/share, challenge, onboarding step screens, and error/recovery states must all fit without vertical or horizontal scrolling on 1280x800.
  • Admin screens may scroll when needed because they are operator/setup surfaces, not guest-facing capture surfaces.
  • Layouts should use responsive constraints, not fixed-only coordinates, so future screen swaps remain possible.
  • Touch targets must remain large enough for bar/event use even when adapting down from 1280x800.

Guest Capture Flow (Booth Touchscreen)​

The guest-facing booth touchscreen is built around a live-preview-as-background concept. The full generation brief and prompt live in booth-touchscreen-design-brief.md. The current screen-level implementation reference is the design-prototypes/booth-handoff/ Claude Design handoff, which refines that brief into the Stage 3.6 booth visual fidelity pass.

Core decisions:

  • The live camera preview is the full-bleed background of the guest screens; all pixel/arcade UI (text, icons, buttons, panels) is overlaid on top, with legibility treatment (vignette/gradient, semi-transparent pixel-panel backings, glow) so it reads over unpredictable live video.
  • The pixel LED marquee is separate hardware above the screen and carries branding/status independently. The touchscreen shows no PIXEL PORTAL wordmark and is never a marketing/brand page.
  • There is no on-screen "camera ready" badge — the live preview itself is the readiness signal. When no camera is available, the preview is replaced by an on-brand arcade scene with a friendly, non-technical message (degraded mode).

Screen flow and back-navigation:

  1. Start / Attract — live preview background, one dominant pressable START. Its prompt/CTA may be centered and may overlap the guest's face, because the live preview behind it still telegraphs movement and keeps the screen alive. No back button.
  2. Mode Select — live preview background; large touch tiles for capture mode (still, collage, animation/GIF-boomerang, video). Optional normal/wide camera-framing toggle. Keep tiles to edges so the guest still sees themselves.
  3. Countdown — live preview background; large pixel 3 → 2 → 1 with strong emphasis, then a capture flash.
  4. Processing — frozen captured frame, dimmed, with an arcade processing animation while overlays/GIF/video render (consistent with fast-capture-path: instant frame first, heavy render in the background).
  5. Result — finished media as the hero; compact arcade toolbar with a delete action and share options (QR, share, print), kept off faces.

A back button appears on every screen except Start. Except on the Start screen, persistent overlay UI stays at the edges/corners and does not cover faces.

This flow refines the older per-screen notes in moodboard.md: the marquee is physical (not an on-screen strip), branding is not on the touchscreen, and Mode Select and Processing are explicit steps.

Frontend Implementation Style​

  • Prefer small reusable components over long screen files.
  • Route-level screens compose feature components and should remain thin.
  • Use shared utilities for formatting, media paths, marquee canvas math, print crop calculations, exposure helpers, and API error normalization.
  • Use proper state ownership: server state in query/cache tools, booth/editor event state in a small client store, local state only for simple local interactions.
  • See frontend-development-standards.md for the detailed frontend engineering rules.
