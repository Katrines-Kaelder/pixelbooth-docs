Pixelbooth Moodboard And Design Brief

This document promotes the local design inspiration images into a practical moodboard for future UI, enclosure, lighting, marquee, and onboarding work. It is a direction-setting brief, not a final visual specification.

Source Images​

Use these as visual references:

Reference app screenshots remain useful for feature inventory, but they should not drive the visual style.

Direction​

Working identity: Pixel Portal: High Score Selfie Booth.

The booth should feel like a compact wall-mounted arcade object that belongs in a student bar: pixel-art native, bright in a dark room, technically confident, easy to use after two beers, and customizable for rentals.

The strongest visual ingredients from the inspiration boards are:

  • bright stepped white portal frame around the camera/screen area
  • wide LED marquee above the main interaction area
  • black physical enclosure with visible screws, depth, and serviceability
  • magenta/cyan/rainbow side glow as session-state feedback
  • short arcade copy, not explanatory UI text
  • chunky pixel icons and score/game language
  • camera treated as a deliberate hardware feature, not hidden as a webcam afterthought
  • event names and custom messages on marquee surfaces

Design Pillars​

Pixel Portal​

The white stepped frame is the signature. It should represent readiness, photo quality, and the moment of capture.

Use it for capture readiness, countdown emphasis, review/result celebration, ring-light calibration UI, and physical enclosure language. Avoid reducing it to a decorative border in the web UI. It should have a purpose: framing the live preview, communicating lighting state, or showing capture status.

High Score Arcade​

The booth should feel like starting a tiny arcade round.

Useful language:

  • PRESS START
  • PLAYER 1 READY
  • INSERT POSE
  • STRIKE A POSE
  • GET THE SHOT
  • NICE SHOT
  • SHARE THE WIN
  • HIGH SCORE

Use this most strongly on kiosk screens and marquee content. Admin screens can borrow icons and status colors, but should stay calm and practical.

Bar-Ready Hardware​

The boards work because the object feels real: black shell, service panel, side fins, wall mount, camera hood, ring light, and accessible controls.

Software implications:

  • admin screens expose hardware status clearly
  • setup flows feel like configuring a booth appliance, not a generic website
  • camera, light, marquee, printer, Wi-Fi, and sync status are first-class
  • rental onboarding supports event naming and marquee customization

State-Driven Light​

Color is not just decoration. It should communicate session state.

Initial state mapping:

  • idle/attract: magenta + cyan ambient glow
  • camera ready: white portal light stable
  • countdown: pulsing white portal with marquee countdown
  • capture: brief white flash, RGB side glow dimmed or controlled
  • review: calmer cyan/magenta frame
  • success: coin yellow + magenta celebration
  • error/recovery: restrained warning color, no panic disco
  • challenge: playful accent color plus pose-specific marquee

Visual System​

Color​

The inspiration leans on a dark base with saturated arcade accents. These are token candidates, not component-local color values.

Suggested semantic tokens:

  • color.background.kiosk: near-black venue backdrop
  • color.background.panel: black acrylic/enclosure surface
  • color.text.primary: warm white
  • color.text.arcade: pixel white
  • color.action.primary: hot pink
  • color.action.secondary: electric cyan
  • color.action.reward: coin yellow
  • color.status.ready: arcade green
  • color.status.error: controlled red/magenta warning
  • color.arcade.marqueeText: hot pink/cyan/yellow palette slots
  • color.arcade.portalLight: warm-white ring light
  • color.hardware.sideGlowA: magenta
  • color.hardware.sideGlowB: cyan/blue

Rules:

  • raw color values belong in theme/profile files
  • kiosk colors may be loud; admin surfaces should use the same system more quietly
  • avoid turning the whole app into a purple-blue gradient
  • keep contrast in the palette: white, black, cyan, magenta, yellow, green, and occasional red

Typography​

Use two typographic voices:

  • font.display: pixel/bitmap display face for kiosk headlines, marquee previews, countdowns, and result labels
  • font.body: readable sans for admin, setup, diagnostics, captions, and longer labels

Guidelines:

  • pixel font is for short phrases only
  • admin should not use pixel type for dense data
  • do not scale font size with viewport width
  • keep letter spacing at 0
  • make button labels short enough for touch UI

Shape And Layout​

Use:

  • rectangular panels
  • stepped/pixel frame motifs
  • 8px-or-less radius for normal UI cards and controls
  • strong 16:10 kiosk composition for 1280x800
  • stable preview/camera regions that do not resize during countdown/result states

Avoid:

  • pill-shaped generic SaaS controls on the kiosk
  • nested cards
  • marketing-style hero sections
  • decorative gradient blobs
  • overly soft glassmorphism

Icons​

Use pixel icons for booth/game language:

  • heart
  • star
  • coin
  • invader
  • diamond
  • camera
  • QR/share
  • print

Use conventional icons for admin operations through the chosen icon library. Pixel icons are acceptable in admin only as small brand accents.

Kiosk Screen References​

The guest touchscreen is built live-preview-as-background: the camera preview fills the screen and pixel/arcade UI is overlaid on top. The pixel LED marquee is separate hardware above the screen and carries branding/status — so the touchscreen has no on-screen marquee strip and no PIXEL PORTAL wordmark. The full flow and generation prompt are in booth-touchscreen-design-brief.md; the agreed flow summary is in design-direction.md (Guest Capture Flow). Every screen fits 1280x800 without scroll; a back button is present on all screens except Start.

Start / Attract​

Primary feeling: walk up and press the big thing.

Required elements:

  • live preview as the full-screen background
  • one dominant, pressable START (also triggerable by the physical button)
  • short arcade prompt, e.g. INSERT POSE / TOUCH TO PLAY
  • attract-mode life: gently pulsing START, soft glow

Notes: the prompt/CTA may be centered and may overlap the guest here, because the live preview behind keeps telegraphing movement. No back button. Degraded (no-camera) fallback: an on-brand arcade scene with a friendly message and a disabled START.

Mode Select​

Primary feeling: pick how I want to play.

Required elements:

  • live preview background
  • large touch tiles for capture mode (still, collage, animation/GIF-boomerang, video) with pixel icons; selected tile clearly highlighted
  • optional normal/wide camera-framing toggle (disabled if only one camera)

Notes: keep tiles to the edges/lower area so the guest still sees themselves.

Countdown​

Primary feeling: game round has started.

Required elements:

  • large pixel countdown over the live preview, with a capture flash at the end
  • live preview remains dominant
  • portal/ring-light state visible
  • marquee mirrors countdown or pose prompt (on the separate hardware)
  • no extra explanation

Processing​

Primary feeling: the machine is working its arcade magic.

Required elements:

  • the just-captured frame, frozen and dimmed, as the background
  • an arcade processing animation (pixel loading bar / blocky spinner)
  • short copy such as RENDERING… / COMBINING… / COMPRESSING…

Notes: consistent with fast-capture-path — show the instant frame immediately and render overlays/GIF/video in the background.

Result​

Primary feeling: the shot landed.

Required elements:

  • the finished media as the hero (photo; GIF/video plays)
  • a compact arcade toolbar with a delete action and share options (QR, share, print)
  • short celebration copy
  • safe progress status if the full render is still completing

Notes: keep the toolbar off faces; show unavailable share/print options as dimmed (with a reason on demand), not hidden.

Gallery​

The gallery is the only guest-facing screen allowed to scroll.

Use newest-first media cards, strong thumbnails, media type labels, QR/share/print actions, and hidden/source media handling for collages.

Admin Direction​

Admin should be quieter than the booth.

Use the moodboard for:

  • section accents
  • status colors
  • marquee preview canvas
  • camera preview framing
  • lighting preset previews

Do not use the moodboard for:

  • dense neon dashboards
  • pixel font everywhere
  • huge decorative panels
  • unclear controls

Admin priorities:

  • camera health
  • exposure/ring-light profiles
  • marquee editor/status
  • sync remotes
  • Wi-Fi/fallback AP
  • printer setup
  • rental onboarding
  • diagnostics/logs

Marquee Direction​

The physical marquee is central to the identity.

Design implications:

  • event name and rental custom text must be easy to set
  • saved marquee designs should preview at 128x32 physical resolution
  • grouped 64x16 preview should be available for 2x2 pixel-art look
  • text, sprite, gradient, timer, and countdown layers should feel native to the display
  • GIF/video support can come later, but the design model should not block it

Good marquee phrases:

  • PLAYER 1 READY
  • INSERT SMILE
  • KATRINES KAELDER
  • GOOD TIMES
  • 3... 2... 1...
  • NICE SHOT
  • HIGH SCORE

Hardware/UI Crossover​

The physical design and software should reinforce each other:

  • screen UI uses the same state names as lighting/marquee events
  • ring-light editor previews portal brightness and warmth
  • side glow maps to lighting presets
  • camera selector maps to actual normal/wide camera roles
  • rental onboarding includes event marquee text
  • result/share state can trigger marquee celebration

Do Not Copy Directly​

The inspiration boards are AI concept art. Treat them as direction, not specification.

Do not copy:

  • exact enclosure dimensions from the boards
  • exact fake tech specs
  • unreadable small text density
  • impossible physical part layouts
  • random icons without product meaning
  • overly busy screen frames inside the kiosk UI

Keep:

  • wall-mounted arcade mood
  • black enclosure and visible hardware language
  • stepped white portal light
  • marquee-first identity
  • magenta/cyan/yellow/green pixel palette
  • playful state copy
  • compact, high-impact bar context

First UI Translation Tasks​

When frontend implementation begins:

  1. Define semantic theme tokens from this moodboard.
  2. Create a kiosk scene layout for 1280x800.
  3. Prototype the guest flow as static screens: start/attract, mode select, countdown, processing, and result (live-preview-as-background; see booth-touchscreen-design-brief.md).
  4. Prototype admin visual language separately so it stays practical.
  5. Add marquee preview components for 128x32, 64x16 hard-2x2, and 64x16 soft-2x2.
  6. Review every guest-facing state for no-scroll behavior.
