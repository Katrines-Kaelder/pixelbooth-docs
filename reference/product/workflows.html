Pixelbooth Workflows

Guest Booth Flow​

  1. Attract screen shows the bar identity, live camera preview, marquee text, and primary capture actions.
  2. Guest can switch the live preview between compatible available camera roles, usually normal and wide, before starting capture.
  3. Guest taps or presses a physical button for a preset such as selfie, wide group shot, collage, GIF, or short video.
  4. The session starts with the currently selected preview camera role unless the chosen preset requires a specific or dual-camera role.
  5. The session enters countdown; lighting switches to countdown and the marquee displays the current beat.
  6. Exposure pre-metering samples the scene and prepares camera exposure, EV/gain, white balance, and ring-light brightness/temperature for the selected role.
  7. The selected camera captures the frame or recording.
  8. The booth shows an instant preview while full-resolution render work runs in the background.
  9. For multi-step presets, the booth either auto-continues or asks the guest to accept/retry the capture.
  10. Processing creates final media plus preview, thumbnail, share, print, and archive outputs.
  11. The booth presents the result with QR, share, print, replay, and gallery options.
  12. The booth returns to attract/idle.

Bartender/Admin Flow​

  1. Open admin from a protected route or local-only shortcut.
  2. Land on the admin dashboard and check opening-night readiness: cameras, storage, temperature, network, lighting, overlays, remotes, printer, queues, and recent errors.
  3. Verify both cameras are detected as normal and wide.
  4. Test capture, focus, exposure, mirror/rotation, preview, and guest-facing camera switching for each camera.
  5. Configure capture presets and map them to on-screen, keyboard, and GPIO triggers.
  6. Upload/select overlay PNGs and preview them against the live stream or latest frame.
  7. Discover or manually configure ESP32 lighting.
  8. Configure exposure profiles for normal lights, club mode, and manual event-locked settings.
  9. Test ring-light brightness and warm/cold channel mix.
  10. Configure the Pi Zero 2 W marquee agent, preview designs, and assign designs to session events.
  11. Configure one or more sync remotes such as local folder, USB, rclone cloud remote, Immich, or rsync/SFTP.
  12. Test each remote and check queue health.
  13. Test printer setup and crop/safe-area preview.
  14. Use Commander for allowlisted operational tasks such as reload services, reboot/shutdown, diagnostic bundle export, queue retry, and hardware tests.
  15. Check diagnostics and logs before opening night.

Gallery Flow​

  1. The attract screen has a dedicated GALLERY button (upper-left) as the entry point; the bottom-right "snaps taken" counter is decorative only.
  2. Gallery is a scoreboard of recent snaps, newest first, as a grid of landscape tiles. It is the only guest-facing screen allowed to scroll; the tile grid scrolls vertically within the fixed 1280x800 stage (the page itself never scrolls). Up/down nudge arrows assist touch.
  3. Each tile shows a stable rank number (oldest is 01, so existing numbers never shift when a new snap lands), a relative timestamp, the mode (photo / collage / gif / video) with a color-coded neon frame, and a play badge for motion snaps.
  4. Tapping a tile opens a fullscreen detail view of that snap with the same controls as the result screen (delete, QR) plus a back button. The guest can move between snaps with left/right nav arrows, side-swipe, or keyboard arrows; navigation has hard stops at the newest and oldest snaps (never circular).
  5. Detail view shows the final render (video plays inline; GIF animates in place). Share is QR only; print is hidden until the printer integration lands (Stage 8). Delete recycles the item (bytes retained, hidden from the board).
  6. Hidden source captures from collages remain stored but are not shown by default.

Sharing Flow​

  1. Guest scans a QR code, selects a share action, or asks for print.
  2. API validates quota, cooldown, media existence, and configured provider.
  3. Share, sync, or print service records a queue job and emits status events.
  4. The UI shows pending, success, or failure without blocking the booth.

Rental Onboarding Flow​

  1. Superadmin archives or resets internal media.
  2. Superadmin switches the booth into rental mode.
  3. Borrower follows setup: Wi-Fi, event name, settings template, overlays, enabled modes, upload remotes, QR style, printer, camera, exposure, lighting, and marquee tests.
  4. Advanced destructive and hardware settings remain superadmin-only.

Settings Template Flow​

  1. Admin creates or imports a named event/rental/venue settings template.
  2. Admin applies the template as a draft and reviews changed sections.
  3. API validates typed settings and checks referenced assets/remotes.
  4. Admin confirms apply.
  5. Pixelbooth persists the active settings revision and emits a config update event.
  6. Services reload only the affected capabilities where possible; otherwise Commander can run a service reload task.

Overlay Flow​

  1. Admin uploads a transparent PNG overlay or selects an existing overlay asset.
  2. API validates file type, dimensions, alpha channel, and compatibility with the target output.
  3. Admin previews the overlay over the live preview stream or a captured still.
  4. Admin assigns the overlay globally, to an event/template, or to specific capture presets.
  5. During idle/countdown, the kiosk may show the overlay over the live stream if enabled.
  6. Render workers composite the overlay into final output without blocking capture feedback.
  7. Future overlay designer uses the same assignment and render pipeline, but stores an editable .pixelbooth-overlay.json definition.

Remote Sync Flow​

  1. Admin creates a remote and chooses its type: local folder, USB, rclone, Immich, rsync/SFTP, or future webhook.
  2. Admin fills common policy fields and type-specific settings.
  3. Admin runs a remote test.
  4. On capture completion, scheduled run, manual run, or USB insert, SyncService creates durable outbox jobs.
  5. Sync workers upload/export selected outputs and emit progress over WebSocket.
  6. Admin can pause a remote, retry failed jobs, inspect last error, or download a diagnostic bundle.

Diagnostics Flow​

  1. Admin opens diagnostics.
  2. System reports platform, Python version, app version, disk space, temperature, CPU, memory, cameras, lighting device, and recent logs.
  3. Camera and lighting checks are exposed as explicit test commands.

Future Challenge Flow​

  1. Guest chooses or receives a challenge such as "team pose", "wide group", or "high score".
  2. Session records the challenge metadata.
  3. A later vision service can analyze pose, faces, framing, or timing.
  4. Challenge result becomes part of the session and media metadata.
