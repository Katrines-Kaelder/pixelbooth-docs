Booth Touchscreen Design Brief (Pixel Portal)

This brief captures the agreed concept and the generation prompt for the guest-facing booth touchscreen, for use with Claude Design (or any UI generator). It is direction, not final specification — generated output is treated as a visual reference and re-expressed through semantic tokens and component primitives before it becomes production code.

See also: design-direction.md (Guest Capture Flow) and moodboard.md.

A generated prototype from this brief lives at design-prototypes/booth-touchscreen/ (reference only — open its index.html; it is rebuilt in apps/web with Tailwind + semantic tokens, see that folder's README for the token-mapping conversion guide).

The current implementation handoff lives at design-prototypes/booth-handoff/. It refines this brief into the production booth look used for the Stage 3.6 visual fidelity pass: Attract, Mode Select, Countdown, Processing, and Present. Treat the handoff as the current screen-level visual reference, and this document as the concept source and prompt history.

Core concept​

  • The live camera preview is the full-bleed background of the guest screens. All UI — pixel/arcade text, icons, buttons, panels — is overlaid on top.
  • There is a separate physical pixel LED marquee above the touchscreen that shows branding and status independently. The touchscreen therefore carries no PIXEL PORTAL wordmark and is not a brand/marketing page; it stays focused on the guest and minimal arcade overlay UI.
  • Overlaid text must stay legible over unpredictable live video (vignette/gradient, semi-transparent pixel-panel backings, neon glow, strong contrast).
  • The flow is: Start → Mode Select → Countdown → Processing → Result, with a back button on every screen except Start.
  • Idle/Start screen exception: its prompt text/CTA may be centered and may overlap faces — the live preview behind it still telegraphs the guest's movement, which is the point. The "keep UI off faces" rule applies to the other screens.

Generation prompt​

Design the guest-facing touchscreen flow for "Pixel Portal," an 8-bit arcade
selfie booth: wall-mounted in a dark, busy university bar. Strong pixel-art /
retro-arcade vibe is the single most important thing.

IMPORTANT CONTEXT
- There is a SEPARATE physical pixel LED marquee ABOVE the touchscreen that shows
  branding and status ("PLAYER 1 READY", etc.) independently. So the touchscreen
  itself must NOT show a big "PIXEL PORTAL" wordmark or act as a brand/marketing
  page. Keep the screen focused on the guest and minimal arcade UI.
- The LIVE CAMERA PREVIEW is the full-bleed background of the guest screens. All
  UI — pixel text, icons, buttons, panels — is overlaid on top of that live video.
  Use a mock person/scene as the preview background in the design.

CORE VISUAL CONCEPT
- Full-screen live preview, with a translucent 8-bit/arcade UI layer on top.
- Because the preview is unpredictable live video, ensure overlaid text/buttons are
  always legible: use subtle dark gradients/vignette, semi-transparent pixel "panel"
  backings behind text, neon glow, and strong contrast. Never let UI wash out
  against a bright camera feed.
- Aesthetic: chunky pixel-block lettering (use the "Press Start 2P" pixel font for
  arcade text; a clean readable sans for any longer text), neon glow/bloom, optional
  scanline/CRT hint, corner HUD pixel brackets/dots framing the screen like an
  arcade cabinet bezel, and a small pixel icon set (star, coin, heart,
  space-invader, trophy, share, trash). Tasteful, not cluttered.

PALETTE (a starting direction — you MAY refine the exact hues/contrast; we will map
your final colors to design tokens afterward, so keep colors few and consistent):
dark near-black violet base, hot pink #ff2d95, electric cyan #22d3ee,
coin yellow #ffd400, arcade green #39ff14, deep violet #4c1d95, white #ffffff.

HARD CONSTRAINTS (non-negotiable)
- Every screen is exactly 1280 x 800 px, landscape, and fits with NO scrolling.
- Touch-first: primary actions are large finger targets (min ~64px tall). Assume a
  physical arcade button can also trigger the primary action.
- A BACK button appears on every screen EXCEPT the Start screen. Place it
  consistently (e.g., top-left), styled as a pixel/arcade control.
- Status/feedback must not rely on text alone — pair wording with color, icon,
  motion, or scale.

SCREENS TO DESIGN (one consistent set; the live preview persists as background
across Start, Mode Select, and Countdown)

1. START / ATTRACT
   - Full-screen live preview. Overlaid: a short arcade prompt (e.g. "INSERT POSE"
     / "TOUCH TO PLAY") and ONE dominant, pressable START button. On THIS screen the
     prompt/CTA may be centered and may overlap the guest — the live preview behind
     keeps telegraphing their movement, so it still feels alive. Attract-mode life:
     gently pulsing/blinking START, soft glow. No back button. No wordmark.
   - Fallback state (no camera available): replace the preview with an on-brand
     arcade scene (starfield + neon grid) and a friendly, non-technical "be right
     back" style message; START is visibly disabled. Keep it playful, never a raw
     error.

2. MODE SELECT
   - Live preview background. Overlaid: large touch tiles to pick a capture mode —
     Photo, GIF/Boomerang, Video, Collage — each with a pixel icon and label.
     Selected tile clearly highlighted. Back button present. Keep tiles to the
     edges/lower area so the guest can still see themselves.
   - Optional secondary control if it fits cleanly: a small toggle to switch camera
     framing (normal / wide); show a disabled state if only one camera exists.

3. COUNTDOWN
   - Live preview background. Huge pixel countdown (3 → 2 → 1) center screen with
     strong emphasis (scale/color/pulse), then a brief capture "flash" moment. Back
     button present (cancels back to mode select).

4. PROCESSING
   - Show the just-captured frame (frozen) as the background, dimmed, with an arcade
     processing animation over it (e.g., pixel loading bar or blocky spinner) and
     short copy like "RENDERING…" / "COMBINING…" / "COMPRESSING…". This stands in
     for applying overlays, assembling a GIF, or compressing video. Back button
     present.

5. RESULT
   - Shows the finished media full-bleed (a photo; for GIF/video assume it plays).
     Overlaid controls: a clearly-labelled DELETE/trash action, and SHARING options
     (QR code to grab it, share, and print) as distinct pixel-icon buttons. Back
     button present. Keep the result the hero; controls are a compact arcade toolbar
     (e.g., bottom bar) that doesn't cover faces.
   - Show a state where a share/print option is unavailable (dimmed + reason on
     demand), not hidden.

MOTION
- Attract pulse, countdown emphasis + capture flash, looping processing animation,
  subtle neon glow throughout. Lively but readable; nothing harsh or seizure-like.

OUTPUT
- React + Tailwind. Each screen is a thin component composing small reusable pieces
  (preview background layer, overlay panel, arcade button, back button, countdown,
  result toolbar, HUD frame). Favor a few shared color/spacing choices over many
  one-off values so the result is easy to re-express through a semantic token
  system later. Layout responsive within the 1280x800 frame (constraint-based, not
  absolute coordinates).

DO NOT
- No scrolling, no marketing/landing layout, no PIXEL PORTAL wordmark on screen.
- On Mode Select / Result, don't cover the guest's face with persistent UI; keep
  overlays to edges/corners. (The Start screen is exempt — centered prompt is fine.)
- Don't rely on text alone for state. Don't use a camera "ready" badge — the live
  preview is the readiness signal.

Notes when output returns​

  • Capture modes map to the documented CaptureMode set (still, collage, animation, video); adjust the tile list if v1 ships fewer modes.
  • The capture moment is folded into the end of the countdown (flash → Processing), matching fast-capture-path: show the instant frozen frame immediately, run heavy render during Processing, then present the final on Result.
  • Re-express the generated palette through --color-* semantic tokens; current token values are not locked and may be improved to match a good baseline.
  • Verify every screen holds at 1280x800 with no scroll before it becomes production.
