QR Sharing Setup (share remote + sharepage)

This guide sets up guest QR sharing: a guest scans the booth's QR and opens their photo on their own phone over mobile data. It ties together the three pieces — a share remote, the sharepage, and the Sharing settings — into one copy-paste setup.

See ADR-0018 for the design and admin-control-plane.md §Remotes for the field reference. Per-remote credential handling (write-only secrets) is ADR-0020.

Why it needs a share host​

Guest phones can't reach the Pi at the venue. So the booth uploads each finished photo to a public host (the share remote) and the QR encodes a public URL on that host. The host is a dumb static/file server — no PHP, no server-side script. It serves two things:

  1. the static sharepage (apps/sharepage/index.html), and
  2. the uploaded photos, under per-event folders.
host-root/
├── index.html                                   ← the sharepage (sharepage_url points here)
└── friday-bar/                                  ← {event slug}
    └── 2026-06-19-21.04.11-Xa9f2Qe.jpg          ← uploaded photo (the QR's ?id=)

Any S3/MinIO, WebDAV/Nextcloud, FTP, or SFTP host with a public base URL works. (Google Drive and Immich are backup/sync remotes, not QR backends — they have no plain public URL surface.)

Prerequisites​

  • A host you control with a public base URL guests can open (e.g. https://share.your-host.example). For MinIO this is typically a bucket exposed over HTTPS with public read on the object keys.
  • rclone on the machine you run the helper from (brew install rclone / rclone downloads). The booth's API image already includes it.
  • The host credentials (for MinIO: endpoint, access key, secret key, bucket).

Step 1 — Add the share remote in admin​

In Admin → Sharing → Remotes → New remote:

  1. Name it (e.g. "Share host").
  2. Destination → S3 / MinIO (or your host type). The form shows exactly that backend's fields.
  3. Fill the connection fields (for MinIO: endpoint, access key ID, secret access key, bucket; provider defaults to Minio). Use the eye toggle to verify the secret key, then add the remote.
  4. Click Test — you should see Connected to '<remote name>'. within a few seconds. (A wrong endpoint fails fast instead of hanging.)

Secrets are write-only: once saved they're never shown back, and on Edit a blank secret keeps the stored one (ADR-0020).

Step 2 — Turn on QR sharing​

In Admin → Sharing (the settings form):

  • Share remote → the remote you just added.
  • Link mode → sharepage (default).
  • Sharepage URL → the public base URL where index.html and the photos live. On S3/MinIO this must include the bucket, e.g. https://s3.your-host.example/your-bucket (not the bare host — the bare host resolves to the MinIO API/console). The QR targets {sharepage_url}/index.html?id=… so it works on object stores, which don't serve a directory index.
  • Share variant → final (default) or original.
  • Enable QR sharing.

Use the live share-link preview on the page to confirm the example URL looks right for the active event.

Step 3 — Deploy the sharepage​

On the share remote's row in Admin → Sharing → Remotes, click Deploy sharepage. The API uploads apps/sharepage/index.html to the remote's root using the credentials it already stores — no second tool and no re-typed secret (ADR-0021). On success it shows an Open sharepage link so you can eyeball it.

For S3/MinIO, make sure the uploaded objects (the page and the photos) are publicly readable, or the QR opens a broken page.

Alternative: the CLI helper (bootstrap / CI)

scripts/share-host-setup deploys the page and prints the admin values from your machine — useful to stand up a host before a remote exists in admin, or in CI. Pass secrets via env vars so they never land in shell history or ps:

cd scripts/share-host-setup
PB_SECRET_ACCESS_KEY=… python sharehost.py --backend s3 \
    --endpoint https://minio.your-host.example --bucket pixelbooth --prefix friday-bar \
    --access-key-id AKIA… --public-base-url https://share.your-host.example

Other backends (WebDAV/Nextcloud, FTP/SFTP) are in scripts/share-host-setup/README.md. You can also copy apps/sharepage/index.html to the host root by any other means.

Step 4 — Verify end to end​

  1. Take a photo at the booth.
  2. When the render finishes, the booth uploads it to the share remote (auto, off the capture path) and the QR becomes scannable.
  3. Scan it on a phone on mobile data → the sharepage loads the photo with a SAVE button.

Troubleshooting​

  • Test hangs / fails: check the endpoint URL (scheme + host, reachable from the booth) and that the bucket exists. The test bounds connect/IO time and disables retries, so a bad endpoint reports an error in ~10s rather than spinning.
  • QR opens a broken page: confirm index.html is at the host root (the sharepage_url), and that uploaded objects are publicly readable at {sharepage_url}/{event}/{file}.
  • QR opens the MinIO console / a login or bucket listing: sharepage_url is pointing at the bare host or a directory. On S3/MinIO it must include the bucket (https://host/bucket); the link targets …/index.html?id=…, which object stores serve (they don't serve a directory index for …/).
  • QR never appears for a photo: check the remote is enabled and designated as the share remote, and watch the sync job/queue in admin; the durable outbox retries failed uploads.
  • Per-event folders: the object key is {event-slug}/{file}, so set the event slug in the Event settings before the night.
  • Deploy/upload fails with a 502 (Test passes): the host accepts listing but not the upload. Two known causes behind a reverse proxy: a stale rclone (the Debian apt package is years old and breaks uploads — install the current official binary, curl https://rclone.org/install.sh | sudo bash), and HTTP/2 — the booth forces rclone to HTTP/1.1, but if you proxy S3 yourself, allow upload bodies (e.g. Caddy protocols h1). The booth surfaces only the proxy's truncated tail; check the host's proxy log for the real reason.
  • Deploy fails with PermanentRedirect (HTTP 308): the remote's Region doesn't match the server, so the signed request is rejected. Set Region on the remote (Admin → Remotes → edit) to the server's region — MinIO defaults to us-east-1; find a bucket's region with curl -s "{endpoint}/{bucket}/?location". Keep Path-style addressing = true for MinIO/self-hosted S3.
