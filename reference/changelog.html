Changelog

All notable Pixelbooth changes are summarized here from the full git history.

This project does not currently use release tags, so entries are grouped by development stage and date range. The changelog was last refreshed against origin/main commit 449c482 on 2026-06-22.

Unreleased​

Gallery detail swipe hardening - 2026-06-22​

Off-hardware rework of KI-024; Pi touchscreen verification pending.

Changed​

  • Reworked MediaDetail swipe from React touch events to Pointer Events with touch-action: none and a horizontal-dominance + threshold guard (KI-024). On a touchscreen the browser otherwise claims the horizontal pan as a native gesture, so the detail view "did not react" to swipes; pointer events also unify mouse-drag in dev. A vertical drag or a tap never navigates, and the newest/oldest hard stops are preserved.

Gallery omits media with missing files - 2026-06-22​

Off-hardware implementation of KI-026; Pi verification pending.

Added / Changed​

  • GET /media now omits completed items whose final byte file is missing on disk (KI-026): MediaService.list_media gained a require_files filter (a per-row stat on the returned page) so a catalog row that outlived its files never appears as a gallery tile that opens a 404. The booth "PICS TAKEN" counter (COUNT(*)) is intentionally left unfiltered.
  • GalleryCard now falls back to an in-style poster when a thumbnail fails to load, covering the race where a file disappears after the list was fetched.

Configurable inactivity auto-return - 2026-06-22​

Off-hardware implementation of KI-027; Pi no-touch verification pending.

Added​

  • Added operator settings booth.auto_return_enabled (default on) and booth.auto_return_seconds (default 45, range 5-600) that return abandoned guest screens to attract after no touch (KI-027). A new useAutoReturn hook in BoothScreen arms on the result/mode-select/error screens — present aborts the finished session, mode-select/error reset to attract — restarting on any pointer/touch/key event. Active capture phases (countdown/capture/record/ processing) and attract are excluded so an in-progress session is never cut short.

Operator control for CRT scanline overlay - 2026-06-22​

First increment of FR-004 (admin-editable theme controls); builds on KI-011.

Added​

  • Added an admin-editable booth.crt_scanlines setting (off / subtle / full) controlling the guest-screen CRT scanline overlay. full keeps the deliberate per-phase treatment (strong in attract/idle, faded back during the capture flow — KI-011), subtle pins a faint uniform pattern, and off fully disables the overlay. Rendered frontend-side in BoothStage via a new pure scanlineIntensity(mode, phase) helper, and exposed in the admin Booth settings as a labelled select.

Admin control plane, runtime settings, and operator telemetry - 2026-06-17 to 2026-06-18​

Added​

  • Rebuilt admin as a desktop-first control plane with top-level operator categories, category sidebars, a full-width dashboard landing page, dark/light admin theming, and a draft-based Save all changes settings workflow.
  • Added a real Cameras admin panel and moved camera-specific controls out of generic settings forms, giving later preview/orientation/focus/exposure work a purpose-built home.
  • Added runtime capture settings wired end-to-end: mode enable/disable flags for Photo, Collage, GIF/Boomerang, and Video; countdown timing; collage shot count; record duration; GIF/video playback fps; and guest-facing booth/gallery settings such as attract counters, gallery enablement, page size, newest-first ordering, and event profile fields.
  • Added network/system settings and dashboard usage: event timezone drives the dashboard clock, display_hostname gives the booth a friendly device label, and log_level applies to the root logger on startup.
  • Added richer Commander support: task descriptions in the UI, a gated expert-mode host command (shell_exec) with audit output (ADR-0019), and scheduled allowlisted tasks that run through the same confirmation, timeout, status, and audit path as manual runs.
  • Added system telemetry for the admin dashboard: CPU, RAM, temperature, uptime, network interfaces, Wi-Fi SSID, and fast dashboard-only metrics polling.

Fixed​

  • Reworked the admin dashboard into data-driven cards with clearer readiness signals, real camera names, a merged System card, better logo/home behavior, and improved readability/spacing across dense operator views.
  • Replaced rough storage estimates with media-folder byte counts, per-format capacity prediction, and a more prominent storage card so opening-night risk is visible before the disk fills.

Changed​

  • Updated the admin architecture docs to match the implemented two-level navigation, settings model, dashboard scope, Commander posture, remote management, and future slots for lighting, print, logs, setup, and media operations.

Operations, validation, and docs-site maintenance - 2026-06-17 to 2026-06-20​

Added​

  • Added a hardware validation test plan and test-notes inbox covering Pi setup, camera verification, media quality, sharing/QR, remote transports, and known issue intake, so booth testing can feed directly back into the roadmap.
  • Added feature-request and known-issue intake workflows plus new roadmap items for overlay designer work, theme controls, and a WLED preset generator.
  • Added a kiosk run mode and self-diagnosing Pi setup improvements so the setup script can launch the intended booth path more reliably on target hardware.
  • Published the root changelog into the Docusaurus reference site and fixed the visual roadmap map to show stages 7, 8, 9, and 10 with contained horizontal scrolling and non-overlapping connectors.

Stage 8 transport completion and sharing hardening - 2026-06-17 to 2026-06-20​

Added​

  • Completed the Stage 8 remote transport set behind the common remote model: local folder, USB export, rclone cloud/self-hosted remotes (S3/MinIO, WebDAV, FTP, SFTP, and Google Drive), Immich native upload, and rsync over SSH. Google Drive and Immich are supported as backup/sync destinations; they are not the recommended public QR backend.
  • Implemented the non-rclone transports in the service layer: Immich uploads via its native asset API with stable deviceAssetId idempotency; rsync uses SSH key auth in BatchMode for LAN/self-hosted targets; and USB export copies the media tree to a configured mounted drive path while leaving privileged mount/unmount to a host helper.
  • Added typed admin remote forms with destination-specific fields, per-remote sync frequency, test actions, and write-only secret handling so stored credentials are redacted on read and preserved unless explicitly replaced (ADR-0020).
  • Added the remote destination picker that exposes rclone backends as operator choices (S3 / MinIO, Google Drive, WebDAV / Nextcloud, SFTP, FTP) alongside Immich, rsync, USB, and local folder instead of hiding everything behind a generic rclone type.
  • Added the operator QR sharing guide for configuring a share remote, deploying the static sharepage, setting the public URL, and testing upload-to-scan end-to-end.
  • Added the validation test plan and issue/feature intake docs so hardware, sync, QR, and future stage findings can be captured consistently.

Fixed​

  • Hardened sharepage deployment and rclone uploads for real object-store behavior: the admin can deploy apps/sharepage/index.html through the stored share remote (ADR-0021), QR links target index.html explicitly, MinIO/S3 defaults use path-style addressing and us-east-1, the API/Pi setup installs the official rclone binary, and rclone is forced to HTTP/1.1 for reverse proxies that drop HTTP/2 request bodies.

Changed​

  • Refreshed the admin control-plane docs for the desktop-first two-level admin layout, Cameras panel, real dashboard metrics, Sharing/Remotes workflow, Commander expert mode and scheduled tasks (ADR-0019), and the current settings model.
  • Updated roadmap, known-issues, API, operator, and Docusaurus reference docs so Stage 8 is recorded as active with sync/QR largely implemented and print still remaining.

Continuous autofocus and sensor exposure controls - 2026-06-22​

Hardware-verified on the target Pi (2026-06-22); KI-023 resolved.

Added​

  • Enabled autofocus on the CSI Camera Module 3 cameras (ADR-0025, KI-023): the Picamera2 adapter applies AfMode=Continuous + AfSpeed=Fast by default so the lens tracks guests as they move into frame, with an admin-editable focus mode/range and a manual lens_position lock (dioptres) as an escape hatch for low-contrast club lighting where continuous AF hunts.
  • Added sensor-side exposure/white-balance controls — the camera-only slice of the exposure work (ADR-0025, partial KI-006): the sensor's own AEC/AGC/AWB plus ae_enable, ae_metering, ae_exposure_mode, exposure_value (EV), an optional manual exposure_time_us/analogue_gain pair, and awb_enable/awb_mode. The coordinated ring-light/metering ExposureService stays deferred to Stage 5.
  • Extended the typed camera settings section with the focus/exposure fields, read live by the adapter via a CameraControls value object (mirroring PreviewTuning), so an operator edit applies without a restart and a patch re-applies to an open camera in place. The admin Cameras form gained Focus and Exposure & white-balance field groups.
  • Added focus/exposure metadata (lens_position, focus_fom, exposure_time_us, analogue/digital gain, lux, colour_temperature) to GET /system/metrics, surfaced on the dashboard Cameras card next to the measured fps, so an operator can verify autofocus and exposure while tuning on hardware.

Changed​

  • ensure_seeded now backfills missing fields within existing sections (not just whole missing sections, KI-003), so an older settings document self-heals the new focus/exposure fields on startup. Bumped SETTINGS_SCHEMA_VERSION to 2.

Low-latency CSI live preview - 2026-06-21​

Added​

  • Added a SQLite-backed, admin-editable camera settings section (preview fps, lores size, preview main size, and image flip) that the Picamera2 adapter reads live, so operator edits apply without a restart — a successful patch reconfigures any open camera in place (ADR-0024, KI-003). This is the correct home for the preview tuning that was briefly (and wrongly) env vars; it mirrors how the render output sizes live in CaptureSettings (ADR-0017). The section is admin-only (absent from the guest config view).
  • Added an image flip via a libcamera Transform: flip_horizontal (mirror), flip_vertical, and a flip_scope — all flips the saved still too (camera mounted mirrored/rotated), preview_only mirrors just the live preview (selfie-mirror UX, the saved photo stays un-mirrored). Edited on the new Cameras settings form, with inline horizontal/vertical quick toggles on the admin dashboard.
  • Added the measured live-preview producer fps to GET /system/metrics (computed from the hardware encoder's real frame timing) and surfaced it on the dashboard Cameras card, so an operator can watch the actual frame rate while tuning the preview.

Fixed​

  • Fixed the preview looking cropped/zoomed versus the saved photo (KI-003): the preview main default was 1536x864, a cropped IMX708 readout, while the still captures the full-FoV native sensor. The default is now the full-FoV 2x2-binned 2304x1296 mode (~56fps), so the preview frames the same scene as the capture.
  • Fixed the admin "Could not load camera settings" on databases seeded before the camera section existed: ensure_seeded now backfills any top-level section added since the stored document was written (and re-persists it), so older databases self-heal on startup instead of returning an incomplete settings document.

Changed​

  • Reworked the Picamera2 (CSI) live preview for a realtime feed (KI-003, ADR-0024). The camera previously idled in a continuous native-resolution still config, which pinned the IMX708 sensor to its ~14fps full-res readout and starved the lores preview, and pulled frames via per-frame capture_file. It now idles in a light create_video_configuration (small main → fast sensor mode) and switch_modes to the native still config only for capture, so the captured ORIGINAL stays native while idle preview runs fast. Live preview is served by a continuous hardware MJPEGEncoder writing a latest-frame buffer, so the stream yields the freshest frame instead of re-encoding per frame. Preview fps/size are now the runtime camera settings above (preview_fps default 30); the env Settings keep only the USB/fake bootstrap preview budget. The CameraAdapter protocol gains measured_fps/refresh_preview_tuning (no-ops on fake/USB); the MJPEG endpoint is unchanged. Hardware-verified on the Pi (2026-06-21): preview frames the same scene as the capture, runs at 30 fps, latency much improved and acceptable, CPU fine, flip works — KI-003 resolved.

In-app sharepage deploy - 2026-06-20​

Added​

  • Added a Deploy sharepage action on the share remote's admin row that uploads apps/sharepage/index.html to the remote's root using the remote's stored credentials — no second tool and no re-typed secret. New endpoint POST /sync/remotes/{id}/deploy-sharepage, a PIXELBOOTH_SHAREPAGE_PATH setting (defaults to the repo-tree copy), and ADR-0021. The scripts/share-host-setup CLI helper is now the bootstrap/CI fallback.

Fixed​

  • Forced the rclone transport to HTTP/1.1 (GODEBUG=http2client=0) so uploads survive S3-compatible reverse proxies that drop HTTP/2 request bodies (e.g. MinIO behind Caddy returning a 502 on PUT while listing succeeds).
  • Defaulted the S3 backend to path-style addressing + region us-east-1 (both overridable, exposed on the remote form), fixing a PermanentRedirect on upload to MinIO/AIStor (virtual-host addressing is 308-redirected without MINIO_DOMAIN, and a mismatched signed region scope is rejected).
  • Installed the official rclone binary in the Pi setup script and API image instead of the stale Debian apt package, which fails to upload to modern S3 servers (e.g. MinIO AIStor).
  • Pointed the sharepage QR link at index.html explicitly ({sharepage_url}/index.html?id=…) so it opens on object stores, which don't serve a directory index; on S3/MinIO sharepage_url must include the bucket.

Full-resolution Pi stills end-to-end (KI-022) - 2026-06-21​

Fixed​

  • Fixed the CSI preview/capture crashing on the native capture-size default: the Picamera2 adapter passed main=None to create_still_configuration to mean "native", but Picamera2 needs an empty dict there (main=None crashes in align_stream). Pass main={}; the test fake now rejects None to guard it.
  • Fixed the processed final still being 1280x720 while the ORIGINAL was full resolution: the render composed the still from the downscaled instant preview variant instead of the full-res original, capping the final at the preview size regardless of the (native) still output box. The render now sources the still/fallback frame from the original, with a regression test asserting the final keeps native dimensions.

Shutdown reworked to the reference-app approach after on-Pi testing (KI-012) - 2026-06-21​

Fixed​

  • Reworked app shutdown to the reference photobooth's proven workaround for uvicorn #1579 after the first (cooperative-stop) attempt proved fragile on the Pi and made shutdown worse. The booth keeps two endless responses open (the MJPEG preview and the /events WebSocket); the python -m pixelbooth entrypoint now sets server.force_exit = True so uvicorn abandons both on the first signal instead of waiting them out — no graceful-drain stall and no force-cancelled CancelledError tracebacks.
  • Because force_exit skips the FastAPI lifespan, the entrypoint's handle_exit now releases the cameras synchronously (CameraService.close()) in the signal handler, so a restart can still reacquire the device.
  • Fixed the process hanging after its last shutdown log line and never exiting: a libcamera/V4L2 worker thread outlives Picamera2.close() and, being non-daemon, blocks the interpreter's own shutdown forever — leaving the orphaned process alive and the camera fds held, which defeats the restart KI-012 is about. main() now os._exit(0)s once the server returns (the cameras were already released). See the revised ADR-0023.

Graceful shutdown that releases the camera on the first Ctrl+C (KI-012) - 2026-06-21​

Fixed​

  • Fixed the Pi being unable to restart the app without a reboot: an open live preview (stream.mjpg) is an endless response, so a bare uvicorn launch hung on the first Ctrl+C and a forced second interrupt skipped the lifespan that releases the camera, leaking the locked CSI/V4L2 handle (uvicorn #1579). The Pi now runs the new python -m pixelbooth entrypoint, which signals preview streams to stop on the first signal (CameraService.request_preview_stop(), a one-way event the preview loop checks each frame and ends without reopening), so streams close, connections drain, and the lifespan releases the cameras; a bounded timeout_graceful_shutdown backstops a wedged stream. New PIXELBOOTH_HOST / PIXELBOOTH_PORT bootstrap settings and ADR-0023.
