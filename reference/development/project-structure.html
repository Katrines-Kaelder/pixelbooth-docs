Project Structure

Pixelbooth is a local-first appliance project with explicit service boundaries from the start. The codebase is intentionally small enough to run locally, but structured so hardware helpers, render workers, sync jobs, and deployment pieces can split out later.

Repository Layout​

apps/api/        FastAPI runtime, services, adapters, SQLite migrations, tests
apps/web/        React + TypeScript + Vite booth/gallery/admin frontend
apps/docs/       Docusaurus documentation site wrapper
.docs/           Canonical product, architecture, roadmap, and decision docs
design-prototypes/
                 Static design prototypes and visual exploration
docker-compose.yml
                 Compose-first local development shape

Backend​

The backend is Python/FastAPI because the hardest local problems are camera control, Raspberry Pi hardware access, media processing, and future computer vision work.

Important backend areas:

  • pixelbooth/http: REST and WebSocket endpoints.
  • pixelbooth/services: application services and orchestration seams.
  • pixelbooth/domain: typed domain models and contracts.
  • pixelbooth/adapters: fake and hardware-facing adapter implementations.
  • pixelbooth/db: SQLite engine and forward-only migrations.

The API keeps privileged operations behind service/helper boundaries. UI code should never reach directly into camera, GPIO, Wi-Fi, USB, or printing concerns.

Frontend​

The frontend is one React app with multiple surfaces:

  • /: booth/kiosk
  • /gallery: event gallery
  • /admin: admin control plane
  • future /standalone/*: secondary display modes

The guest-facing kiosk targets 1280x800 and capture screens must not scroll. Admin and gallery views may scroll where appropriate.

Documentation​

.docs/ is the canonical documentation source. Docusaurus does not own the Markdown; it mirrors .docs/ into an ignored generated folder before serving or building the site.

Use documentation to explain product behavior, architecture, APIs, hardware contracts, operator workflows, and staged implementation scope. Internal AI collaboration notes may exist in the repo, but they should not drive the public shape of the project documentation.

Staging Model​

The roadmap is staged deliberately:

  • Foundation and admin control plane first.
  • Camera spike and capture session orchestration next.
  • Media rendering, lighting, marquee, physical deployment, sync/print, challenge modes, and observability after that.

Use the roadmap for current status and future scope.
