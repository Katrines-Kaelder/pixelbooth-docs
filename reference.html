AI Collaboration Layer​

AI-assisted work is governed by the root CLAUDE.md and the .claude/ layer. The responsibility split: rules define enforceable constraints (one per concern), skills hold build-time domain knowledge, and agents review after building. See .claude/README.md for the full inventory and the pixelbooth-review routing index that maps each concern to its rule and reviewer. The decision behind this layer is ADR-0001.

The Docusaurus site is documented in project/docusaurus-documentation-plan.md and ADR-0010. .docs/ remains the canonical internal/reference content root; apps/docs owns curated public pages and publishes selected source docs under /reference/* without creating a second source of truth.

New AI session / new contributor checklist:

  1. Read this index in order.
  2. Identify the doc that owns the area you are changing.
  3. Check the relevant rule in .claude/rules/ and skill in .claude/skills/.
  4. While in architecture mode, prefer doc/ADR updates over writing code.

Current Baseline​

  • The first implementation target is a local-first Raspberry Pi 5/CM5-class booth.
  • The main cameras are Raspberry Pi Camera Module 3 normal and wide. The booth must degrade gracefully if only one is available.
  • Guests can switch the live preview camera before capture. The selected preview role becomes the capture role unless the preset requires a specific or dual-camera role.
  • The backend runtime should start with Python/FastAPI because the hardware, camera, media, and CV ecosystem is strongest there.
  • The architecture remains service-oriented so workers, privileged helpers, sync jobs, and future services can split into containers or separate processes later.
  • The frontend is React + TypeScript + Vite with pnpm, Tailwind CSS, semantic tokens, and small reusable components.
  • The frontend is one application: booth, gallery, admin, setup, and future standalone displays share the same React app and API/event model.
  • Admin is the appliance control plane, with dashboard, settings, assets, overlays, remotes, Commander, logs, and setup routes.
  • Admin-set settings are persisted through typed models and SQLite, with JSON import/export for templates, profiles, overlays, marquee definitions, and backups.
  • Stage 3 camera work is beyond the adapter-only spike: dual Camera Module 3 discovery/capture, low-latency CSI preview, native still capture, autofocus, image flip, and camera-side exposure metadata have been verified on the target Pi. USB/V4L2 selection, GIF preview recovery, and lighting-coupled exposure remain tracked as known issues/future work.
  • Stage 4 media foundations are active: captured media is cataloged in SQLite, bytes live in the file store, render work runs through the shared job queue, generated OpenAPI types feed the frontend, and the booth/gallery can show real still, collage, GIF/boomerang, and video outputs. Current work is now focused on Pi-observed media quality, progress, gallery, and performance gaps.
  • Stage 8 share/sync/QR has moved ahead of the later hardware stages: sync jobs reuse the shared queue; remotes cover local folder, USB, rclone S3/MinIO/WebDAV/FTP/SFTP/Google Drive, Immich native upload, and rsync/SSH; kiosk/gallery QR panels become scannable only after the designated share remote verifies the upload; and the static sharepage can be deployed from admin. Print remains the major unbuilt part of Stage 8.
  • The primary kiosk display is 1280x800. Guest capture screens must not scroll; only the gallery may scroll.
  • Heavy render work must be queued and parallelized so capture feedback is immediate.

Open Questions Before Implementation​

  • Exact Pi OS/container strategy for Picamera2/libcamera access — resolved in ADR-0009: bare-metal Pi OS with the system python3-picamera2 package, exposed to a --system-site-packages venv.
  • First version of the Docker Compose layout — the development Compose shape is decided in ADR-0002. The production Pi Compose layout and which functions need host-side helpers remain open (Stage 7).
  • Exact Canon Selphy printing path after hardware testing.
  • Which Immich upload API/import path, and rclone packaging/auth on the Pi — sync transport engines are decided in ADR-0005 (rclone default for cloud/SFTP, rsync/SSH for LAN, Immich-native upload). The exact Immich endpoint is still confirmed against current official docs at implementation time.
  • Whether the first marquee agent protocol should be HTTP, WebSocket, MQTT, or a small file/pull model.
  • Exact kiosk typography choices and license-friendly pixel/display fonts — the guest arcade UI uses Press Start 2P for display text and VT323 for arcade labels/body accents.

Resolved since the first draft (now scheduled or decided):

  • Capture session orchestration is an explicit stage/service — ADR-0003 (Stage 3.5).
  • Admin access control lands with the first admin surface — ADR-0004 (Stage 2.5).
  • Observability is local-first with no Sentry — ADR-0006; the logging/diagnostic-id foundation moves to Stage 2.5.
