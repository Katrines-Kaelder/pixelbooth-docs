ADR-0009: Picamera2 (CSI) camera adapter on bare-metal Pi OS

  • Status: Accepted
  • Date: 2026-06-13
  • Deciders: Jonas Landsfeldt, Claude

Context​

ADR-0008 brought the USB/V4L2 adapter forward as the first real camera, deliberately deferring the CSI Camera Module 3 path on two blockers: (a) no dual-camera hardware on hand, and (b) an open question — the Pi OS/container strategy for Picamera2/libcamera access (.docs/README.md §Open Questions).

Both blockers are now cleared. Two Raspberry Pi Camera Module 3 units are available for testing, and we are choosing the host strategy. Picamera2/libcamera is awkward to run inside a container (it needs libcamera IPC plus device and DMA-heap access, and the python binding is built against the system libcamera), whereas the Compose dev shape (ADR-0002) explicitly scoped real hardware out of the container and into a bare-metal Pi path.

python3-picamera2 is distributed as a system apt package on Raspberry Pi OS, built against the system libcamera; it is not reliably pip-installable. So the decision is really two coupled choices: where the runtime lives (bare-metal vs container) and how the API process imports a system-provided Python package.

The existing seam keeps this cheap: CameraAdapter is a small Protocol (discover / is_available / capture_test / stream_frames) that CameraService depends on abstractly, and the availability table already covers both / normal_only / wide_only / none. Unlike USB (one device → one role), CSI gives us two cameras, so this adapter is the first to drive true dual-camera (both / dual) operation.

Decision​

Implement the Picamera2 (CSI) adapter and run the API on bare-metal Pi OS, using the system python3-picamera2 package. No container for the camera runtime.

  • Host strategy (resolves the open question). The local runtime runs directly on Raspberry Pi OS. libcamera/Picamera2 come from the OS (python3-picamera2, rpicam-apps), not from pip or a container.
  • venv exposure. The backend virtualenv is created with --system-site-packages and is based on the system python3 that owns python3-picamera2, so the venv can import it while our pinned dependencies (installed into the venv) still take precedence. Current Pi OS (Trixie / Debian 13) ships Python ≥ 3.12 system-wide, satisfying the backend's requires-python.
  • Not a pip dependency. picamera2 is not added to pyproject.toml and gets no optional extra (you cannot reliably pip install it). It is lazy-imported in the adapter (mirroring cv2), so fake/dev/CI paths never touch it; a missing import becomes a clear RuntimeError pointing back at this ADR.
  • Dual-camera role mapping. The adapter manages up to two Picamera2 instances, one per role, via a role → camera_num map (picamera2_normal_index=0, picamera2_wide_index=1 by default; camera_num indexes Picamera2.global_camera_info()). A role is available only when its index is actually connected, so every degraded state falls out of the hardware with no new availability logic. Each camera has its own lock so preview and still capture on the same device serialize.
  • JPEG path. Frames (preview and still) are produced by Picamera2's own capture_file(..., format="jpeg") pipeline (backed by simplejpeg, which ships with picamera2), keeping colour conversion inside the library and avoiding a dependency on the OpenCV usb extra on a CSI-only booth. Overlay compositing is unaffected — it stays downstream in the render worker per fast-capture-path.
  • Default backend. The Pi setup helper (pi-pixelbooth-setup.sh) defaults camera_backend to picamera2; usb and fake remain selectable by env var.

Consequences​

  • Easier: true dual-camera (normal + wide) operation and the both/dual preset modes are now exercisable on real optics; the libcamera/container question is closed; the CSI-only booth stays lean (no OpenCV needed for the camera path); degraded modes need no new code.
  • Constrained: the API must run on bare-metal Pi OS for the real camera path — it is not containerised on the Pi, diverging from the Compose dev shape (which remains the off-Pi dev and CI environment, with fakes). The venv is coupled to the system Python (so apt picamera2 is visible); a mismatched/pyenv interpreter will not see it. Real frame-rate/latency/CPU and stream colour/format are verified on the Pi, not the dev laptop.
  • Follow-up: production packaging (systemd unit, autostart, serving the built frontend) remains Stage 7. Capture quality, exposure, and ring-light tuning remain Stage 5. Hard synchronisation between the two cameras is still out of scope for V1.

Alternatives considered​

  • Containerise the camera runtime (device + libcamera passthrough). Keeps Compose parity on the Pi, but libcamera-in-a-container needs IPC, /dev device and DMA-heap access, and a python binding matched to the host libcamera — exactly the unresolved, fragile work that blocked Stage 3. Rejected for V1; bare-metal is the supported Picamera2 path.
  • pip install picamera2 into an isolated venv. Avoids the system-site coupling, but the wheel is not reliably installable (it builds against system libcamera) and Debian guards system packages (PEP 668). Rejected: fights the platform.
  • Run the API on system Python with --break-system-packages (no venv). Simplest import story, but pollutes the OS Python and risks breaking system tooling. Rejected in favour of a --system-site-packages venv that stays isolated for our pinned deps.
  • Encode frames with OpenCV (reuse the usb extra). Reuses one imaging library, but forces the heavy OpenCV wheel onto a CSI-only booth and adds a colour-order conversion the picamera2 pipeline already handles. Rejected: use Picamera2's native JPEG path; OpenCV stays optional.
