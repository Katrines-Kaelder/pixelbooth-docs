ADR-0001: AI collaboration layer (CLAUDE.md, rules, skills, agents)

  • Status: Accepted
  • Date: 2026-06-10
  • Deciders: Project owner

Context​

The project reached the end of architecture mode with decisions captured across docs/. Before implementation resumes, AI-assisted and human work needs a consistent entry point and guardrails so future sessions stay aligned with those decisions instead of re-deriving or contradicting them. docs/project/ai-collaboration-setup.md specified what this layer should preserve.

Constraints:

  • The repo is in architecture mode; accidental scaffolding must be prevented.
  • The guidance must point to docs/ as the source of truth, not duplicate it.
  • The project owner maintains a global library of reusable skills/agents that should be reused, not reinvented.

Decision​

Introduce a project-local AI collaboration layer:

  • A root CLAUDE.md as the forceful entry point that links to docs rather than restating them.
  • 7 rules in .claude/rules/ — short, enforceable constraints, each the single source of truth for one concern.
  • 5 authored skills in .claude/skills/ (pixelbooth-architecture|frontend|backend|hardware|review) holding build-time domain knowledge; pixelbooth-review is a routing index, not a checklist.
  • 5 authored agents in .claude/agents/ (architecture-curator, backend-runtime-reviewer, frontend-kiosk-reviewer, hardware-integration-reviewer, test-and-release-reviewer) that are thin: they read the paired skill + rules as source of truth and add only review workflow, top blockers, and severity output.
  • Relevant library skills/agents copied into .claude/ so the repo is self-contained and portable.
  • This ADR practice (docs/architecture/decisions/) for significant future decisions.

Responsibility split to avoid overlap: rules define the enforceable line, skills teach how to build, agents review after building. Each concern has exactly one defining rule.

Consequences​

  • Future sessions start from docs/README.md and inherit the guardrails automatically.
  • Skills are the single source of truth for domain standards; agents stay small and cannot drift from them.
  • Copied library files duplicate the global versions and must be re-synced if the originals change materially.
  • Significant decisions now require an ADR (enforced via the docs-first-architecture rule), adding light process overhead in exchange for traceability.
  • docs/roadmap/roadmap.md Stage 1 AI-setup items are satisfied (moodboard excluded).

Alternatives considered​

  • Reference the global library only, copy nothing — leaner repo, but not self-contained for collaborators/CI and dependent on one machine's setup. Rejected for portability.
  • Inline all rules inside CLAUDE.md — single file, but makes CLAUDE.md long and unscannable and loses one-file-per-constraint enforceability. Rejected.
  • Agents that restate the full domain standard — self-contained agents, but ~80% duplicated text that drifts from the skills over time. Rejected in favor of thin, skill-deferring agents.
  • No ADR folder; keep decisions as prose only — lower overhead, but loses the "why / what was rejected" record and a stable supersession trail. Rejected; existing prose decisions are folded in by reference.
